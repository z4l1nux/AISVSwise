# C2.2 Adversarial-Example Resistance

> **Parent:** [C02: User Input Validation](C02-User-Input-Validation)

## Purpose

AI models are vulnerable to subtle input perturbations that humans cannot perceive but cause models to misclassify, misinterpret, or behave unexpectedly. In the LLM context, this includes Unicode tricks, homoglyph substitutions, invisible characters, and encoding manipulations that bypass text-based filters while preserving semantic meaning to the model. For vision and audio models, adversarial perturbations can cause misclassification with minimal visible/audible changes. This section requires normalization, anomaly detection, and hardened model variants to resist these attacks.

## Requirements

| # | Requirement | Level | Role | Threat Mitigated | Verification Approach | Gaps / Notes |
|---|-------------|:-----:|:----:|-----------------|----------------------|--------------|
| **2.2.1** | **Verify that** basic input normalization steps (Unicode NFC, homoglyph mapping, whitespace trimming, removal of control and invisible Unicode characters) are run before tokenization or embedding and before parsing into tool or MCP arguments. | 1 | D | Unicode-based filter evasion; homoglyph attacks (e.g., using Cyrillic "a" to bypass ASCII pattern matching); invisible character injection (zero-width spaces, zero-width joiners); control character injection that alters parsing behavior. | 1. Inspect the input preprocessing pipeline for Unicode NFC normalization (e.g., Python `unicodedata.normalize('NFC', text)`). 2. Confirm a homoglyph mapping table is applied (e.g., confusables.txt from Unicode Consortium). 3. Test with inputs containing zero-width spaces (U+200B), zero-width joiners (U+200D), right-to-left overrides (U+202E), and other control characters. 4. Verify normalization runs before any pattern matching or tokenization step. | Unicode NFC normalization is well-understood and has mature library support (Python `unicodedata`, ICU). Homoglyph mapping is more complex -- the Unicode confusables list has thousands of entries. Implementations should decide whether to map, reject, or flag. Normalization before tokenization is critical because tokenizers may split confusable characters differently. |
| **2.2.2** | **Verify that** suspected adversarial inputs are quarantined, and logged with payload snippets and trace metadata (source, tool or MCP server, agent ID, session). | 1 | V | Undetected adversarial probing; inability to perform incident response on adversarial attacks; loss of forensic evidence for attack pattern analysis. | 1. Review logging configuration to confirm adversarial detection events are captured. 2. Verify log entries include: redacted payload snippet, detection method, confidence score, source IP/user, tool/MCP server if applicable, agent ID, session ID. 3. Confirm quarantined inputs are stored separately and not processed further. 4. Test by submitting known adversarial inputs and checking log output. | "Quarantine" needs operational definition -- does the input get stored for later analysis, or simply blocked and logged? Storage of adversarial payloads must itself be handled carefully to avoid secondary injection risks (e.g., log injection). Payload snippets should be redacted to avoid logging sensitive user data. |
| **2.2.3** | **Verify that** statistical anomaly detection flags inputs with unusually high edit distance to language norms or abnormal embedding distances and that flagged inputs are gated before concatenation into prompts or execution of actions. | 2 | D/V | Adversarial perturbation attacks that produce syntactically unusual text; gibberish injection; token-level attacks that exploit tokenizer behavior; gradient-based adversarial suffix attacks (e.g., GCG attack by Zou et al., 2023). | 1. Confirm an anomaly detection component exists in the input pipeline. 2. Review the statistical model or heuristic used (e.g., perplexity scoring, embedding distance from a reference corpus, character n-gram frequency analysis). 3. Test with known adversarial suffixes (e.g., GCG-generated suffixes which appear as gibberish). 4. Verify that flagged inputs are blocked or routed through additional review before processing. | Perplexity-based detection (e.g., using a small language model to score input naturalness) is effective against GCG-style suffix attacks but can be evaded by more natural-sounding adversarial inputs. Embedding distance thresholds require a well-calibrated reference corpus. False positive rates can be high for multilingual inputs, code snippets, or technical jargon. Tools: Hugging Face `evaluate` library for perplexity, custom embedding distance calculators. |
| **2.2.4** | **Verify that** the inference pipeline supports adversarial-training-hardened model variants or defense layers (e.g., randomization, defensive distillation, alignment checks) for high-risk endpoints. | 2 | D | Model-level vulnerability to adversarial inputs; transferable adversarial examples; attacks that bypass input-layer defenses and directly exploit model weaknesses. | 1. Identify high-risk endpoints (e.g., those with financial, safety, or access control implications). 2. Review whether hardened model variants are deployed at these endpoints (e.g., adversarially trained models, models with SmoothLLM-style randomization). 3. Check for defense layers such as input randomization (random token dropping/swapping), defensive distillation, or ensemble verification. 4. Review adversarial evaluation results if available. | Adversarial training for LLMs is still an active research area and not widely available as a turnkey solution. SmoothLLM (Robey et al., 2023) applies random perturbations to detect adversarial inputs. Defensive distillation has been shown effective for classification models but is less studied for generative models. Most organizations will implement this as a wrapper defense layer rather than retraining models. |
| **2.2.5** | **Verify that** encoding and representation smuggling in both inputs and outputs (e.g., invisible Unicode/control characters, homoglyph swaps, or mixed-direction text) are detected and mitigated. Approved mitigations include canonicalization, strict schema validation, policy-based rejection, or explicit marking. | 3 | D/V | Encoding smuggling attacks; bidirectional text attacks (Trojan Source style); output-side smuggling where model responses contain hidden instructions for downstream consumers; homoglyph-based phishing in model outputs; invisible text injection. | 1. Test both input and output paths with: invisible Unicode characters, homoglyph sequences, mixed RTL/LTR text, Unicode tag characters (U+E0001-U+E007F), variation selectors. 2. Verify that outputs are scanned for the same encoding attacks as inputs (output-side smuggling is critical for agent chains). 3. Confirm one of the approved mitigations is applied: canonicalization, schema validation, rejection, or explicit marking/escaping. 4. For agent-to-agent communication, verify both sides apply encoding controls. | This requirement covers both input and output, which is unusual for C2. The output-side coverage is important because adversarial content in model outputs can attack downstream systems (especially in agent chains). The Trojan Source attack (Boucher et al., 2021) demonstrated that bidirectional text can hide malicious logic in source code; similar techniques apply to prompt content. Unicode tag characters are particularly dangerous as they are invisible in most renderers. Level 3 reflects the complexity of comprehensive encoding smuggling defense. |

## Open Research Questions

- How effective are perplexity-based detectors against adversarial suffixes that are optimized to appear natural?
- Can adversarial training be applied to large language models at scale, or will defense layers (randomization, ensembles) remain the primary approach?
- What is the false positive rate of encoding smuggling detection in production multilingual systems?
- How should adversarial detection thresholds be calibrated for different languages and domains?
- Are there reliable adversarial robustness benchmarks for generative AI (beyond classification-focused benchmarks like RobustBench)?
