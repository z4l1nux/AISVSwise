# C6.1 Pretrained Model Vetting & Origin Integrity

> **Parent:** [C06: Supply Chain Security](C06-Supply-Chain.md) | **Requirements:** 5 | **IDs:** 6.1.1–6.1.5

## Purpose

Pretrained models downloaded from public hubs (Hugging Face, Model Zoo, Civitai, etc.) represent one of the highest-risk supply-chain entry points in AI systems. Unlike traditional software dependencies, model weight files can encode arbitrary executable code (via pickle deserialization), contain hidden backdoor triggers, or carry undisclosed licensing or data-provenance obligations. This section ensures every third-party model is authenticated, scanned, and reviewed before it enters the organization's pipeline.

| **6.1.1** | **Verify that** every third-party model artifact includes a signed origin-and-integrity record identifying its source, version, and integrity checksum. | 1 | D/V | **Tampered or substituted model weights.** An attacker replaces a legitimate model file on a public hub with a trojaned version. Without checksums, the swap is invisible (MITRE ATLAS AML.T0010.002). The OWASP LLM Top 10 2025 elevated Supply Chain from #5 to **LLM03**, reflecting the growing severity of model-origin attacks. **Model namespace reuse** is a newer vector: Palo Alto Unit 42 showed (Feb 2025) that when a model author deletes their Hugging Face account, anyone can re-register the namespace and upload a malicious replacement. Pipelines referencing the old name silently pull the attacker's model — researchers demonstrated RCE on Google Vertex AI and Azure AI Foundry this way. Google now runs daily scans for orphaned model references. **Model Confusion attacks** (Checkmarx, January 2026) add a dependency-confusion-style vector: when code calls `from_pretrained("checkpoints/model-name")` without a local-path prefix (`./` or `/`), the Transformers library may silently download a remote model from Hugging Face instead of using a local file. Researchers found vulnerable code in Fortune 500 repositories, and with `trust_remote_code=True` set, this escalates to full RCE. Mitigations include setting `HF_HUB_OFFLINE=1`, using `local_files_only=True`, and always using absolute paths. The EU AI Act GPAI obligations (in force August 2025) require providers to maintain technical documentation and share provenance information with downstream users. The **NIST Cyber AI Profile** (IR 8596, preliminary draft December 2025) extends CSF 2.0 to AI systems, explicitly directing organizations to "extend supply chain risk management to model and data supply chains" and verify data provenance with the same rigor as software and hardware. | Check that every imported model has a corresponding manifest containing: publisher identity, version tag, and SHA-256/SHA-512 hash. Verify the signature chain back to the publisher's public key. As of June 2025, the **OpenSSF Model Signing (OMS) specification** provides a formal, PKI-agnostic standard for model signing — it supports bare keys, PKI certificate chains, and Sigstore keyless signing, and uses a detached signature format covering weights, configs, and tokenizers in a single verifiable unit. NVIDIA has been signing all NGC catalog models with OMS since March 2025; Google is prototyping OMS integration for Kaggle. Pin models to specific commit hashes (not just names) to defend against namespace-reuse and Model Confusion attacks. Set `local_files_only=True` in all `from_pretrained()` calls in production, and enforce `HF_HUB_OFFLINE=1` in CI/CD environments. Cross-reference against CycloneDX ML-BOM or SPDX AI profile for standardized provenance metadata. | The OMS specification is a significant step forward, but adoption beyond NVIDIA and Google remains limited as of early 2026. Most Hugging Face publishers still do not sign. Organizations should self-sign upon first import and treat the internal hash as the trust anchor. Always pin model references to commit hashes, not just namespace/name — name-based trust is demonstrably broken. The Model Confusion attack (January 2026) shows that even local path references can trigger remote downloads — audit all `from_pretrained()` calls for implicit remote fetching. NIST AI RMF "Map" function explicitly calls for AI-BOMs, acknowledging traditional SBOMs do not capture model-specific risks. The NIST Cyber AI Profile (IR 8596, December 2025 draft) further reinforces this, requiring organizations to maintain inventories covering models, agents, APIs/keys, datasets/metadata, and embedded AI integrations, plus maps of end-to-end AI data flows. EU AI Act enforcement powers take effect August 2026, with fines for non-compliance. |
| **6.1.2** | **Verify that** models are scanned for malicious layers or Trojan triggers using automated tools before import. | 1 | D/V | **Pickle deserialization RCE and embedded malware.** PyTorch `.pt`/`.bin` files use Python pickle, which can execute arbitrary code on load. Backdoored models may also contain hidden trigger patterns that activate on specific inputs. (CVE-2024-3568 in Hugging Face Transformers, JFrog 2024 Hugging Face malware findings). **2025 incidents:** (1) *nullifAI attack* (Feb 2025) — ReversingLabs discovered malicious models on Hugging Face using broken/corrupt pickle streams and 7z compression to evade Picklescan; payloads were reverse shells connecting to hardcoded IPs. Hugging Face removed them within 24 hours. (2) *MalHug detection pipeline* found 91 malicious models and 9 malicious dataset-loading scripts across 705K+ models on Hugging Face. (3) JFrog disclosed three critical PickleScan zero-days (CVE-2025-10155, CVE-2025-10156, CVE-2025-10157, all CVSS 9.3) exploiting divergence between how PickleScan parses and PyTorch loads files — patched in PickleScan 0.0.31 (Sep 2025). (4) Sonatype independently discovered four additional PickleScan bypass vulnerabilities (CVE-2025-1716, CVE-2025-1889, CVE-2025-1944, CVE-2025-1945, disclosed March 2025) exploiting unsafe deserialization, non-standard file extensions, and ZIP archive manipulation — patched in PickleScan 0.0.23. (5) *Fake Alibaba SDK attack* (ReversingLabs, May 2025) — three malicious PyPI packages (`aliyun-ai-labs-*-sdk`) masquerading as Alibaba Cloud AI Labs SDKs hid info-stealing malware inside pickle-formatted PyTorch model files; downloaded ~1,600 times before takedown. This showed that malicious models are now being embedded inside traditional package managers, adding a layer of obfuscation beyond Hugging Face-hosted models. (6) Cisco Talos published **pickle-fuzzer** (December 2025), a structure-aware fuzzing tool that found crash-inducing bugs in ModelScan, Fickling, and PickleScan via memo-key validation and unhashable-bytearray issues — patched in Fickling v0.1.5 and PickleScan v0.0.32. **Configuration file attacks** are an overlooked vector: the "Rusty Link in the AI Supply Chain" study (2025) analyzed 150,000 Hugging Face repositories and found 13,091 repos with file-operation risks and 35,761 with repository-redirection risks via malicious configuration files. The ConfigScan tool was developed for LLM-based detection of these threats. | Run **multiple** scanning tools — no single scanner is sufficient given demonstrated bypass rates. As of March 2026, Hugging Face hosts over 2 million public models, and the Protect AI Guardian + Hugging Face partnership (launched October 2024) has scanned **4.47 million model versions** across 1.41 million repositories, identifying **352,000 unsafe/suspicious issues across 51,700 models**. Guardian has launched four new detection modules since the partnership began: archive-slip vulnerabilities (PAIT-ARV-100), Joblib suspicious execution (PAIT-JOBLIB-101), TensorFlow SavedModel backdoors (PAIT-TF-200), and Llamafile malicious execution (PAIT-LMAFL-300). Primary open-source tools: **Protect AI ModelScan** (supports H5, Pickle, SavedModel; commercial Guardian extends to 35+ formats), **Promptfoo ModelAudit** (42+ formats, allowlist-first architecture, SARIF output for CI/CD, detected 16 issues vs. ModelScan's 3 in head-to-head testing), **Trail of Bits Fickling** (pickle decompilation + allowlist scanner, v0.1.5 with fuzzer-hardened parsing). A promising new approach is **SafePickle** (February 2026 paper): instead of blocklist/allowlist inspection, it extracts opcode-frequency vectors from pickle bytecode and trains ML classifiers, achieving 90% F1-score versus 7–63% for existing scanners with sub-millisecond runtime — well-suited for CI/CD integration. Use Cisco's open-source **pickle-fuzzer** to stress-test whichever scanner you deploy. Prefer SafeTensors/ONNX format over pickle-based formats. For Trojan detection, run Neural Cleanse, ABS, or MNTD scans on candidate models. Scan configuration files alongside model weights — the ConfigScan approach demonstrates that `config.json` and related files can contain malicious repository redirections. | Trojan-trigger scanning (Neural Cleanse, etc.) is computationally expensive and has known false-negative rates. **Pickle scanning alone is insufficient:** seven distinct PickleScan bypass CVEs were disclosed in 2025, and Cisco's pickle-fuzzer (December 2025) found additional crash-inducing bugs across all three major open-source scanners, proving that blocklist-based scanners remain brittle. The SafePickle paper (February 2026) found that ~98% of files flagged as unsafe by existing scanners on Hugging Face are actually benign — the false-positive burden is a real obstacle to adoption. Defense-in-depth requires: (1) multiple scanners with different detection architectures, (2) format restrictions (SafeTensors-only policy where feasible), (3) sandboxed model loading, and (4) configuration file scanning. As of March 2025, only ~42% of Hugging Face models use SafeTensors — an empirical study (arxiv 2501.02170) found that many maintainers ignore automated SafeTensors conversion PRs, and 98.7% of repos that adopted SafeTensors retained their old pickle files alongside the new format. HiddenLayer's 2026 AI Threat Landscape Report found that malware in public model and code repositories is now the most-cited source of AI-related breaches (35% of reported incidents), and one in eight AI breaches involves agentic systems pulling models autonomously. The academic paper "The Art of Hide and Seek" (2025) demonstrates further stealthy pickle poisoning techniques. |
| **6.1.3** | **Verify that** model licenses, export-control tags, and data-origin statements are recorded in an AI BOM entry. | 2 | V | **Legal and compliance risk from undisclosed training data or license terms.** Models trained on copyrighted, GDPR-covered, or export-controlled data can expose the organization to lawsuits or regulatory action. The EU AI Act GPAI obligations (effective August 2025) require providers to publish sufficiently detailed summaries of training-data content, comply with EU copyright law, and share documentation with regulators and downstream deployers. Non-compliance fines apply from August 2026. NIST AI 600-1 (Generative AI Profile) extends the AI RMF to explicitly address training data transparency, model provenance, and third-party dependency risk for generative AI systems. | Audit the AI BOM for each model artifact. Confirm it includes: SPDX license identifier, data-origin summary (e.g., "trained on Common Crawl + proprietary corpus"), and export-control classification (EAR/ITAR if applicable). Cross-reference against the model card if one exists. Use **CycloneDX ML-BOM** (stable through v1.7, ratified as ECMA-424 2nd Edition, December 2025) or **SPDX 3.0.1** AI profile as the BOM format. CycloneDX v1.7 added a **Citations** element that lets BOM authors declare exactly which tool, build system, or person contributed each piece of provenance data — useful for audit trails showing who attested to training-data composition. Tools like Promptfoo ModelAudit can generate SBOMs and extract license information as part of scanning. | Many open-weight models lack clear data-origin disclosures. Organizations may need to perform independent provenance research. EU AI Act GPAI obligations are now enforceable — providers of models placed on market before August 2025 must comply by August 2027. NIST AI RMF and NIST AI 600-1 explicitly call for AI-BOMs. The systemic-risk tier (models exceeding 10^25 FLOP) carries additional safety and security obligations. CycloneDX v1.7 Citations provide the granular attribution needed for regulatory evidence, but tooling to auto-populate these fields is still emerging. |
| **6.1.4** | **Verify that** high-risk models (e.g., publicly uploaded weights, unverified creators) remain quarantined until human review and sign-off. | 2 | D/V | **Deployment of unvetted or backdoored models.** Automated pipelines may pull and deploy models without human judgment, allowing compromised artifacts to reach production. The MalHug study found 91 malicious models across 705K+ on Hugging Face — demonstrating that public hubs contain real threats at scale, not just theoretical risks. The Rapid7 "From .pth to p0wned" analysis (2025) documented end-to-end exploitation chains from model download to remote access trojan deployment. The fake Alibaba SDK attack (ReversingLabs, May 2025) showed that malicious pickle models are now being embedded inside traditional PyPI packages — quarantine controls must extend beyond model hubs to any dependency that could bundle model files. | Confirm that a quarantine zone (separate registry, staging namespace, or approval gate) exists. Verify that models from unverified publishers cannot progress past the quarantine stage without a documented human approval record (e.g., Jira ticket, signed review form). Run automated scanning (ModelScan, ModelAudit, Fickling) during quarantine. Integrate Palo Alto AI Runtime Security or Black Duck AI model scanning (released Oct 2025) for enterprise-grade quarantine workflows. The Protect AI + Hugging Face scanning partnership has identified 352,000 unsafe issues across 51,700 models out of 4.47 million versions scanned — roughly 1 in 85 model versions has a security concern, underscoring the need for automated gate-keeping. | Defining "high-risk" criteria requires organizational policy. Suggested triggers: unknown publisher, no model card, pickle format, model size anomalies, flagged by automated scanning, namespace squatting indicators, **recently re-registered namespaces** (the Unit 42 namespace-reuse attack exploits exactly this gap), or **path-based model references** that could trigger Model Confusion (Checkmarx, January 2026). Commercial tools like Protect AI Guardian, Black Duck, and HiddenLayer Model Scanner now offer managed quarantine-and-scan pipelines with integration into model registries like Databricks Unity Catalog. Guardian proactively detected CVE-2025-1550 (a critical Keras vulnerability) before public disclosure, demonstrating the value of continuous scanning partnerships. Sonatype's 2026 Software Supply Chain Report documented over 454,600 new malicious packages across ecosystems including Hugging Face, bringing the cumulative total of known malware to over 1.23 million packages. |
| **6.1.5** | **Verify that** transfer-learning fine-tunes pass adversarial evaluation to detect hidden behaviors. | 3 | D | **Sleeper-agent attacks and backdoor persistence through fine-tuning.** Research (Anthropic "Sleeper Agents" 2024, BadNets, TrojAI) shows that backdoors in base models can survive fine-tuning. An attacker contributes a poisoned base model; the victim fine-tunes it, unknowingly inheriting the backdoor. The "Models Are Codes" study (2024) systematically measured malicious code poisoning across pre-trained model hubs, confirming that poisoned models are a real and growing phenomenon, not just a lab demonstration. As of early 2026, research from Anthropic, the UK AI Security Institute, and the Alan Turing Institute has shown that as few as 250 malicious documents can compromise a model's behavior. A medical-LLM study (NYU/Columbia, January 2025) demonstrated that replacing just 0.001% of training tokens with fabricated medical articles — costing approximately $5 — boosted harmful outputs by 7.2%, and model size offered no protection (600M and 13B parameter models were equally easy to backdoor). | Run adversarial test suites against fine-tuned models, including: trigger-pattern sweeps (randomized input perturbations), activation-clustering analysis, and behavioral probes for known attack patterns. Compare fine-tuned model behavior against a clean baseline on a held-out adversarial evaluation set. The PickleBall framework (2025) proposes secure deserialization of pickle-based ML models as a complementary control. | This is computationally expensive and currently lacks standardized tooling. The TrojAI program (IARPA/NIST) is developing benchmarks. Most organizations will need to build custom adversarial evaluation pipelines. Level 3 is appropriate given the tooling gap. EU AI Act systemic-risk GPAI models (>10^25 FLOP) face additional safety evaluation requirements that may drive tooling maturity in this area. |

## Open Research Questions

- What is the false-negative rate of current Trojan-detection tools (Neural Cleanse, ABS, MNTD) against state-of-the-art backdoor attacks? The "Art of Hide and Seek" (2025) paper suggests attackers continue to outpace defenses.
- Can model provenance be established for models where training data and training process are not disclosed (closed-weight models accessed via API)?
- How should organizations handle models that are "fine-tune chains" (Model A fine-tuned from Model B fine-tuned from Model C) where provenance may be lost at each step?
- The OpenSSF Model Signing (OMS) specification now exists (June 2025) with NVIDIA and Google as early adopters — but will the broader Hugging Face ecosystem actually adopt it? What incentives or platform enforcement mechanisms are needed to drive signing adoption past the current handful of publishers?
- **How should organizations defend against model namespace reuse attacks?** Pinning to commit hashes helps, but many codebases and documentation still reference models by name alone. Is there a scalable way to detect when a previously trusted namespace has changed ownership?
- **How should organizations layer multiple scanning tools given demonstrated bypasses?** The JFrog PickleScan zero-days (CVE-2025-10155/10156/10157) and the nullifAI evasion show that no single scanner is reliable. What is the right combination of blocklist (PickleScan), allowlist (ModelAudit/Fickling), and format-restriction (SafeTensors-only) controls?
- **Will EU AI Act GPAI obligations drive standardization of model provenance metadata?** The August 2025 requirements for training-data documentation and downstream transparency may create de facto standards, but enforcement details remain unclear until August 2026.
- **Can the PickleBall secure-deserialization approach scale to production workloads?** If so, it could provide a runtime defense complementing static scanning.
- **How should organizations address configuration file attack vectors?** The "Rusty Link" study found tens of thousands of suspicious repos using malicious `config.json` redirections, but existing scanning tools focus almost exclusively on model weight files. ConfigScan demonstrates LLM-based detection, but adoption and false-positive rates in production remain unvalidated.
- **Is the Model Confusion attack pattern (Checkmarx, January 2026) the tip of the iceberg for implicit-download vulnerabilities?** The `from_pretrained()` path-resolution logic is just one library; other frameworks may have similar implicit remote fetching behavior that has not yet been audited.
- **Can ML-based scanning approaches like SafePickle replace or complement blocklist scanners in production?** The February 2026 paper shows dramatically better accuracy and speed, but real-world deployment at scale — with evolving adversarial techniques — remains unvalidated. What is the right trust model when the scanner itself uses ML?
- **How should organizations handle malicious models embedded inside traditional package managers (PyPI, npm)?** The fake Alibaba SDK attack (May 2025) showed pickle models being smuggled via PyPI packages — current model-scanning workflows focus on model hubs and miss this vector entirely.
- **Will the NIST Cyber AI Profile (IR 8596) drive convergence between AI and traditional supply-chain security requirements?** The December 2025 draft treats model and data provenance on par with software provenance, which could reshape procurement practices once finalized in 2026.
