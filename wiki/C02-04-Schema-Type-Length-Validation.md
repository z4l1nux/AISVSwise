# C2.4 Schema, Type & Length Validation

> **Parent:** [C02: User Input Validation](C02-User-Input-Validation)

## Purpose

AI systems increasingly expose structured interfaces -- APIs, tool calls, MCP endpoints, and function-calling interfaces -- where malformed or oversized inputs can cause parsing errors, prompt spillage across fields, type confusion, and resource exhaustion. Strict schema enforcement ensures that inputs conform to expected structure before they reach the model or trigger tool execution. This is especially critical for agentic systems where tool calls are generated by the model and must be validated before execution. Schema validation is a deterministic defense layer that does not depend on model behavior.

## Requirements

| # | Requirement | Level | Role | Threat Mitigated | Verification Approach | Gaps / Notes |
|---|-------------|:-----:|:----:|-----------------|----------------------|--------------|
| **2.4.1** | **Verify that** every API, tool or MCP endpoint defines an explicit input schema (e.g., JSON Schema, Protocol Buffers, or multimodal equivalent), rejects extra or unknown fields and implicit type coercion, and validates inputs server-side before prompt assembly or tool execution. | 1 | D | Schema injection attacks; field spillage (extra fields containing injection payloads); type confusion attacks; client-side validation bypass; unexpected parameter injection into tool calls. | 1. Inventory all API, tool, and MCP endpoints. 2. Confirm each has a defined schema (JSON Schema, Protobuf, Pydantic model, Zod schema, etc.). 3. Test with: extra/unknown fields (should be rejected, not silently ignored), wrong types (string where number expected), nested object injection. 4. Verify validation is server-side (not relying on client-side checks). 5. For tool-calling interfaces, confirm the model's generated tool arguments are validated against the schema before execution. | JSON Schema with `additionalProperties: false` is the standard approach. For Python, Pydantic v2 provides both schema definition and validation. For TypeScript, Zod is standard. MCP protocol defines tool input schemas in JSON Schema format -- implementations must enforce these schemas, not just document them. A common gap: tool schemas are defined but not actually validated at runtime. Implicit type coercion (e.g., JavaScript converting "123" to 123) is a specific risk that must be tested. |
| **2.4.2** | **Verify that** inputs exceeding maximum token or byte limits are rejected with a safe error and never silently truncated. | 1 | D/V | Context window overflow; silent truncation causing loss of safety instructions; resource exhaustion via oversized inputs; budget exhaustion through excessive token consumption. | 1. Review token/byte limit configuration for each endpoint. 2. Test with inputs at and above the limit. 3. Confirm oversized inputs are rejected with a clear error (HTTP 413 or equivalent), not silently truncated. 4. Verify that token counting uses the correct tokenizer for the target model (e.g., tiktoken for OpenAI, sentencepiece for others). 5. Check that byte limits and token limits are both enforced (a short text can have many tokens in some tokenizers). | Silent truncation is dangerous because it can remove safety-relevant content that was intentionally placed at the end of input. The error response should indicate the input was too large without revealing exact limits (to avoid helping attackers calibrate). Token counting must match the model's tokenizer -- a mismatch can allow inputs that exceed the actual context window. For multimodal inputs, image/audio token equivalents must be accounted for. |
| **2.4.3** | **Verify that** type checks (e.g., numeric ranges, enum values, MIME types for images/audio) are enforced server-side including for tool or MCP arguments. | 1 | D/V | Type confusion attacks; out-of-range values causing unexpected model behavior; MIME type spoofing for file uploads; enum bypass enabling access to unintended functionality. | 1. For each input field, confirm expected type, range, and enum values are documented and enforced. 2. Test with: out-of-range numbers, invalid enum values, spoofed MIME types (e.g., changing Content-Type header while sending different actual content), boundary values. 3. For file inputs, verify MIME type is checked by file content inspection (magic bytes), not just the Content-Type header. 4. Confirm tool/MCP arguments have the same type enforcement as direct API inputs. | MIME type validation by magic bytes is well-supported (Python `python-magic`, Node.js `file-type`). For tool calls, the model may generate arguments with wrong types (e.g., string "5" instead of integer 5) -- the validation layer must catch this. Enum validation should use a closed set; open enums defeat the purpose. Numeric range validation must consider edge cases (NaN, Infinity, negative zero in floating point). |
| **2.4.4** | **Verify that** semantic validators run in constant time and avoid external network calls to prevent algorithmic DoS. | 2 | D | Algorithmic complexity attacks (ReDoS); denial of service via slow validation; SSRF via validation-triggered network calls; resource exhaustion in validation layer. | 1. Review all regex patterns used in validation for catastrophic backtracking (use tools like regex101.com or `rxxr2`). 2. Confirm no validation step makes external HTTP/DNS calls (which could be exploited for SSRF or used to slow down validation). 3. Benchmark validation time with adversarial inputs designed to trigger worst-case complexity. 4. Verify timeout limits exist on all validation operations. | ReDoS is well-understood and testable. Tools like `rxxr2`, `regexploit`, or the `re2` library (which guarantees linear-time matching) can help. The "no external network calls" requirement prevents SSRF through validation (e.g., a URL validation step that actually fetches the URL). Semantic validation that requires database lookups should have timeouts and caching. This is a standard security engineering practice, not AI-specific, but included because AI validation pipelines often add novel semantic checks. |
| **2.4.5** | **Verify that** validation failures are logged with redacted payload snippets and unambiguous error codes and include trace metadata (source, tool or MCP server, agent ID, session) to aid security triage. | 3 | V | Inability to detect attack patterns; missing forensic evidence; inability to tune validation rules based on real-world failure data; undetected systematic probing of validation boundaries. | 1. Review logging configuration for validation failure events. 2. Confirm log entries include: error code (structured, not free text), redacted payload snippet (PII/sensitive data removed), source identification, tool/MCP server if applicable, agent ID, session ID, timestamp. 3. Verify redaction logic removes sensitive data but preserves enough context for security analysis. 4. Confirm logs are structured (JSON) for SIEM ingestion. 5. Test that validation failure logs are actually generated for each type of validation failure. | Level 3 because comprehensive structured logging of all validation failures requires significant infrastructure. Redaction is the key challenge -- must remove PII and sensitive content while preserving the security-relevant portion of the payload. Error codes should follow a taxonomy (e.g., `VALIDATION_SCHEMA_001`, `VALIDATION_TYPE_002`) to enable automated alerting and correlation. Logs should be compatible with SIEM tools (Splunk, Elastic, Sentinel). |

## Open Research Questions

- How should schema validation be applied to free-form text inputs that inherently lack structure?
- What are best practices for validating tool call arguments generated by the model in function-calling / tool-use architectures?
- Should validation schemas be shared between the model's tool definition and the runtime validation layer, or should they be independent?
- How do multimodal schemas (combining text, image, and audio constraints in a single request) evolve as models gain more modalities?
