# C10.1: Component Integrity & Supply Chain Hygiene

> **Parent:** [C10 MCP Security](C10-MCP-Security)
> **Requirements:** 3 | **IDs:** 10.1.1–10.1.3

## Purpose

This section ensures that MCP server and client software is obtained from trusted sources, that configuration artifacts do not leak credentials, and that only approved MCP servers are permitted in production. MCP components are typically distributed as npm packages, Python packages, Docker images, or standalone binaries — all of which are susceptible to supply-chain attacks including typosquatting, dependency confusion, and build-time tampering. Because MCP servers execute arbitrary code with access to sensitive resources (databases, APIs, file systems), compromised components have outsized blast radius compared to typical library dependencies.

The severity of these risks was demonstrated in real-world incidents throughout 2025-2026. The Postmark MCP supply chain breach (2025) saw attackers backdoor an npm package used in AI automation pipelines, inserting a single line of malicious code that directed compromised MCP servers to BCC every outgoing email to an attacker-controlled address. The Smithery supply chain attack (October 2025) affected 3,000+ hosted applications and their API tokens. Research using the MCPTox benchmark showed that tool poisoning attacks — where malicious instructions are embedded in tool metadata invisible to human users but processed by AI models — pass seamlessly into agent contexts with alarming frequency. A 2026 survey of 2,614 MCP implementations found 82% use file operations vulnerable to path traversal, two-thirds have code injection risks, and over a third are susceptible to command injection. These findings underscore why component integrity and supply-chain hygiene are foundational to MCP security.

| **10.1.1** | **Verify that** MCP server and client components are obtained only from trusted sources and verified using signatures, checksums, or secure package metadata, rejecting tampered or unsigned builds. | 1 | D/V | **Supply-chain compromise / trojanized MCP servers.** An attacker publishes a malicious MCP server package (e.g., via typosquatting on npm or PyPI) that exfiltrates data or injects malicious tool definitions. CVE-2025-6514 demonstrated this concretely: a critical command-injection bug in the `mcp-remote` npm package (437,000+ downloads) turned its OAuth proxy into an RCE vector — malicious OAuth endpoints were passed directly to system shell handlers, enabling full credential theft and repository exfiltration. Verify that MCP server binaries or images are pulled from pinned, authenticated sources. Evaluate adoption of [MCPS (MCP Secure)](https://mcp-secure.dev/) for cryptographic identity and signed tool definitions — it provides ECDSA P-256 "Agent Passports" and signed JSON-RPC message envelopes with nonce/timestamp replay protection. Run [Snyk agent-scan](https://github.com/snyk/agent-scan) (v0.4+, March 2026) to auto-discover MCP servers across Claude, Cursor, VS Code, and other agents and scan for prompt injection, tool poisoning, tool shadowing, and toxic flow vulnerabilities. Evaluate [Credence](https://credence.securingthesingularity.com/) for pre-install trust scoring — it performs static analysis, dependency CVE scanning, author provenance verification (detecting cloned repos and fake maintainers), and MCP-specific checks via adversarial AI deliberation, producing cryptographically signed attestations pinned to exact commit hashes. Sandbox-test third-party MCP servers before production use; inspect source code, verify maintainers, and run `npm audit` / Snyk scans to catch typosquatting and malicious exfiltration logic. Generate SBOMs for MCP server dependencies using tools like Syft or Trivy (CycloneDX/SPDX formats) — traditional manifest parsing often misses build-time dependencies that execute during installation. Pair SBOMs with VEX (Vulnerability Exploitability eXchange) documents to distinguish exploitable CVEs from false positives. For broader visibility, consider BuildGuard for AI-assisted SBOM generation that captures actual dependency chains beyond manifest parsing. | As of March 2026, the MCPS project offers the first MCP-specific signing layer (five trust levels L0–L4, from unsigned through audited), but adoption is early-stage. General package manager features (npm provenance, PyPI attestations, Sigstore) remain the primary verification mechanism. The MCP spec itself does not enforce audit, sandboxing, or verification — enforcement is left to implementers. A scan of 39 agent frameworks found 13 failed baseline security tests. Research by Invariant Labs found 5.5% of MCP servers exhibit tool poisoning characteristics — malicious instructions hidden in tool descriptions that are invisible to users but processed by AI models. Chained vulnerabilities in Anthropic's own `mcp-server-git` (CVE-2025-68145, CVE-2025-68143, CVE-2025-68144) demonstrated that even vendor-maintained components lack hardening, achieving full RCE via malicious `.git/config` files. Tool shadowing — where a malicious server registers a tool with the same name as a trusted one, exploiting naming collisions in the absence of namespace isolation — is an emerging class of supply-chain attack; CVE-2026-30856 in the Tencent WeKnora framework demonstrated this concretely, enabling tool execution hijacking through identifier collisions until patched with UUID-based tool identifiers in v0.3.0. |
| **10.1.2** | **Verify that** MCP client and server configurations do not contain plaintext secrets (API keys, tokens, client secrets) and that credentials are injected or resolved at runtime rather than stored in configuration files, environment variables, or source code. | 1 | D/V | **Credential exposure via configuration leakage.** MCP configurations (e.g., `claude_desktop_config.json`, `.mcp.json`) commonly embed API keys and database connection strings in plaintext. GitGuardian's 2026 State of Secrets Sprawl report found 24,008 unique secrets exposed in MCP-related configuration files across public GitHub, including 2,117 valid credentials (8.8% validity rate). Overall, 28.65 million new hardcoded secrets were pushed to public repos in 2025 — a 34% year-over-year increase. MCP credentials can also persist in agent context memory, creating "contextual secret leakage" where the protocol layer itself becomes an unintentional secret repository. Ranked MCP01 in the [OWASP MCP Top 10](https://owasp.org/www-project-mcp-top-10/2025/MCP01-2025-Token-Mismanagement-and-Secret-Exposure). | Scan configuration files for plaintext secrets using `trufflehog`, `gitleaks`, or `detect-secrets`. As of March 2026, the [GitHub MCP Server](https://github.blog/changelog/2026-03-17-secret-scanning-in-ai-coding-agents-via-the-github-mcp-server/) can scan code changes for exposed secrets before commit, integrating directly with MCP-compatible IDEs. Verify production deployments use secret managers (HashiCorp Vault, AWS Secrets Manager, Doppler, 1Password CLI) with runtime injection — tools like `doppler run` can inject secrets into MCP server processes at startup with automated rotation and audit trails. Check that `.gitignore` excludes MCP config files. Verify credentials are rotated every 30–90 days and that OAuth 2.1 short-lived tokens are used where supported (only ~8.5% of MCP implementations currently use OAuth). Where OAuth 2.1 is used, confirm Resource Indicators (RFC 8707) bind tokens to specific MCP servers — preventing tokens issued for one server from being replayed against another. Verify least-privilege scopes are enforced per tool/capability rather than catch-all scopes. Ensure secrets are never baked into Docker images and that credentials are separated across dev/staging/production environments with no overlap. For enterprise-scale discovery of MCP servers that may harbor credential issues, [Qualys TotalAI](https://blog.qualys.com/product-tech/2026/03/19/mcp-servers-shadow-it-ai-qualys-totalai-2026) can detect MCP servers at the network, host, and supply-chain levels — including localhost-bound servers on random high ports that evade traditional security tools — and extract tool catalogs, resource URIs, and filesystem access scopes for audit. | This is the most pervasive problem in the MCP ecosystem — approximately 48% of reviewed MCP servers recommend storing sensitive credentials in plaintext files. The MCP spec does not define a standard secret-reference mechanism (e.g., `${vault:secret/path}`), so each implementation handles this differently. The OWASP MCP Top 10 ranks this as MCP01, emphasizing that attack scenarios include prompt recall exposure (crafted prompts retrieve stored API keys from model memory), log scraping (raw MCP payloads containing tokens in debug logs), and context poisoning (malicious meta-instructions trigger secret leakage). |
| **10.1.3** | **Verify that** only allowlisted MCP server identifiers (name, version, and registry) are permitted in production and that the runtime rejects connections to unlisted or unregistered servers at load time. | 1 | D/V | **Rogue or unauthorized MCP server connections.** Without an explicit allowlist, an attacker (or misconfigured agent) can connect to arbitrary MCP servers — including attacker-controlled servers that serve poisoned tool definitions or exfiltrate data from tool arguments. The WhatsApp MCP breach (April 2025) demonstrated this: a malicious MCP server silently exfiltrated a user's entire chat history through tool poisoning. The 2026 finding that 8,000+ MCP servers were publicly exposed, with 492 lacking basic authentication, underscores the scale of this risk. Over 40% of public MCP servers tested were susceptible to command injection. Allowlisting prevents unauthorized server connections from entering the trust boundary. | Review production MCP client configuration for an explicit allowlist of permitted server identifiers (name, version, registry/source). Verify the runtime rejects connections to servers not on the allowlist. Test by configuring a connection to an unlisted MCP server and confirming it is blocked at load time. Check that the allowlist is version-pinned to prevent connecting to a compromised newer version. For GitHub Copilot environments, verify the organization has configured [MCP registry and allowlist controls](https://docs.github.com/en/copilot/how-tos/administer-copilot/manage-mcp-usage/configure-mcp-server-access) with "Registry only" policy — this prevents developers from using non-approved servers. For Amazon Q Developer environments, verify MCP governance controls are enabled — Q Developer supports registry-based allowlisting with 24-hour sync cycles, automatically terminating unlisted servers and relaunching version-mismatched servers with registry-defined versions. For [Kiro IDE](https://kiro.dev/blog/enterprise-governance-mcp-and-models/) environments, verify equivalent registry enforcement is configured (supports npm, PyPI, and OCI server packages with deterministic policy enforcement). For custom deployments, evaluate [MCP Gateway & Registry](https://github.com/agentic-community/mcp-gateway-registry) for centralized enforcement with OAuth authentication and dynamic tool discovery. | As of March 2026, GitHub's MCP registry controls (available to Copilot Business/Enterprise) provide the most mature allowlisting enforcement, matching on server IDs with enhanced command-path matching. The MCP spec itself does not define registries, allowlists, or access control — enforcement depends entirely on the hosting platform. The [MCP Gateway & Registry](https://agentic-community.github.io/mcp-gateway-registry/) project (with Keycloak/Entra integration) offers an open-source alternative for self-hosted environments, though it addressed SQL injection vulnerabilities in its own codebase as recently as March 2026. MCP gateway architectures are debated — some argue they introduce a single point of failure and should be replaced with per-server enforcement. A critical limitation across all current allowlist implementations: enforcement is name-based, not cryptographic. No mechanism in any MCP host today ties a server name to a verified binary or endpoint through cryptographic attestation, meaning a compromised registry entry could serve a trojanized server under a legitimate name. AWS Q Developer and Kiro now offer registry-based governance with automatic enforcement cycles, broadening enterprise options beyond GitHub Copilot. A related threat beyond rogue servers is tool shadowing within approved servers: even when only allowlisted servers are connected, a compromised or malicious server can register tool names that collide with trusted tools on other servers. Without namespace isolation or UUID-based tool identifiers, the agent may invoke the attacker-controlled version — current MCP hosts do not enforce cross-server tool uniqueness. |

## Real-World Incidents & Research (2025-2026)

- **WhatsApp MCP Tool Poisoning (April 2025):** A malicious MCP server silently exfiltrated a user's entire WhatsApp chat history through tool poisoning, exposing personal conversations and business data. One of the earliest documented cases of MCP-specific data exfiltration.
- **GitHub MCP Prompt Injection (May 2025):** A prompt-injection attack against a GitHub MCP integration enabled theft of private repository contents, internal project details, and personal financial information. Overly broad API tokens amplified the blast radius.
- **Asana MCP Cross-Tenant Leak (June 2025):** An access control flaw in an Asana MCP integration permitted cross-tenant data visibility between different customer organizations.
- **Anthropic MCP Inspector RCE (June 2025):** An unauthenticated remote code execution vulnerability in the MCP Inspector development tool exposed developer workstations' filesystems, API keys, and environment secrets.
- **CVE-2025-6514 — mcp-remote Command Injection (July 2025):** Critical command-injection bug (CVSS 10.0) in the `mcp-remote` npm package affected 437,000+ downloads. The package's OAuth proxy passed malicious endpoints directly to system shell handlers (PowerShell on Windows, shell commands on Unix), enabling credential theft, Git repository exfiltration, and full system compromise. The first documented case of full system compromise achieved through MCP infrastructure.
- **Anthropic Filesystem MCP Sandbox Escape (August 2025):** Sandbox escape vulnerabilities allowed arbitrary file access through symlink bypass and containment flaws in the official filesystem MCP server.
- **Postmark MCP Supply Chain Breach (September 2025):** Attackers injected a single line of malicious code into an npm package used for sending transactional emails through Postmark in AI automation pipelines. Compromised MCP servers silently BCC'd every outgoing email to the attackers.
- **Smithery Registry Path Traversal (October 2025):** A path-traversal bug in the Smithery.ai build configuration exposed Docker credentials controlling 3,000+ hosted MCP servers and their API tokens.
- **Figma MCP Command Injection (October 2025):** A command-injection flaw via unsanitized input in the Figma/Framelink MCP integration enabled shell execution and design data access.
- **Supabase Cursor Agent Exploit (mid-2025):** Supabase's Cursor agent, running with privileged service-role access, processed support tickets containing user-supplied input as commands. Attackers embedded SQL instructions to exfiltrate sensitive integration tokens by leaking them into a public support thread.
- **CVE-2026-21852 — Claude Code Project File RCE (January 2026):** Critical vulnerability allowing remote code execution and API credential theft through malicious project configuration files.
- **MCP CVE Surge (early 2026):** 30+ CVEs filed against MCP implementations in a 60-day period, covering SSRF, command injection, path traversal, and authentication bypass. Over 40% of public MCP servers tested were susceptible to command injection.
- **8,000+ Exposed MCP Servers (2026):** Research identified over 8,000 publicly exposed MCP servers, with 492 lacking basic authentication or encryption.
- **GitGuardian Secrets Report (2026):** 24,008 unique secrets found in MCP-related configuration files across public GitHub, including 2,117 valid credentials. AI coding tools correlated with a doubling of secret leak rates.
- **mcp-server-git CVE Chain (CVE-2025-68145/68143/68144):** Three chained vulnerabilities in Anthropic's own `mcp-server-git` package combined with the Filesystem MCP server to achieve full remote code execution via malicious `.git/config` files. Demonstrated that even vendor-maintained MCP components can harbor critical security flaws.
- **CoSAI MCP Threat Analysis (early 2026):** The Coalition for Secure AI identified 40+ distinct MCP threat categories. Most organizations were addressing fewer than two categories, suggesting widespread underestimation of MCP attack surface.
- **Tool Poisoning Prevalence (2026):** Invariant Labs research found 5.5% of MCP servers in the wild exhibit tool poisoning characteristics — malicious instructions hidden in tool descriptions that are invisible to human users but processed by AI models.
- **Shadow MCP Servers (2026):** Developers increasingly connect MCP servers to IDEs without IT oversight, granting file system and shell access outside traditional security monitoring perimeters. These "shadow MCP" deployments bypass organizational security controls entirely.
- **MCPTox Benchmark (2025-2026):** Research benchmark evaluating how often malicious or manipulated tool definitions pass into AI agent contexts, finding tool poisoning attacks frequently result in unauthorized execution or data leakage.
- **CVE-2026-30856 — WeKnora Tool Execution Hijacking (2026):** A naming collision flaw (CWE-706) in Tencent's WeKnora framework allowed attackers to register malicious MCP servers with tool identifiers that overwrote legitimate tool references in memory, enabling indirect prompt injection and data exfiltration. Fixed in v0.3.0 with UUID-based tool identifiers and "first-wins" registration policy.
- **Unit 42 MCP Sampling Attack Vectors (2026):** Palo Alto Networks' Unit 42 research identified three novel attack vectors through MCP's sampling feature: resource theft via hidden prompts (servers inject covert instructions causing LLMs to generate unauthorized content), persistent conversation hijacking (injected directives alter AI behavior across multiple turns), and covert tool invocation (servers manipulate prompts to trigger unauthorized file system operations). These attacks exploit the bidirectional control reversal inherent in MCP sampling, where servers become active requesters of LLM completions with minimal oversight.
- **Qualys TotalAI Shadow MCP Discovery (March 2026):** Qualys research found MCP servers frequently evade traditional security tools by binding to localhost, using random high ports, or hiding behind proxies. Over 10,000 public MCP servers appeared within one year of the protocol's introduction, with many operating outside IT governance perimeters.

## Open Research Questions

- [ ] Should the MCP ecosystem adopt a standardized package-signing mechanism? MCPS proposes ECDSA P-256 "Agent Passports" with five trust levels, but adoption is early-stage — is this the right foundation, or should the MCP spec itself incorporate signing requirements?
- [ ] How should MCP configuration files handle secret references in a cross-platform, cross-SDK way? Only ~8.5% of MCP implementations currently use OAuth 2.1 — what would it take to make secretless access the default?
- [ ] Is there a viable model for an MCP server registry with enforced integrity metadata? GitHub, AWS Q Developer, and Kiro now offer registry-based enforcement, but all rely on name-based matching rather than cryptographic attestation — can the ecosystem converge on a standard that ties server identifiers to verified binaries?
- [ ] How do you verify integrity of MCP servers that are dynamically discovered at runtime rather than pre-configured? MCPS's signed tool definitions are one approach, but runtime discovery introduces trust bootstrapping challenges.
- [ ] Can MCPTox-style automated benchmarking be integrated into CI/CD pipelines to detect tool poisoning before deployment?
- [ ] The MCP gateway debate: centralized gateways offer unified enforcement but create single points of failure. Should per-server enforcement (signed envelopes, per-connection authentication) replace or complement gateway architectures?
- [ ] How should organizations handle the "contextual secret leakage" problem, where credentials persist in agent context memory across sessions?
- [ ] Should AI-BOM (AI Bill of Materials) standards extend to MCP server dependencies? Traditional SBOMs miss model versions, training data lineage, and agentic dependency chains — is CycloneDX/SPDX sufficient, or does the MCP ecosystem need purpose-built transparency formats?
- [ ] How should organizations detect and govern "shadow MCP" deployments where developers connect servers to IDEs without IT oversight, bypassing security monitoring? Qualys TotalAI's three-layer detection (network/host/supply-chain) is one approach — are there lighter-weight alternatives?
- [ ] MCP sampling creates a bidirectional control reversal where servers can request LLM completions — should the protocol require explicit user consent for sampling requests, or can capability declarations and prompt sandboxing suffice?
- [ ] Tool shadowing via namespace collisions (demonstrated by CVE-2026-30856) is a gap in the MCP spec. Should tool identifiers be globally unique (UUID-based) by default, or is registry-backed namespace isolation sufficient?
- [ ] NIST's AI Agent Standards Initiative (February 2026) is collecting input on agent security and identity — should the MCP community align with NIST's forthcoming framework for AI agent identity and authorization, or develop MCP-specific standards independently?
