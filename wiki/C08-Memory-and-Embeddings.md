# C08: Memory, Embeddings & Vector Database Security

> **Source:** [`1.0/en/0x10-C08-Memory-Embeddings-and-Vector-Database.md`](https://github.com/OWASP/AISVS/blob/main/1.0/en/0x10-C08-Memory-Embeddings-and-Vector-Database.md)
> **Requirements:** 25 | **Sections:** 5

## Control Objective

Embeddings and vector stores act as semi-persistent and persistent 'memory' for AI systems via RAG. This memory can become a high-risk data sink and data exfiltration path. This control family hardens memory pipelines and vector databases so that access is least-privilege, data is sanitized before vectorization, retention is explicit, and systems are resilient to embedding inversion, membership inference, and cross-tenant leakage.

## C8.1 Access Controls on Memory & RAG Indices

Enforce fine-grained access controls and query-time scope enforcement for every vector collection.

| # | Requirement | Level | Role | Threat Mitigated | Verification Approach | Gaps / Notes |
|---|-------------|:-----:|:----:|-----------------|----------------------|--------------|
| **8.1.1** | **Verify that** vector insert, update, delete, and query operations are enforced with namespace/collection/document-tag scope controls (e.g., tenant ID, user ID, data classification labels) with default-deny. | 1 | D/V | Cross-tenant data leakage via unrestricted vector queries; unauthorized document retrieval where Tenant A's similarity search returns Tenant B's chunks. Directly maps to OWASP LLM08:2025. | Review vector DB configuration for namespace isolation. Attempt cross-namespace queries with test credentials. Confirm default-deny by issuing queries without explicit scope and verifying rejection. Inspect middleware/SDK wrappers that append scope filters. | Pinecone supports namespaces natively; Weaviate has multi-tenancy since v1.20; Qdrant added payload-based filtering but lacks native RBAC on collections. Many teams rely on application-layer enforcement only, which is fragile. |
| **8.1.2** | **Verify that** API credentials used for vector operations carry scoped claims (e.g., permitted collections, allowed verbs, tenant binding). | 1 | D/V | Credential misuse or lateral movement — a compromised API key with full read/write across all collections enables mass data exfiltration or poisoning of the entire knowledge base. | Inspect API key/token scopes in the vector DB management console. Verify JWT claims or API key metadata restrict operations to specific collections and verbs. Test that a scoped key cannot perform out-of-scope operations. | Most managed vector DBs (Pinecone, Weaviate Cloud) support API key scoping. Self-hosted deployments (Milvus, Chroma) often have a single master key with no granular scoping — compensating controls at the API gateway or service mesh layer are needed. |
| **8.1.3** | **Verify that** cross-scope access attempts (e.g., cross-tenant similarity queries, namespace traversal, tag bypass) are detected and rejected. | 2 | D/V | Namespace traversal attacks where an attacker crafts queries or metadata filters to escape their tenant boundary. Similar to SQL injection but targeting vector query DSLs and metadata filter syntax. | Perform penetration testing with crafted metadata filter payloads attempting to bypass tenant scoping (e.g., injecting `OR tenant_id != 'attacker'` in filter expressions). Review logs for cross-scope query attempts. Validate that alerting fires on detection. | Vector query languages vary widely — Weaviate uses GraphQL filters, Pinecone uses JSON filter syntax, Qdrant uses its own filter DSL. Each needs separate injection testing. Filter injection is an emerging attack class with limited tooling for automated testing. |
| **8.1.4** | **Verify that** every ingested document is tagged at write time with source, writer identity (authenticated user or system principal), timestamp, batch ID, and embedding model version, and that these tags are immutable after initial write. | 2 | D/V | Data provenance attacks — without lineage metadata, poisoned vectors cannot be traced back to their source for quarantine or removal. Also prevents "ghost documents" inserted by compromised ingestion pipelines. Maps to MITRE ATLAS AML.T0071 (False RAG Entry Injection). | Inspect ingestion pipeline code for metadata attachment at write time. Query stored vectors and confirm all required metadata fields are present. Attempt to update immutable metadata fields via API and verify rejection. Audit batch ingestion logs for completeness. | Immutability enforcement varies: Pinecone metadata can be overwritten via upsert; Weaviate properties can be updated unless schema-level restrictions are set. True immutability may require append-only storage patterns or write-once object storage for metadata alongside the vector index. |
| **8.1.5** | **Verify that** RAG pipeline retrieval events log the query issued, the documents or chunks retrieved, similarity scores, the knowledge source, and whether retrieved content passed prompt injection scanning before being incorporated into model context. | 2 | D/V | Undetected RAG poisoning — without retrieval logging, poisoned documents that influence model outputs cannot be identified forensically. Also enables detection of indirect prompt injection via retrieved content (OWASP LLM01:2025). | Review logging pipeline configuration. Issue test queries and confirm logs contain all required fields (query text, retrieved doc IDs, similarity scores, source, injection scan result). Verify logs are shipped to a SIEM or centralized logging system. Test log completeness under high-throughput conditions. | Logging similarity scores and retrieved chunks at scale generates significant data volume. Teams need to balance completeness with storage costs. LlamaIndex and LangChain both support callback-based logging but require custom integration for injection scan results. Consider sampling strategies for Level 2 compliance. |
| **8.1.6** | **Verify that** retrieval anomaly detection identifies embedding density outliers, repeated dominance of specific documents in similarity results, and sudden shifts in retrieval bias distribution that may indicate vector database poisoning. | 3 | D/V | Sophisticated vector DB poisoning where an attacker injects many semantically similar vectors to dominate retrieval results for target queries, steering model outputs. Maps to MITRE ATLAS AML.T0070 (RAG Poisoning). The attacker creates a "gravity well" in embedding space. | Review anomaly detection implementation. Verify it monitors: (1) document retrieval frequency distributions, (2) embedding space density metrics per namespace, (3) temporal shifts in which documents are retrieved for stable query sets. Test with synthetic poisoning scenarios and confirm alerts fire. | This is an emerging area with limited production tooling. Custom implementations typically use statistical process control on retrieval frequency histograms or embedding space density estimation (e.g., k-distance plots). No major vector DB offers this natively. Research from Song et al. (2020) on backdoor detection in embeddings is relevant but not directly productized. |

## C8.2 Embedding Sanitization & Validation

Pre-screen content before vectorization; treat memory writes as untrusted inputs; prevent ingestion of unsafe payloads.

| # | Requirement | Level | Role | Threat Mitigated | Verification Approach | Gaps / Notes |
|---|-------------|:-----:|:----:|-----------------|----------------------|--------------|
| **8.2.1** | **Verify that** regulated data and sensitive fields are detected prior to embedding and are masked, tokenized, transformed, or dropped based on policy. | 1 | D/V | PII/PHI/PCI data leakage via embeddings — once sensitive text is embedded, the original content can potentially be reconstructed via inversion attacks, and the vector store becomes a regulated data repository subject to GDPR/CCPA deletion requirements. Maps to OWASP LLM02:2025. | Review the ingestion pipeline for DLP/PII scanning stages before the embedding step. Test with synthetic documents containing known PII patterns (SSNs, credit card numbers, email addresses). Verify sensitive fields are masked or dropped in both the stored text chunks and metadata. | Microsoft Presidio, AWS Comprehend, and Google DLP API can be integrated into ingestion pipelines. The challenge is that NER-based PII detection has recall gaps for domain-specific sensitive data (e.g., medical record numbers, internal project codes). Custom entity recognizers are often needed. Embedding the masked version may reduce retrieval quality for legitimate queries. |
| **8.2.2** | **Verify that** embedding ingestion rejects or quarantines inputs that violate required content constraints (e.g., non-UTF-8, malformed encodings, oversized payloads, invisible Unicode characters, or executable content intended to poison retrieval). | 1 | D/V | Indirect prompt injection via RAG — attackers inject adversarial instructions into documents that get embedded and later retrieved into the model's context window. Invisible Unicode characters (e.g., zero-width joiners, right-to-left overrides) can hide malicious instructions from human reviewers while remaining effective against LLMs. | Submit test documents containing: non-UTF-8 byte sequences, invisible Unicode (U+200B, U+200E, U+FEFF), oversized payloads exceeding defined limits, documents with embedded scripts or encoded executable content. Verify each is rejected or quarantined. Inspect quarantine storage for review capability. | Content constraint enforcement is straightforward for encoding and size checks but harder for semantic attacks. Invisible Unicode detection requires explicit character class filtering. "Executable content" in the context of RAG typically means prompt injection payloads, which require LLM-based classifiers (e.g., Rebuff, Lakera Guard) rather than simple pattern matching. |
| **8.2.3** | **Verify that** vectors that fall outside normal clustering patterns are flagged and quarantined before entering production indices. | 2 | D/V | Adversarial embedding injection — an attacker crafts vectors positioned in unusual regions of embedding space to manipulate retrieval behavior. Out-of-distribution vectors may be designed to be retrieved for a wide range of queries (high-centrality poisoning) or to target specific query clusters. | Review the outlier detection mechanism in the ingestion pipeline. Verify it uses statistical methods (e.g., z-score on cosine distance to cluster centroids, isolation forest, or DBSCAN-based novelty detection). Inject synthetic outlier vectors and confirm they are quarantined. Check that quarantine does not block legitimate novel content without review. | Defining "normal clustering patterns" requires a baseline — cold-start scenarios or rapidly evolving knowledge bases make this challenging. False positive rates for outlier detection in high-dimensional spaces can be high. Dimensionality reduction (UMAP/t-SNE) before outlier detection is common but lossy. Consider adaptive baselines that update with verified-clean ingestion batches. |
| **8.2.4** | **Verify that** an agent's own outputs are not automatically written back into its trusted memory without explicit validation (such as content-origin checks or write-authorization controls that verify the content's source before committing writes). | 2 | D/V | Memory poisoning feedback loops — if an agent can write its own (potentially hallucinated or manipulated) outputs back into its memory, an attacker can use a single successful prompt injection to permanently corrupt the agent's knowledge base. This creates a persistent backdoor that survives conversation resets. | Review agent architecture for write-back paths from agent output to memory stores. Verify that content-origin tagging distinguishes user-provided content, retrieved content, and agent-generated content. Test that agent-generated content requires explicit approval or a separate validation pipeline before memory write. | This is critical for autonomous agents (e.g., AutoGPT-style systems) that have self-modification capabilities. LangChain's memory modules (ConversationBufferMemory, ConversationSummaryMemory) write back by default without origin checks. Custom guardrails are needed. The distinction between "agent output" and "validated knowledge" must be architecturally enforced. |
| **8.2.5** | **Verify that** new content written to memory is checked for contradictions with what is already stored and that conflicts trigger alerts. | 3 | D/V | Knowledge base corruption and slow-burn poisoning — an attacker gradually introduces contradictory information to degrade the reliability of RAG responses. Without contradiction detection, conflicting facts coexist silently, and the model may retrieve either version unpredictably, undermining trust in the system. | Review contradiction detection implementation (e.g., NLI-based entailment checking, semantic similarity with polarity analysis). Test by ingesting documents that directly contradict existing stored content. Verify alerts fire and conflicting content is flagged for human review rather than silently merged. | Automated contradiction detection at scale is an unsolved problem in NLP. Current approaches using Natural Language Inference (NLI) models (e.g., DeBERTa fine-tuned on MNLI) work for explicit contradictions but miss subtle or contextual conflicts. Cross-document consistency checking is computationally expensive — O(n^2) pairwise comparisons in the worst case. Practical implementations focus on checking new content against a curated "golden" knowledge set. |

## C8.3 Memory Expiry, Revocation & Deletion

Retention must be explicit and enforceable; deletions must propagate to derived indices and caches.

| # | Requirement | Level | Role | Threat Mitigated | Verification Approach | Gaps / Notes |
|---|-------------|:-----:|:----:|-----------------|----------------------|--------------|
| **8.3.1** | **Verify that** retention times are applied to every stored vector and related metadata across memory storage. | 1 | D/V | Unbounded data accumulation — without TTLs or retention policies, vector stores grow indefinitely, increasing the attack surface for data exfiltration and complicating regulatory compliance (GDPR Article 5(1)(e) storage limitation principle). Stale data also degrades retrieval quality. | Review vector store configuration for TTL settings or retention policy enforcement. Query stored vectors and verify TTL metadata is present. Check for automated expiration jobs or garbage collection processes. Verify retention policies cover both vector data and associated metadata/text chunks. | Pinecone does not support native TTL — requires application-layer deletion jobs. Weaviate supports TTL via `autoTenantDeletion` for multi-tenant setups. Milvus added TTL support in v2.3. Redis Vector Search inherits Redis key TTL. For databases without native TTL, a separate cron-based cleanup process scanning metadata timestamps is the standard pattern. |
| **8.3.2** | **Verify that** only information required for the system's defined function is persisted in memory (such as user preferences and conversation decisions, not credentials or full conversation transcripts), and that context not needed beyond the current session is discarded at session end. | 1 | D/V | Over-retention of sensitive conversational data — full conversation transcripts in memory create a rich target for exfiltration and may contain credentials, personal disclosures, or business secrets shared in context. Maps to data minimization principles (GDPR Article 5(1)(c)). | Review memory write logic to identify what data categories are persisted. Compare persisted data against the system's functional requirements. Verify session-scoped memory is cleared at session termination. Test by conducting a session with sensitive content, ending it, and confirming the data is not retrievable in a new session. | Defining "required for the system's defined function" needs explicit documentation — a data retention specification. ChatGPT's memory feature, for example, persists user preferences but not full transcripts. Systems using ConversationBufferMemory in LangChain persist everything by default; ConversationSummaryMemory is better but may still retain sensitive details in summaries. |
| **8.3.3** | **Verify that** deletion requests purge vectors, metadata, cache copies, and derivative indices within an organization-defined maximum time. | 1 | D/V | Right-to-erasure non-compliance (GDPR Article 17) and residual data exposure — if deletion only removes the vector but leaves metadata, cached text chunks, or derived indices (e.g., BM25 keyword indices alongside vector indices), the sensitive content remains accessible through alternative retrieval paths. | Issue a deletion request for a known document. After the defined maximum time, attempt to retrieve the document via: (1) vector similarity search, (2) metadata filter query, (3) any keyword/BM25 index, (4) application-layer caches. Verify all paths return no results. Check backup systems for residual copies. | Hybrid search architectures (vector + keyword) are increasingly common (e.g., Weaviate's hybrid search, Pinecone's sparse-dense). Each index must be covered by deletion propagation. Cache invalidation is particularly challenging — Redis caches, CDN-layer caches, and in-memory application caches all need explicit purge logic. Document the full data flow map before implementing deletion. |
| **8.3.4** | **Verify that** deleted or expired vectors are removed reliably and are unrecoverable. | 2 | D | Data remanence in vector stores — soft-delete implementations or lazy garbage collection may leave vector data recoverable through direct storage-layer access, low-level API calls, or backup restoration even after logical deletion. | Verify the deletion mechanism is hard-delete, not soft-delete/tombstone with deferred cleanup. For HNSW-based indices, confirm that deleted node slots are overwritten on compaction. Check that storage-layer files (e.g., segment files in Milvus, WAL in Qdrant) do not retain deleted data after compaction. Review backup retention policies. | Most vector databases use HNSW or IVF indices that do not immediately reclaim deleted vector slots — they mark them as deleted and reclaim space during compaction/merge operations. Qdrant uses WAL with periodic snapshots; deleted data persists in WAL until the next snapshot. Milvus requires explicit `compact()` calls. True unrecoverability may require encryption with key destruction (crypto-shredding). |
| **8.3.5** | **Verify that** expired vectors are excluded from retrieval results within a measured and monitored propagation window. | 2 | D/V | Stale data serving — expired or deleted vectors that remain retrievable during a propagation window can serve outdated, revoked, or sensitive content. The window must be defined, measured, and kept as short as possible. | Define the maximum acceptable propagation delay. Issue a deletion/expiration, then immediately query for the affected vectors. Measure the actual time until they stop appearing in results. Verify monitoring/alerting exists for propagation delays exceeding the defined maximum. | In distributed vector databases, propagation delay depends on replication lag, cache TTLs, and index rebuild frequency. Pinecone's serverless architecture has eventual consistency with typical delays under 10 seconds. Weaviate's distributed mode uses Raft consensus but query-time filtering may still serve stale replicas. Application-layer caching adds another propagation layer that must be accounted for. |
| **8.3.6** | **Verify that** memory can be reset for security reasons (quarantine, selective purge, full reset) separately from retention deletion, and that quarantined content is kept for investigation but excluded from retrieval. | 2 | D/V | Incident response capability — when a memory poisoning attack is detected, the team needs to quarantine suspected vectors immediately (excluding them from retrieval) while preserving them for forensic analysis. Without this separation, teams face a choice between continued exposure and evidence destruction. | Simulate a poisoning incident. Verify that quarantine operations: (1) immediately exclude targeted vectors from all retrieval paths, (2) preserve quarantined vectors in a separate, access-controlled storage for investigation, (3) do not trigger retention-based deletion timers. Test selective purge (specific documents/batches) and full reset independently. | Most vector databases do not have native quarantine functionality. Implementation patterns include: moving vectors to a separate "quarantine" collection/namespace, adding a metadata flag and enforcing query-time filtering, or maintaining a deny-list of vector IDs checked at retrieval time. The deny-list approach has the lowest latency impact but requires consistent enforcement across all query paths. |

## C8.4 Prevent Embedding Inversion & Leakage

Address inversion, membership inference, and attribute inference with explicit threat modeling, mitigations, and regression testing gates.

| # | Requirement | Level | Role | Threat Mitigated | Verification Approach | Gaps / Notes |
|---|-------------|:-----:|:----:|-----------------|----------------------|--------------|
| **8.4.1** | **Verify that** sensitive vector collections are protected against direct read access by infrastructure administrators via technical controls such as application-layer encryption, envelope encryption with strict KMS policies, or equivalent compensating controls. | 2 | D/V | Insider threat and infrastructure compromise — database administrators or operators with storage-layer access can dump raw vectors and apply inversion attacks to reconstruct sensitive source text. Embedding inversion research (Morris et al., 2023, "Text Embeddings Reveal Almost as Much as Text") demonstrated high-fidelity text reconstruction from embeddings. Maps to MITRE ATLAS AML.T0024.001. | Verify encryption at rest covers vector data files specifically (not just general disk encryption). Check KMS policies restrict key access to application service accounts, not infrastructure admin roles. Test that a simulated admin with storage access but no KMS key access cannot read vector data. Review envelope encryption implementation if used. | Standard database encryption at rest (e.g., AWS EBS encryption, GCP CMEK) protects against physical media theft but not against privileged administrators who can access the decrypted database process. Application-layer encryption (encrypting vectors before storage) is stronger but breaks similarity search — the vector DB cannot compute cosine similarity on encrypted vectors. Practical approaches include: encrypted metadata with cleartext vectors (partial protection), or using encrypted search schemes (still largely research-stage for vector similarity). |
| **8.4.2** | **Verify that** privacy/utility targets for embedding leakage resistance are defined and measured, and that changes to embedding models, tokenizers, retrieval settings, or privacy transforms are gated by regression tests against those targets. | 3 | D/V | Embedding inversion and membership inference — attackers with access to embedding vectors can reconstruct source text (inversion) or determine whether specific documents are in the corpus (membership inference). Privacy transforms (e.g., dimensionality reduction, noise addition, quantization) reduce these risks but degrade retrieval utility. Without defined targets and regression testing, model or config changes can silently weaken privacy guarantees. | Review documented privacy/utility targets (e.g., "inversion attack accuracy below X% at retrieval recall above Y%"). Verify a regression test suite exists that measures both metrics. Test that pipeline changes (model swap, tokenizer update, dimension change) trigger the regression suite and block deployment if targets are not met. | This is a research-frontier requirement. Key papers: Morris et al. (2023) on text embedding inversion, Song & Raghunathan (2020) on membership inference for embeddings. Practical mitigations include: dimensionality reduction (PCA/random projection — reduces inversion fidelity), additive Gaussian noise (calibrated to privacy budget), vector quantization (lossy compression). No standard benchmark suite exists for measuring embedding privacy — teams must build custom evaluation harnesses. Differential privacy for embeddings (DP-SGD during fine-tuning) is theoretically sound but impacts embedding quality significantly. |

## C8.5 Scope Enforcement for User-Specific Memory

Prevent cross-tenant and cross-user leakage in retrieval and prompt assembly.

| # | Requirement | Level | Role | Threat Mitigated | Verification Approach | Gaps / Notes |
|---|-------------|:-----:|:----:|-----------------|----------------------|--------------|
| **8.5.1** | **Verify that** every retrieval operation enforces scope constraints (tenant/user/classification) in the vector engine query and verifies them again before prompt assembly (post-filter). | 1 | D/V | Cross-tenant data leakage in RAG — single-layer filtering (either at query time or post-retrieval, but not both) is vulnerable to bypass. Query-time filter bypass can occur through metadata filter injection; post-filter-only approaches waste resources retrieving unauthorized documents and may leak information through timing side channels. Defense-in-depth requires both layers. | Trace a retrieval request end-to-end: verify scope filters are applied in the vector DB query (pre-filter). Then verify a second scope check occurs in the application layer before retrieved chunks are assembled into the prompt. Test with a scenario where query-time filtering is deliberately weakened — confirm post-filter catches the leakage. | Pre-filtering vs. post-filtering has performance implications. Weaviate and Qdrant support filtered vector search (pre-filter) natively. Pinecone supports metadata filtering at query time. The post-filter layer is always application code — typically in the RAG orchestration framework (LangChain, LlamaIndex). Ensure both layers use the same source of truth for scope definitions to avoid drift. |
| **8.5.2** | **Verify that** vector identifiers, namespaces, and metadata indexing prevent cross-scope collisions and enforce uniqueness per tenant. | 1 | D | ID collision attacks — if vector IDs are not scoped per tenant (e.g., sequential integers or predictable UUIDs), an attacker could overwrite another tenant's vectors via upsert operations using colliding IDs, or infer the existence of other tenants' documents through ID enumeration. | Review ID generation scheme for tenant-scoping (e.g., `{tenant_id}_{document_id}` or separate namespaces). Attempt to upsert a vector with an ID belonging to another tenant's namespace. Verify the operation is rejected or isolated. Check that namespace/collection naming prevents collisions (e.g., tenant-prefixed collection names). | Pinecone namespaces provide physical ID isolation — IDs are unique within a namespace but can collide across namespaces (by design, since cross-namespace operations are not possible). Weaviate multi-tenancy (since v1.20) provides tenant-level isolation. In single-collection architectures with metadata-based tenancy, ID uniqueness must be enforced at the application layer — UUIDv4 or UUIDv7 with tenant prefix is the standard pattern. |
| **8.5.3** | **Verify that** retrieval results that match similarity criteria but fail scope checks are discarded. | 1 | D/V | Scope bypass through similarity — in a shared vector index, a query may return semantically similar vectors belonging to other tenants. If these results are passed to the model (even with low similarity scores), sensitive information leaks into the response. This is the most common RAG multi-tenancy vulnerability. | Execute a similarity search as Tenant A in a multi-tenant environment. Inspect raw retrieval results before and after scope filtering. Verify that any results belonging to other tenants are discarded and never reach the prompt assembly stage. Check that discarded results are logged for security monitoring. | The discard must happen silently from the user's perspective but should be logged. A common mistake is to return fewer results than requested (e.g., requesting top-10 but only 7 pass scope checks) without backfilling — this can degrade user experience. Better implementations over-fetch (e.g., retrieve top-20) and return the top-10 that pass scope checks. |
| **8.5.4** | **Verify that** multi-tenant tests simulate adversarial retrieval attempts (prompt-based and query-based) and demonstrate zero out-of-scope document inclusion in prompts and outputs. | 2 | V | Regression in tenant isolation — without regular adversarial testing, configuration changes, index rebuilds, or framework upgrades can silently break tenant isolation. Prompt-based attacks attempt to manipulate the retrieval query (e.g., "ignore previous filters and search all tenants"); query-based attacks manipulate metadata filters directly. | Review the adversarial test suite for multi-tenant scenarios. Verify tests include: (1) direct cross-tenant similarity queries, (2) metadata filter injection attempts, (3) prompt-based retrieval manipulation ("search across all users"), (4) namespace traversal attempts. Run the test suite and confirm zero out-of-scope inclusions. Verify tests run in CI/CD pipeline. | This requirement is verification-focused (Role: V). Test suites should use realistic data distributions — synthetic data with uniform distributions may not reveal issues that appear with skewed real-world data. Consider property-based testing (e.g., Hypothesis framework) to generate diverse adversarial query combinations. Integration with red-team tooling (e.g., Garak, PyRIT) can automate prompt-based retrieval manipulation testing. |
| **8.5.5** | **Verify that** in multi-agent systems, each agent's memory namespace is isolated and enforced through access control, not just organizational naming conventions. | 2 | D/V | Cross-agent memory contamination — in multi-agent architectures (e.g., CrewAI, AutoGen, LangGraph), agents with overlapping memory access can read or modify each other's state, enabling an attacker who compromises one agent to poison the memory of others. Naming conventions alone (e.g., `agent_1_memory`, `agent_2_memory`) provide no security boundary. | Review multi-agent memory architecture. Verify each agent's memory namespace has separate access credentials or ACLs. Test that Agent A's credentials cannot read or write to Agent B's memory namespace. Verify this is enforced at the vector DB or middleware layer, not just by convention in application code. | Most multi-agent frameworks (CrewAI, AutoGen) use shared memory by design for inter-agent communication. Isolating memory while still allowing controlled sharing requires an explicit architecture — e.g., a "shared" namespace with write controls and per-agent private namespaces. LangGraph's state management can enforce this if properly configured. The key distinction is access-controlled sharing vs. unrestricted sharing. |
| **8.5.6** | **Verify that** encryption keys and access policies are segregated per tenant for memory/vector storage, providing cryptographic isolation in shared infrastructure. | 3 | D/V | Cryptographic tenant isolation — in shared-infrastructure deployments, a single encryption key across all tenants means that a key compromise exposes all tenants' data. Per-tenant keys ensure that a breach affecting one tenant's key material does not compromise others. This is the strongest form of multi-tenant isolation short of dedicated infrastructure. | Verify each tenant has a distinct encryption key in the KMS. Confirm key access policies restrict each tenant's key to that tenant's service principal only. Test that rotating or revoking one tenant's key does not affect other tenants' data availability. Review key hierarchy (master key wrapping per-tenant DEKs is acceptable). | Per-tenant encryption in vector databases is operationally complex: key management overhead scales linearly with tenant count, and index-level encryption (encrypting entire collections per tenant) requires separate indices per tenant. Envelope encryption with per-tenant DEKs wrapped by a shared KEK in HSM/KMS is the practical middle ground. Pinecone and Weaviate Cloud manage encryption transparently but do not offer per-tenant key segregation — this requires self-hosted deployments or custom encryption layers. |

## Threat Landscape

Known attacks, real-world incidents, and threat vectors relevant to this chapter:

- RAG poisoning — injecting malicious content into knowledge bases retrieved at inference time
- Embedding inversion attacks — reconstructing original text from embedding vectors
- Cross-tenant data leakage through shared vector indices
- Memory persistence attacks — injecting instructions that persist across conversations
- PII accumulation in vector stores without adequate deletion mechanisms
- Membership inference — determining whether a specific document exists in an embedded corpus
- Feedback loop poisoning — agent outputs written back to memory create self-reinforcing hallucinations
- Metadata filter injection — manipulating vector query filters to bypass tenant isolation

### Notable Incidents & Research

| Date | Incident / Paper | Relevance | Link |
|------|------------------|-----------|------|
| 2023 | Morris et al. — "Text Embeddings Reveal Almost as Much as Text" | Demonstrated high-fidelity text reconstruction from OpenAI and other embeddings, establishing embedding inversion as a practical threat | [arXiv:2310.06816](https://arxiv.org/abs/2310.06816) |
| 2023 | Greshake et al. — "Not What You've Signed Up For: Compromising RAG" | Showed indirect prompt injection through poisoned retrieval documents in RAG pipelines | [arXiv:2302.12173](https://arxiv.org/abs/2302.12173) |
| 2024 | Zou et al. — "PoisonedRAG: Knowledge Poisoning Attacks to RAG" | Demonstrated targeted knowledge poisoning attacks against RAG systems with high success rates | [arXiv:2402.07867](https://arxiv.org/abs/2402.07867) |
| 2024 | Zeng et al. — "Good and Bad Memories in RAG" | Analyzed how persistent memory poisoning creates lasting effects across sessions | [arXiv:2404.01096](https://arxiv.org/abs/2404.01096) |

## Tooling & Implementation

Current tools, frameworks, and libraries that help implement these controls:

- **Vector databases:** Pinecone (with namespaces), Weaviate (with multi-tenancy and RBAC since v1.20), Qdrant (with payload filtering and API key scoping), Milvus (with RBAC and partition keys)
- **Embedding security:** dimensionality reduction via PCA/random projection, additive noise calibrated to privacy budgets, vector quantization
- **RAG frameworks:** LlamaIndex (with metadata filtering, callback logging), LangChain (with retrievers supporting metadata-based access control)
- **PII detection:** Microsoft Presidio, AWS Comprehend, Google DLP API — integrate pre-embedding in ingestion pipelines
- **Prompt injection detection in retrieved content:** Rebuff, Lakera Guard, custom NLI-based classifiers
- **Data cleanup:** TTL-based expiration (Milvus, Redis), application-layer deletion jobs (Pinecone), tombstone-based deletion with compaction

### Implementation Maturity

| Control Area | Tooling Maturity | Notes |
|--------------|:---:|-------|
| C8.1 Access Controls on Memory & RAG Indices | Medium | Namespaces and API key scoping available in major managed DBs; self-hosted deployments lag behind. Anomaly detection (8.1.6) has no off-the-shelf tooling. |
| C8.2 Embedding Sanitization & Validation | Medium | PII detection and encoding validation are well-supported. Outlier detection and contradiction checking require custom implementations. |
| C8.3 Memory Expiry, Revocation & Deletion | Low-Medium | Native TTL support is inconsistent across vector DBs. Deletion propagation to derived indices and caches requires custom engineering. Crypto-shredding support is rare. |
| C8.4 Prevent Embedding Inversion & Leakage | Low | Application-layer encryption breaks similarity search. Privacy/utility regression testing is research-stage. No standard benchmarks exist. |
| C8.5 Scope Enforcement for User-Specific Memory | Medium | Pre-filter and post-filter patterns are well-understood. Per-tenant encryption and multi-agent isolation require significant custom work. |
