# C2.6 Input Rate Limiting & Abuse Prevention

> **Parent:** [C02: User Input Validation](C02-User-Input-Validation)

## Purpose

AI systems are expensive to operate (each inference consumes GPU resources and incurs API costs) and are attractive targets for denial-of-service attacks, brute-force prompt injection attempts, and automated abuse. Rate limiting and abuse detection prevent resource exhaustion, budget overruns, and systematic probing. In agentic architectures, additional controls are needed to prevent runaway planning loops and unbounded tool call chains that can consume resources without explicit user action.

The threat landscape has shifted significantly since 2024. Attackers now target AI infrastructure not just for disruption but for profit — stealing cloud credentials to resell LLM access (LLMjacking), crafting inputs that maximize compute cost (sponge examples, energy-latency attacks), and exploiting the high per-token cost of generative AI to inflict "denial of wallet" damage that can reach tens of thousands of dollars per day. As of early 2026, the scale of active reconnaissance is staggering: GreyNoise documented over 91,000 attack sessions targeting exposed LLM services between October 2025 and January 2026, with attackers systematically probing nearly every major model family using deliberately benign queries to avoid triggering abuse detection. On the enterprise side, 96% of organizations report AI costs exceeding initial estimates, and 68% of digital leaders attribute budget overruns to runaway tool loops and recursive agent logic — prompting Gartner to declare in February 2026 that "FinOps is critical to maximizing ROI of AI agents."

It is worth distinguishing "denial of wallet" (DoW) from traditional denial of service (DoS). DoS aims to exhaust system resources causing unavailability; DoW exploits cost variance — a single user can stay well within request-per-second rate limits while running up hundreds of dollars in LLM charges. The cost variance in AI systems is extreme: a cached response might cost ~$0.0001 while an agentic workflow with tool calls can run $0.20–$0.50 per request, a 3,000x difference. A malicious actor sending 90 requests/minute at $0.30 each — comfortably under a 100 req/min rate limit — would generate $38,880/day per API key, and with 10 compromised keys that exposure reaches nearly $400K/day. This asymmetry makes request-count rate limiting alone fundamentally insufficient for AI workloads.

The MCP attack surface has also crystallized rapidly. Between April and October 2025, at least nine significant MCP security breaches were publicly documented — including tool poisoning in WhatsApp MCP (April 2025), prompt injection against the official GitHub MCP server leaking private repository data (May 2025), unauthenticated RCE in Anthropic's MCP Inspector (June 2025), OS command injection in the `mcp-remote` package affecting 437,000+ downloads (July 2025), and a path traversal in the Smithery hosting platform that leaked Docker credentials controlling 3,000+ apps (October 2025). These breaches underscore that MCP endpoints are not theoretical attack surfaces — they are actively exploited, and rate limiting is one layer of a defense-in-depth strategy that must also include authentication, input validation, and least-privilege tool scoping.

Zuplo adds cost-weighted token counting that accounts for different model pricing tiers, with separate prompt and completion token quota tracking. For MCP endpoints, the landscape has shifted dramatically in early 2026 with the emergence of dedicated MCP gateways: MintMCP Gateway (SOC 2 Type II certified, tool-level rate limiting with team-based quota enforcement); Lunar.dev MCPX (three-tier ACLs at global, service, and tool granularity, ~4ms p99 latency); Lasso Security (real-time prompt injection detection with MCP server reputation scoring, Gartner Cool Vendor 2024); Operant AI (3D Runtime Defense — Discovery, Detection, Defense — with Shadow Escape attack detection); TrueFoundry (unified LLM and MCP management with in-memory rate limiting at single-digit millisecond overhead); and Traefik Hub v3.20 (early access March 2026, general availability April 2026) introduces Task-Based Access Control (TBAC) — a three-layer authorization model (Tasks → Tools → Transactions) purpose-built for AI agents, with token-level cost controls tracking input, output, and total tokens independently, and proactive token estimation that blocks abusive requests before LLM processing. These gateways collectively move MCP rate limiting from "nonexistent" to a competitive market with enterprise-grade options. On the infrastructure side, backpressure mechanisms are emerging as a complement to rate limiting: CascadeInfer groups requests by sequence length to reduce latency by up to 67% and improve throughput 2.89x; Kthena (v2.1.0) provides KV cache-aware routing that reduced queue time by 40% and improved GPU utilization by 35%; and Crusoe MemoryAlloy (v1.0.0) achieves 9.9x faster TTFT via cluster-wide KV caching with a distributed memory fabric. These systems help prevent cascading overload even when rate limits are not yet breached. |
| **2.6.2** | **Verify that** burst and sustained rate limits are tuned to prevent DoS and brute force attacks, and that per-task budgets (for example tokens, tool/MCP calls, and cost) are enforced for agent planning loops. | 2 | D/V | Burst DoS attacks; slow-and-steady resource exhaustion ([MITRE ATLAS AML.T0034 — Cost Harvesting](https://atlas.mitre.org/)); unbounded agent execution (infinite planning loops); cost overruns from runaway agent tasks — a $47K LangChain multi-agent loop ran for 11 days before detection in November 2025; a separate $47K retry storm in February 2026 made 2.3M erroneous API calls in a single weekend; tool call amplification attacks. OWASP LLM10:2025 (Unbounded Consumption) documents resource-intensive query patterns causing 20–30x budget overruns. | 1. Review rate limit configuration for both burst (short window, e.g., 10 req/s) and sustained (longer window, e.g., 1000 req/hr) limits — use token bucket or sliding window algorithms. 2. For agent systems, verify per-task budgets exist: maximum tokens consumed, maximum tool/MCP calls, maximum cost (in dollars), and maximum execution time. 3. Test agent loop termination by creating a task that triggers recursive planning — verify it halts within budget. 4. Verify budget enforcement triggers graceful task termination with a status report, not a crash. 5. Verify agent frameworks enforce limits: LangChain/LangGraph `ModelCallLimitMiddleware` (caps model calls per thread/run), `ToolCallLimitMiddleware` (caps tool invocations globally or per specific tool), and `GraphRecursionError` for infinite graph loops; CrewAI `max_iter` (default 20), `max_execution_time`, `max_rpm`; OpenAI Assistants `max_turns`. | Agent budget enforcement has matured considerably in early 2026. The emerging industry consensus treats budgets as a four-dimensional resource — token budgets (prevent runaway prompts), step budgets (prevent infinite loops), time budgets (prevent stuck workflows), and cost budgets (prevent cascading spend) — analogous to Kubernetes CPU/memory limits. LangChain/LangGraph middleware (v1.0+) remains the most feature-complete with distinct model-call and tool-call limit middleware and configurable exit behaviors (`'end'` for graceful, `'error'` for exception). CrewAI provides per-agent `max_iter`, `max_execution_time`, and `max_rpm` controls. AutoGen uses centralized transcript management with automatic pruning when token limits approach. New entrants: AgentBudget provides real-time cost enforcement for AI agent sessions with a single-line integration; Google's BATS (Budget Aware Test-time Scaling) framework enables agents to determine when deeper reasoning is worth the cost versus pivoting to cheaper paths. The graduated response pattern (log at 50% → warn at 70% → throttle at 90% → hard stop at 100%) is becoming standard practice. Cost-based budgets are still the weakest link — most require custom integration with provider billing APIs — but the gap is closing. The **circuit breaker pattern** has emerged as a critical enforcement mechanism for agent systems: when a cost or step threshold is breached, the circuit trips and either halts execution or routes to a cheaper fallback model. Bifrost's circuit breaker detects provider failures in real-time and routes to healthy alternatives within milliseconds, tracking failure rates, latency, and errors. Arnica (as of March 2026) offers AI Budget Control with automatic API call halting the moment a budget limit is hit, per-capability budgeting across multiple providers (OpenAI, Anthropic, Google), and intelligent caching that prevents rescanning unchanged inputs. The three-level budget hierarchy — per-request `max_tokens`, per-task token/cost budgets, and per-day/per-month spending caps — is becoming the baseline architecture for production agent deployments. Traefik Hub v3.20 (March 2026) adds a composable safety pipeline with parallel guard execution across four tiers (Regex Guard, Microsoft Presidio content filtering, NVIDIA NIM jailbreak detection, IBM Granite Guardian harm detection) plus multi-provider failover routing — when a provider circuit-trips, requests automatically route to healthy alternatives while maintaining all safety policies. Cost-aware model tiering is emerging as a complementary defense: routing low-complexity tasks to cheaper models (e.g., Phi-4 Mini at $0.03/M tokens) rather than flagship models ($15–$75/M tokens) can reduce costs by 90%+ while keeping expensive models for tasks that justify the spend. Gartner projects more than 40% of agent projects will fail by 2027, with runaway costs and policy-violating agent behavior cited as primary drivers — reinforcing that enforcement infrastructure, not just visibility, is the gap. |
| **2.6.3** | **Verify that** anomalous usage patterns (e.g., rapid-fire requests, input flooding, repetitive failing tool/MCP calls or recursive agent loops) trigger automated blocks or escalations. | 2 | D/V | Automated attack campaigns; systematic vulnerability probing — GreyNoise documented 91,000+ sessions (Oct 2025–Jan 2026) probing 70+ LLM endpoints using benign-looking queries to evade detection, with 80,469 sessions from just two IPs over eleven days; distributed attacks from multiple sources; low-and-slow attacks that stay under individual rate limits but are anomalous in aggregate; agent loop exploitation; energy-latency attacks — LoopLLM (AAAI 2026) demonstrated transferable attacks achieving 90%+ of maximum output length on 12 open-source LLMs and ~40% transferability to commercial APIs; ThinkTrap (December 2025) exploits extended-thinking features to multiply cost and latency; Fill-and-Squeeze attacks (February 2026, Zhejiang University) target LLM serving framework schedulers rather than the model itself, exhausting KV cache to cause 20–280x TTFT slowdown at 30–40% lower cost than prior attack methods. | 1. Review anomaly detection rules for: request rate spikes, repeated validation failures from same source, repetitive identical or near-identical requests, high error rates from tool calls, recursive patterns in agent execution, and abnormally long output generation (energy-latency attack indicator). 2. Test with simulated anomalous patterns and verify automated response (block, CAPTCHA challenge, or escalation to SOC). 3. Verify automated blocks are time-limited (auto-expire) and can be manually overridden. 4. Confirm escalation path exists for human review. 5. Check for OWASP LLM10:2025's recommended four-stage defense: pre-ingress (WAF, IP reputation, CAPTCHA), gateway (per-tenant rate limiting, request cost estimation before processing), inference (max tokens, complexity scoring, per-request timeouts, streaming with early termination), post-inference (anomaly detection, automated budget kill-switches). | Anomaly detection goes beyond simple rate limiting — it requires behavioral analysis. Simple heuristics (>N failures in M minutes triggers block) are a good start. WAF-level AI protection is emerging: Cloudflare Firewall for AI provides prompt injection and jailbreak detection integrated with WAF; LLM Guard (Protect AI, open-source, 2.5M+ downloads) scans prompts and responses for injection, toxicity, PII, and secrets; Alibaba Cloud WAF 3.0 includes dedicated prompt injection protection rules. For agent loops, detecting recursive patterns (agent calls tool → result triggers same call) requires execution trace analysis. Energy-latency attacks (sponge examples, LoopLLM, ThinkTrap) are particularly challenging because each individual request looks legitimate — detection requires monitoring output length, latency, and token consumption per request relative to input complexity. The Fill-and-Squeeze attack (arXiv 2602.07878, February 2026) introduced a new class of system-layer DoS that targets the serving framework's KV cache scheduler rather than the model — proposed defenses include perplexity-based input filtering, fairness-aware schedulers with resource penalties, and strict output length constraints. The GreyNoise campaign showed attackers using consistent JA4H TLS fingerprints (99% of sessions), suggesting centralized automation — TLS fingerprint analysis can help identify coordinated campaigns even when individual requests appear benign. Additional bypass techniques to detect include: **header manipulation** — rate limiting systems that trust forwarded headers (X-Forwarded-For, X-Real-IP) are vulnerable to spoofing, where attackers set arbitrary values to evade per-IP limits; **recursive loop induction** — crafted prompts that encourage infinite reasoning loops consume context windows while generating minimal visible output, making them hard to detect via token-count monitoring alone; and **KV cache timing side channels** — shared KV cache optimizations, while improving performance, introduce API-visible timing differences that let adversaries infer whether their inputs match cached entries, enabling targeted probing without triggering anomaly detection. Fairness-aware scheduling is emerging as a defense layer: the Deficit Longest Prefix Match (DLPM) algorithm provides the first locality-aware fair scheduling for LLM serving, using a quantum-based deficit counter system that prevents any single client from monopolizing prefix cache locality — achieving 2.87x higher throughput than fair-only approaches while bounding allocation disparity. |

Logs should feed into a SIEM for automated pattern detection. Tools like Helicone provide cost-based observability with per-user segmentation and rate limiting by cost (e.g., "$5/hour per user"). LiteLLM Proxy tracks token/cost metrics at every organizational level. The MITRE SAFE-AI framework maps ATLAS threats (AML.T0029, AML.T0034) directly to NIST SP 800-53 controls, providing a bridge between AI-specific threat taxonomy and established security control frameworks for compliance mapping. Emerging attack pattern detection can use simple trend analysis or ML-based clustering of attack behaviors. As of March 2026, Qualys TotalAI provides discovery and assessment of MCP attack surfaces — modeling MCP servers as a graph where each node connects to exposed tools and resources, revealing transitive trust paths between otherwise isolated environments. Qualys uses layered detection: network-level probing for MCP endpoints, host-level inspection for locally accessible servers, and supply chain analysis for MCP SDK dependencies in application code. This addresses the "shadow IT" problem where MCP servers evade traditional security visibility by binding to localhost, using random ports, or hiding behind proxies — research shows 53% of MCP servers still rely on static secrets. Over 10,000 active public MCP servers now exist, many becoming production dependencies without formal security approval. Lunar.dev MCPX provides immutable audit trails designed for multi-tenant SaaS with tool-level granularity, and Operant AI's 3D Runtime Defense adds MCP-specific anomaly detection alongside its Shadow Escape attack research. |

## Implementation Maturity

| Area | Maturity | Notes |
|------|----------|-------|
| Request-count rate limiting | **High** | Standard API gateway feature (Kong, AWS API Gateway, Nginx, Cloudflare). Well-understood algorithms (token bucket, sliding window). |
| Token-aware rate limiting | **Medium–High** | As of March 2026, six major gateways support token-based rate limiting: Kong AI Rate Limiting Advanced, Envoy AI Gateway, LiteLLM Proxy, Apache APISIX `ai-rate-limiting`, Azure API Management `llm-token-limit`, and Zuplo. Bifrost adds hierarchical budget management. Cloudflare AI Gateway still does not support token-based limits. |
| Agent per-task budgets | **Medium** | The four-dimensional budget model (tokens, steps, time, cost) is now industry consensus. LangChain/LangGraph middleware, CrewAI limits, OpenAI `max_turns`, AgentBudget (real-time cost enforcement), and Google BATS all provide enforcement mechanisms. The graduated response pattern (log → warn → throttle → hard stop) is becoming standard. Cost-based budgets remain the weakest dimension. |
| Energy-latency attack detection | **Low** | Sponge examples, LoopLLM, ThinkTrap, and Fill-and-Squeeze attacks are documented in research but no production-ready detection tools exist. Fill-and-Squeeze paper proposes perplexity-based filtering and fairness-aware schedulers as defenses, but these are not yet productized. |
| MCP endpoint rate limiting | **Medium** | Rapidly maturing as of March 2026. Dedicated MCP gateways now provide tool-level rate limiting: MintMCP (SOC 2 Type II certified), Lunar.dev MCPX (three-tier ACLs), Lasso Security (reputation scoring), Operant AI (3D Runtime Defense), TrueFoundry (in-memory rate limiting), Traefik Hub v3.20 (TBAC with token-level controls), and Azure API Management (distributed rate limiting across gateway instances). Still requires integration work, but "build from scratch" is no longer necessary. |
| AI-specific WAF rules | **Medium** | Cloudflare Firewall for AI, LLM Guard (open-source), and Alibaba Cloud WAF 3.0 provide prompt injection and jailbreak detection. Not yet integrated with rate limiting decisions. |
| Circuit breaker / cost kill-switch | **Medium** | Bifrost provides real-time circuit breaking with automatic failover; AgentBudget offers single-line integration for per-session cost caps; Arnica AI Budget Control halts API calls at budget thresholds across multiple providers. The pattern is well understood (inspired by Hystrix/Resilience4j) but AI-specific implementations are still maturing. |
| Backpressure / intelligent scheduling | **Medium–Low** | CascadeInfer (length-aware scheduling), Kthena v2.1.0 (KV cache-aware routing), and Crusoe MemoryAlloy v1.0.0 (distributed memory fabric) address infrastructure-level overload, but require specialized deployment and are not yet available as drop-in gateway features. |
| Fairness-aware LLM scheduling | **Low–Medium** | The DLPM (Deficit Longest Prefix Match) algorithm provides the first locality-aware fair scheduling for LLM serving, preventing cache monopolization by malicious clients. Its distributed variant (D2LPM) maintains global fairness guarantees across GPU clusters. Research-stage but addresses a real gap — prefix cache abuse is a practical attack vector in multi-tenant serving. |
| Denial-of-wallet detection | **Low** | DoW attacks stay within request-rate limits while maximizing per-request cost. Detection requires per-request cost estimation, rolling per-user cost aggregation, and cost distribution analysis (p50/p95/p99 tracking). No turnkey solution exists — requires custom integration of cost telemetry with anomaly detection. |
| MCP shadow IT discovery | **Low–Medium** | Qualys TotalAI (March 2026) provides graph-based MCP server discovery with network probing, host inspection, and supply chain analysis. Addresses the critical visibility gap: over 10,000 public MCP servers, 53% using static secrets, many operating as unmanaged shadow IT. Without discovery, rate limiting and abuse detection are blind to these endpoints. |

## Open Research Questions

- What are appropriate rate limits for AI inference endpoints given the high per-request cost compared to traditional APIs? OWASP LLM10:2025 suggests request cost estimation before processing — how practical is this at scale?
- How should per-task budgets be set for autonomous agents where the number of steps needed is unpredictable? The $47K LangChain incident suggests fixed caps are necessary, but overly tight caps may prevent legitimate complex tasks from completing.
- Can rate limiting be made adaptive (automatically tightening under attack) without causing false positives for legitimate burst traffic? Back-pressure queuing with dynamic concurrency adjustment is one approach, but production implementations are scarce.
- How should rate limiting account for token cost variation across different models and providers? Cost-weighted token counting (as implemented by Zuplo) is promising but adds complexity.
- How can energy-latency attacks (sponge examples, LoopLLM, ThinkTrap, Fill-and-Squeeze) be detected in real time? These attacks use legitimate-looking requests that maximize compute cost — detection may require correlating input complexity with output length and latency baselines. The Fill-and-Squeeze attack specifically targets serving framework schedulers via KV cache exhaustion, which is invisible at the application layer — detection may require infrastructure-level telemetry (cache utilization, preemption rates, scheduling queue depth).
- As MCP adoption grows, what rate-limiting patterns should emerge for tool-calling endpoints? Operation Bizarre Bazaar showed 60% of attacks targeting MCP — dedicated middleware is needed but doesn't exist yet.
- Can TLS fingerprint analysis (JA4H) reliably distinguish automated reconnaissance from legitimate client traffic at AI endpoints? The GreyNoise campaign showed 99% fingerprint consistency across 80,000+ sessions — this could be a high-signal detection feature but needs validation across diverse legitimate client populations.
- How should the graduated response pattern (log → warn → throttle → hard stop) be parameterized for different AI workloads? Agent tasks may need different thresholds than interactive chat, and overly aggressive throttling can cause cascading failures in multi-agent pipelines.
- How should organizations implement denial-of-wallet detection when the cost variance between cached and agentic requests can reach 3,000x? Per-request cost estimation, rolling per-user cost aggregation at multiple time windows (1m, 5m, 1h, 1d), and cost distribution analysis (tracking p50/p95/p99) are proposed approaches, but no turnkey solution integrates all three. At what cost-variance threshold (10x? 100x? 1000x?) should organizations invest in cost-aware rate limiting versus request-count limits?
- Given the nine documented MCP breaches in seven months (April–October 2025), what minimum security baseline should MCP endpoints enforce before rate limiting is even relevant? The breaches showed attack classes ranging from tool poisoning and prompt injection to supply-chain compromise and sandbox escape — rate limiting alone cannot address this diversity, but what layered defense architecture provides adequate coverage?
- How should organizations discover and govern shadow MCP servers? Qualys TotalAI's graph-based approach (network probing + host inspection + supply chain analysis) is a start, but with 53% of MCP servers using static secrets and many binding to localhost on ephemeral ports, continuous discovery remains challenging — especially when developers embed MCP servers in their local toolchains without IT awareness.
- Is Task-Based Access Control (TBAC), as proposed by Traefik Hub, the right authorization model for AI agents, or does it introduce excessive configuration complexity? TBAC's three-layer model (Tasks → Tools → Transactions with parameter-level constraints) is more expressive than RBAC for agent workloads, but the operational burden of maintaining task definitions and transaction schemas at scale is untested.
- How can KV cache timing side channels be mitigated without sacrificing the performance benefits of cache sharing? Selective cache sharing and per-tenant isolation are possible but reduce cache hit rates. Fairness-aware scheduling (DLPM) helps prevent cache monopolization but does not address information leakage through timing differences.
