# C4.2 Secure Build & Deployment Pipelines

[Back to C04 Index](C04-Infrastructure.md)

## Purpose

Ensure cryptographic integrity and supply chain security through reproducible builds and signed artifacts. AI systems depend on complex build pipelines that pull model weights, training data, framework dependencies, and custom operators from multiple sources. A compromised build pipeline can inject backdoors into models or serving infrastructure that persist undetected through deployment. This section ensures that every artifact is traceable, verified, and tamper-evident.

## Requirements

| # | Requirement | Level | Role | Threat Mitigated | Verification Approach | Gaps / Notes |
|---|-------------|:-----:|:----:|-----------------|----------------------|--------------|
| **4.2.1** | Verify that builds are completely automated and produce a software bill of materials (SBOM). | 1 | D/V | Manual build processes introducing untracked dependencies or unauthorized modifications. Lack of visibility into what components are in production. Only 21% of organizations have complete dependency visibility (Datadog DevSecOps 2026), and that gap directly translates into untracked components in CI/CD that adversaries target through build-time injection or dependency confusion. The Sonatype 2026 State of the Software Supply Chain report found open source malware grew 75% year-over-year to over 1.2 million malicious packages, with open source consumption reaching 9.8 trillion downloads -- underscoring the scale of untracked dependencies flowing through build pipelines. | Confirm CI/CD pipeline automates the full build without manual steps. Verify SBOM generation (SPDX or CycloneDX format) is part of the build. Check SBOM includes ML framework versions, CUDA/cuDNN versions, Python packages, and native libraries. Validate that the SBOM covers AI-specific components: model architecture metadata, training dataset references, and fine-tuning lineage. For Hugging Face-hosted models, consider the [OWASP AIBOM Generator](https://genai.owasp.org/resource/owasp-aibom-generator/) which produces CycloneDX-formatted AI BOMs. | AI builds have unique SBOM challenges: model weights, training data provenance, and custom CUDA kernels are not well represented in standard SBOM formats. **CycloneDX 1.5+** introduced ML-BOM fields (model architecture family, training/inference configuration, dataset references, performance metrics, license information), and **CycloneDX 1.7 (2025)** added patent/IP metadata, citations for data provenance, and enhanced cryptographic transparency. The **OWASP AIBOM project** (targeting v0.1 in late 2025) is assessing both CycloneDX 1.6+ and SPDX 3.0 for AI-system requirements. **SPDX 3.0** includes an AI/ML profile with hyperparameters entries. Consider supplementing standard SBOM with an ML-specific manifest covering model lineage. The **AIBoMGen** research platform (published Jan 2026) demonstrates automated AIBOM generation during model training itself, capturing datasets, base models, container images, and hyperparameters with cryptographic hashing and in-toto attestations -- providing a CycloneDX-rooted provenance chain from training through deployment. Sonatype's 2026 report also found that roughly 27% of dependency suggestions from AI coding assistants are invalid or risky hallucinations, adding a new vector for untracked components entering build pipelines. **Regulatory pressure is now concrete**: the [EU Cyber Resilience Act](https://digital-strategy.ec.europa.eu/en/policies/cyber-resilience-act) legally mandates SBOM generation for all products with digital elements sold in the EU, with vulnerability reporting obligations effective September 2026 and full compliance required by December 2027 -- non-compliance carries fines up to EUR 15M or 2.5% of global turnover. The CanisterWorm cascade (March 2026) demonstrated that a single compromised CI/CD tool can spawn self-propagating malware across 47+ npm packages, underscoring why SBOM-driven dependency tracking is essential for detecting unauthorized components entering build pipelines. |
| **4.2.2** | Verify that build artifacts are cryptographically signed with build-origin metadata (source repository, build pipeline, commit hash) that can be independently verified. | 2 | D/V | Artifact tampering between build and deployment. Rogue builds injecting backdoored model-serving code. Lack of provenance making it impossible to trace a production artifact to its source. The **Ultralytics supply chain attack (Dec 2024)** demonstrated this threat: attackers exploited a `pull_request_target` vulnerability in GitHub Actions to inject cryptominer payloads into ultralytics v8.3.41 (60M+ PyPI downloads, 30K+ GitHub stars). More recently, the **Clinejection attack (Feb 2026)** showed how an AI-powered issue triage bot could be prompt-injected to poison GitHub Actions caches ("Cacheract"), ultimately publishing an unauthorized npm package to 5M+ Cline users for ~8 hours. The **hackerbot-claw campaign (Feb-Mar 2026)** demonstrated autonomous, bot-driven exploitation at scale: an automated agent systematically scanned public repositories for exploitable `pull_request_target` workflows, targeting Microsoft, Datadog, Aqua Security, and CNCF projects. It exfiltrated a write-privileged GitHub token from the awesome-go repository (140K+ stars) and stole a PAT from the Trivy project that was subsequently used to privatize the repository and delete release history. | Verify artifacts are signed using Sigstore/cosign, Notary, or GPG. Check that SLSA provenance metadata is attached. Validate signatures independently using public keys. Confirm signing occurs in a hardened build environment (not developer laptops). Verify that GitHub Actions workflows use network egress monitoring (e.g., [StepSecurity Harden-Runner](https://www.stepsecurity.io/)) to detect unauthorized outbound connections during builds, and that workflows are statically analyzed for dangerous `${{ }}` expression injection patterns. For ML model files specifically, verify adoption of the **[OpenSSF Model Signing (OMS) specification](https://openssf.org/blog/2025/06/25/an-introduction-to-the-openssf-model-signing-oms-specification/)**, which uses a detached signature format covering model weights, configuration files, tokenizers, and datasets as a single verifiable unit. OMS is PKI-agnostic, supporting private enterprise systems, self-signed certificates, bare keys, and keyless signing with public or private Sigstore instances. As of March 2026, OMS has production adoption: **NVIDIA signs all NGC catalog models**, and **Google's Kaggle** automatically signs models during upload using OIDC-backed keyless Sigstore certificates, with verification displayed directly in the model hub UI. | SLSA Level 2+ provenance is the target. **SLSA v1.2** (released Nov 2025) introduces a new **Source Track** alongside the existing Build Track, covering threats from source code authoring, review, and management -- Level 2 focuses on history and provenance, Level 3 on continuous enforcement of technical controls. This multi-track approach is directly relevant to AI build pipelines where source integrity (training scripts, configuration, data preprocessing code) is as critical as build integrity. Container images should use cosign with keyless signing via Fulcio/Rekor for transparency log. **OMS solves the large-model signing problem** by employing a detached format that avoids repackaging original content -- the manifest includes file listings with cryptographic hashes and optional annotations for domain-specific fields. The **[model-signing v1.0 CLI](https://blog.sigstore.dev/model-transparency-v1.0/)** (April 2025) makes model signing accessible via `pip install model-signing`, providing sign and verify commands backed by Sigstore with integrations for Kaggle, HuggingFace, TensorFlow, and PyTorch -- supporting keyless OIDC-based signing, PKI, and self-signed certificates. The [Coalition for Secure AI (CoSAI)](https://www.coalitionforsecureai.org/building-trust-in-ai-supply-chains-why-model-signing-is-critical-for-enterprise-security/) now positions model signing as critical enterprise infrastructure. Adapting SLSA to AI supply chains requires addressing the long, resource-intensive nature of model training; SLSA provenance for training data and RAG data remains challenging since dynamic data can render signatures outdated rapidly. Tools like cosign, Syft, Kyverno, and GUAC now make **SLSA Level 2 achievable in an afternoon** for teams with existing CI/CD pipelines, thanks to GitHub's built-in attestation support and maturity of cosign and slsa-github-generator. Future GUAC integration is planned for generating AI-BOMs and inspecting ML supply chains during incident response. |
| **4.2.3** | Verify that build artifact signatures and build-origin metadata are validated at deployment admission, and unverified artifacts are rejected. | 2 | D/V | Deploying tampered or unauthorized artifacts to production. Bypassing the build pipeline by pushing images directly to registries. **Malicious pickle models on Hugging Face (Feb 2025)**: researchers found ML models using 7z compression (instead of standard ZIP) to evade Picklescan detection ("NullifAI" technique), containing reverse-shell payloads targeting Windows/Linux/macOS. Some models went undetected for over 8 months. **JFrog discovered 3 zero-day vulnerabilities in Picklescan** that allowed further evasion of Hugging Face's scanning. **[Model Namespace Reuse](https://unit42.paloaltonetworks.com/model-namespace-reuse/)** (Palo Alto Unit 42, Sept 2025): attackers can re-register deleted Hugging Face account namespaces and upload backdoored models under the original trusted name. Orphaned model references were found in Google Vertex AI Model Garden and Microsoft Azure AI Foundry, achieving remote code execution on cloud inference endpoints. | Configure admission controllers (Kyverno, OPA Gatekeeper, Sigstore Policy Controller) to enforce signature verification. Test by attempting to deploy an unsigned artifact and confirming rejection. Verify admission policies cover all deployment paths (Kubernetes, serverless, edge). On **Kubernetes 1.35+**, leverage the new `imagePullCredentialsVerificationPolicy` (beta) which re-verifies registry credentials even for cached images, preventing unauthorized image reuse. The new PodCertificateRequest API simplifies workload mTLS certificate provisioning for pod-to-pod supply chain verification. For model serving frameworks, verify that **Seldon Core** (with Istio integration for automatic mTLS and JWT-based RequestAuthentication), **BentoML** (structured APIs with Docker containerization), **KServe**, and **NVIDIA Triton Inference Server** enforce artifact validation before model loading. Verify that all model references in code, notebooks, and configuration files are pinned to specific commit hashes (e.g., `revision="commit_hash"`) rather than fetched by name alone -- the Model Namespace Reuse attack demonstrated that name-based trust is exploitable across Google Vertex AI and Azure AI Foundry. For organizations managing model weights at scale, consider using **Harbor** as an AI model registry: models packaged as OCI artifacts with immutable tags, SHA256 digests, Cosign/Notation signature verification enforced before distribution, and RBAC separating push (training) from pull (inference) roles. | Admission control for model weight files is less standardized than container images. Organizations serving models from object storage (S3, GCS) need custom validation hooks. Consider integrating signature checks into model loading code as defense-in-depth. The Hugging Face pickle incidents demonstrate that format-level scanning (Picklescan) is insufficient alone -- deployment admission should combine signature verification, format validation, and sandboxed model loading. The **OpenSSF Secure MLOps whitepaper (Aug 2025)** recommends extending DevOps admission controls to cover datasets and models using SLSA, Sigstore, and OpenSSF Scorecard. The **[NIST IR 8596 Cyber AI Profile](https://nvlpubs.nist.gov/nistpubs/ir/2025/NIST.IR.8596.iprd.pdf)** (preliminary draft, Dec 2025) maps CSF 2.0 to AI systems, recommending that organizations inventory AI models, APIs, keys, agents, and data alongside their permissions, and establish threat detection processes for supplier-provided AI models -- reinforcing the need for admission control at every boundary where AI artifacts enter the deployment pipeline. The **CanisterWorm cascade** (March 2026) demonstrated a new escalation pattern: credential theft from a compromised CI/CD tool (Trivy) spawned a self-propagating worm across 47 npm packages, using ICP blockchain C2 to resist takedown -- admission control that validates package signatures would have blocked the unauthorized publications. |
| **4.2.4** | Verify that builds are reproducible, producing identical output from identical source inputs, enabling independent verification of build integrity. | 3 | D/V | Compromised build infrastructure silently inserting backdoors (Ken Thompson attack). Inability to verify that a production artifact matches its claimed source. The 2025 State of Pipeline Security report documented "Pipeline Parasitism" as a systemic pattern: attackers living off CI/CD using shell injections, over-privileged tokens, invisible Unicode payloads, blockchain C2, and wiper failsafes. The **Trivy/TeamPCP supply chain attack (March 2026)** escalated the threat: attackers force-pushed 75 of 76 version tags in the `aquasecurity/trivy-action` GitHub Action, injecting the "TeamPCP Cloud stealer" that dumped Runner.Worker process memory, harvested SSH/cloud/K8s secrets, and exfiltrated them via AES-256+RSA-4096 encryption -- all within a security scanning tool trusted by thousands of CI/CD pipelines. Root cause: incomplete credential rotation from an earlier incident left a valid token accessible. The **CodeBreach vulnerability (disclosed Jan 2026)** revealed that AWS CodeBuild webhook actor-ID filters for four AWS repositories (including aws-sdk-js-v3) used a regex without start/end anchors, allowing any GitHub user with a superstring user ID to trigger builds and obtain admin-privileged PATs. Wiz estimated a successful exploit could have enabled platform-wide compromise of AWS accounts via the SDK. AWS fixed the issue in Sept 2025 and added a global Pull Request Comment Approval build gate. | Rebuild from the same source commit and compare output hashes. Verify build environment is hermetic (pinned toolchains, no network access during build). Check that non-deterministic elements (timestamps, random seeds) are controlled. For ML serving infrastructure, verify that container images and serving code produce reproducible builds even if model weights themselves are non-deterministic. Validate that all CI/CD workflows use pinned action versions (not mutable tags) and restrict `pull_request_target` trigger usage. As of August 2025, GitHub Actions organization policies can enforce SHA pinning at the platform level -- administrators should enable this so that any workflow referencing an action by mutable tag is automatically rejected. After any credential compromise, perform **atomic credential rotation** -- revoke all tokens simultaneously before issuing replacements to prevent the incomplete-rotation attack pattern seen in both the Clinejection and Trivy incidents. | Full reproducibility for ML artifacts is extremely challenging. Model training is inherently non-deterministic on GPUs (floating-point non-associativity, cuDNN algorithm selection). This requirement is more practical for serving infrastructure builds than for model weights. PyTorch `torch.use_deterministic_algorithms(True)` helps but has significant performance penalties and doesn't cover all operations. The Trivy/TeamPCP and Clinejection attacks (both early 2026) reinforce that even "trusted" CI/CD pipelines and security tools can be weaponized -- hermetic builds with no network access and pinned dependencies are necessary but not sufficient without hardened workflow trigger configurations, atomic credential rotation, and GitHub Actions tag immutability enforcement via [StepSecurity Harden-Runner](https://www.stepsecurity.io/). The CodeBreach incident (Jan 2026) adds another layer: even managed CI/CD services (AWS CodeBuild) can have webhook filter flaws that bypass build authorization entirely. Recent reproducibility research found that only 2.7% of Dockerfiles produce bitwise-reproducible images without infrastructure modifications, rising to 21.3% with configuration changes -- underscoring the practical difficulty of this requirement. |

## Implementation Maturity

| Requirement | Maturity | Notes |
|-------------|:---:|-------|
| 4.2.1 Automated builds with SBOM | Mature | CI/CD automation is standard. CycloneDX 1.7 ML-BOM and SPDX 3.0.1 AI/Dataset profiles provide AI-specific SBOM formats. OWASP AIBOM Generator produces CycloneDX output from HF model references. The AIBoMGen platform (Jan 2026) demonstrates automated AIBOM generation during training with in-toto attestations. The EU Cyber Resilience Act now legally mandates SBOM generation (vulnerability reporting from Sept 2026, full compliance by Dec 2027). Gap: only 21% of orgs have complete dependency visibility (Datadog DevSecOps 2026), and the CanisterWorm cascade (March 2026) showed that a single compromised CI/CD tool can spawn self-propagating malware across 47+ npm packages -- SBOM tracking is essential but insufficient without signature verification. |
| 4.2.2 Cryptographic signing with provenance | Mature | Sigstore/cosign for containers is production-ready. OpenSSF Model Signing (OMS) now has real-world adoption: NVIDIA signs all NGC catalog models and Google's Kaggle signs models automatically during upload using OIDC-backed keyless Sigstore. The **model-signing v1.0 CLI** (April 2025) makes this accessible via `pip install model-signing` with integrations for Kaggle, HuggingFace, TensorFlow, and PyTorch. The Coalition for Secure AI (CoSAI) now positions model signing as critical enterprise infrastructure. **SLSA v1.2** (Nov 2025) adds a Source Track alongside the Build Track, extending provenance to source code authoring and review. SLSA Level 2 is achievable in an afternoon with GitHub's built-in attestation support and cosign + slsa-github-generator. Gap: SLSA provenance for training data and long-running training jobs remains challenging. The CanisterWorm cascade (March 2026) -- where credential theft from Trivy spawned a self-propagating worm across 47 npm packages -- demonstrates that without signed provenance, a single stolen token enables exponential unauthorized publication. |
| 4.2.3 Admission control rejecting unsigned artifacts | Maturing | **Kyverno reached CNCF Graduated status (March 16, 2026)**, a significant maturity milestone for policy-as-code admission control; Kyverno 1.16 introduces CEL-based policies alongside existing YAML-native definitions. OPA Gatekeeper and Sigstore Policy Controller also enforce container image signature verification. Kubernetes 1.35 adds `imagePullCredentialsVerificationPolicy` (beta) for re-verifying registry credentials on cached images, and PodCertificateRequest API for workload mTLS. Harbor can now serve as an AI model registry with OCI artifact packaging and Cosign/Notation signature enforcement. NIST IR 8596 Cyber AI Profile (draft Dec 2025) formalizes admission control as a CSF 2.0 expectation for AI systems. Gap: admission control for model weight files loaded from object storage (S3, GCS, HF Hub) remains less standardized. The **Model Namespace Reuse** attack (Unit 42, Sept 2025) demonstrated that name-based model trust without commit pinning is exploitable across Google Vertex AI and Azure AI Foundry, adding urgency to admission control that verifies model provenance, not just signatures. |
| 4.2.4 Reproducible builds | Emerging | Practical for serving infrastructure container images, though research shows only 2.7% of Dockerfiles produce bitwise-reproducible output without infrastructure changes (21.3% with configuration). Extremely challenging for GPU-trained model weights due to floating-point non-determinism. `torch.use_deterministic_algorithms(True)` helps but has performance penalties. The CI/CD attack surface continues to expand in both scope and sophistication: the CanisterWorm cascade (March 2026) demonstrated a new escalation pattern where a single compromised CI/CD tool (Trivy) spawned a self-propagating npm worm across 47 packages using ICP blockchain C2 -- the first documented use of decentralized smart contracts for malware command infrastructure, making takedown significantly harder than traditional C2 servers. This incident chained hackerbot-claw's autonomous scanning, TeamPCP's credential theft, tag hijacking, IDE extension compromise, and now ecosystem-wide worm propagation into a single sustained campaign spanning February-March 2026. Hermetic builds, pinned dependencies, tag immutability, network egress monitoring, and atomic credential rotation are all required layers. GitHub Actions now supports organization-level SHA pinning enforcement (Aug 2025), which would have mitigated the initial tag hijacking entry point. |

| **Ultralytics PyPI supply chain attack** | Dec 2024 | Trojanized versions 8.3.41-42 of the ultralytics YOLO library (60M+ downloads) delivered XMRig cryptominer via compromised GitHub Actions workflow. Attackers exploited `pull_request_target` to inject malicious code during automated builds, bypassing code review. | Demonstrates need for 4.2.2 (artifact signing), 4.2.3 (admission control), and 4.2.4 (hermetic builds with hardened CI/CD triggers). |
| **Malicious pickle models on Hugging Face** | Feb 2025 | Models using "NullifAI" evasion (7z compression) bypassed Picklescan to deliver reverse-shell payloads. The package installed OpenClaw (an AI agent with shell access) via postinstall. | Demonstrates that AI-powered CI/CD automation itself is an attack surface. Validates need for 4.2.2 (signing), 4.2.3 (admission control), and 4.2.4 (hermetic builds with isolated publication credentials). Also shows why AI agent tooling in CI/CD should follow least-privilege. |

Compromised tags were live for ~12 hours; a malicious `trivy v0.69.4` binary was live for ~3 hours. Root cause: incomplete credential rotation from a prior compromise. | A security scanning tool weaponized against its own users. Demonstrates need for 4.2.4 (tag immutability, atomic credential rotation) and the importance of verifying the integrity of security tooling itself. |
| **Shai-Hulud 2.0 npm worm** | Nov 2025 | Renewed npm ecosystem compromise targeting CI/CD pipelines via pre-install hooks (instead of post-install), dramatically widening the blast radius across developer machines and automated build systems. | Demonstrates that package manager hooks remain a potent CI/CD injection vector. Pre-install execution means builds are compromised before any scanning can run. |
| **CodeBreach (AWS CodeBuild)** | Disclosed Jan 2026 (fixed Sept 2025) | Regex flaw in AWS CodeBuild webhook actor-ID filters for four AWS repositories (aws-sdk-js-v3, aws-lc, amazon-corretto-crypto-provider, open-data-registry) allowed any GitHub user with a superstring user ID to trigger builds and obtain admin PATs with full repo control. Validates need for 4.2.4 (network egress monitoring, workflow static analysis) and reinforces that `pull_request_target` remains the most exploited GitHub Actions trigger. |
| **Trivy VS Code extension compromise (CVE-2026-28353)** | March 2026 | Following the Trivy GitHub Actions compromise, injected logic in Trivy VS Code extension v1.8.12-1.8.13 executed local coding assistants in highly permissive modes, instructing them to perform extensive system inspection and exfiltrate results. | Demonstrates that supply chain attacks can cascade from CI/CD into developer IDE extensions. Validates need for 4.2.3 (admission control at every deployment surface, not just server-side). |
| **CanisterWorm npm cascade** | March 2026 | Self-propagating worm cascaded from the Trivy/TeamPCP credential theft to compromise 47 npm packages across @EmilGroup (28), @opengov (16), and others. Assessed as AI-generated ("vibe-coded") malware with minimal obfuscation. | Demonstrates that CI/CD supply chain attacks can cascade from a single compromised tool (Trivy) into self-propagating ecosystem-wide worms. ## Open Research Questions

- [ ] How should SBOMs represent model weights, training data dependencies, and fine-tuning lineage? CycloneDX 1.7 ML-BOM and OWASP AIBOM are converging but not yet comprehensive -- gaps remain for RAG data sources, adapter/LoRA lineage, and quantization provenance.
- [ ] What is the practical path to reproducible builds for GPU-trained models given floating-point non-determinism? Current approaches (deterministic algorithms, fixed seeds) incur significant performance penalties and do not cover all operations.
- [ ] How can SLSA provenance be extended to cover the full ML pipeline (data ingestion, preprocessing, training, evaluation, packaging)? SLSA v1.1 does not yet address dynamic data or long-running training jobs natively.
- [ ] What signing infrastructure works well for multi-gigabyte model weight files stored in object storage? The OpenSSF OMS specification (2025) addresses this with detached signatures, and as of early 2026 NVIDIA NGC and Google Kaggle have production deployments -- but broader ecosystem adoption beyond these early movers needs tracking.
- [ ] How can admission control be extended beyond container images to validate model files loaded from object storage (S3, GCS, Hugging Face Hub) at serving time? Current Kubernetes admission controllers only cover container image deployment.
- [ ] Given the demonstrated insufficiency of format-based scanning (Picklescan evasions), what layered defenses should be standard for model artifact validation before deployment?
- [ ] How should organizations secure AI-powered CI/CD automation (issue triage bots, code review agents) against prompt injection attacks that can cascade into supply chain compromises? The Clinejection attack (Feb 2026) demonstrated this risk concretely, but defensive patterns are still ad hoc.
- [ ] What is the right approach to credential rotation in CI/CD pipelines to prevent the incomplete-rotation attack pattern seen in the Clinejection and Trivy/TeamPCP incidents? Atomic rotation, short-lived OIDC tokens, and separate publication scopes are emerging best practices but lack standardized implementation guidance.
- [ ] How should organizations defend against autonomous bot-driven CI/CD exploitation (e.g., hackerbot-claw)? Network egress monitoring and workflow static analysis help, but the scale and speed of automated scanning suggests that reactive defenses are insufficient -- what proactive architectural patterns (isolated build identities, ephemeral runners, mandatory PR approval gates) should be baseline?
- [ ] As managed CI/CD services (AWS CodeBuild, Azure DevOps, GCP Cloud Build) are shown to have their own authorization vulnerabilities (CodeBreach), how should defense-in-depth strategies account for trusted-platform compromise? Multi-vendor signing and independent verification may be needed even within a single cloud provider's ecosystem.
- [ ] With OMB M-26-05 (Jan 2026) rescinding the standardized federal software attestation Common Form in favor of agency-specific risk-based approaches, how will this fragmentation affect AI/ML supply chain security requirements for government contractors? Organizations may need to track divergent requirements across agencies rather than a single SBOM/attestation standard.
- [ ] How should AIBOMs capture the provenance of AI-generated code and AI-suggested dependencies, given that Sonatype's 2026 data shows ~27% of AI dependency suggestions are hallucinations? Standard SBOM tools cannot currently distinguish human-selected from AI-suggested components.
- [ ] How should supply chain defenses adapt to decentralized C2 infrastructure (ICP blockchain canisters, IPFS, etc.) that resists traditional takedown? The CanisterWorm (March 2026) demonstrated that self-propagating malware using blockchain dead-drop resolvers cannot be disrupted by domain seizure or server takedowns, requiring a shift from reactive blocklisting to proactive signature verification and anomaly detection at build-time.
- [ ] As supply chain attacks demonstrate sustained multi-phase campaigns (hackerbot-claw scanning -> TeamPCP credential theft -> tag hijacking -> IDE compromise -> CanisterWorm propagation over February-March 2026), what organizational structures and incident response playbooks are needed to track and respond to attack chains that span CI/CD, package registries, and developer environments simultaneously?
- [ ] How should model hubs (Hugging Face, Kaggle, NGC) handle namespace lifecycle to prevent Model Namespace Reuse attacks? Namespace reservation, tombstoning deleted accounts, and mandatory commit-hash pinning in downstream integrations are all proposed mitigations, but no hub has implemented comprehensive protections yet. Cloud platforms that auto-fetch models by name (Google Vertex AI Model Garden, Azure AI Foundry) remain particularly exposed.
