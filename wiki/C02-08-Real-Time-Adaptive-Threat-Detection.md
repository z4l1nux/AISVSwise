# C2.8 Real-Time Adaptive Threat Detection

> **Parent:** [C02: User Input Validation](C02-User-Input-Validation)

## Purpose

Static detection rules and fixed classifiers will always lag behind evolving attack techniques. This section requires threat detection systems that adapt to new patterns in real time, adjusting sensitivity based on observed attack activity and contextual signals. This ranges from basic pattern matching (Level 1) to fully adaptive detection with continuous performance monitoring (Level 3). The goal is a detection layer that improves over time rather than degrading as attackers learn to bypass fixed rules.

## Requirements

| # | Requirement | Level | Role | Threat Mitigated | Verification Approach | Gaps / Notes |

| **2.8.1** | **Verify that** pattern matching (e.g., compiled regular expressions) runs on all inputs and outputs (including tool/MCP surfaces). | 1 | D/V | Known prompt injection patterns; known jailbreak phrases; SQL/code injection via tool arguments; known adversarial prefixes/suffixes; policy-violating keyword patterns. | 1. Confirm a pattern matching engine is integrated into the input/output pipeline. 2. Review the ruleset: does it cover known prompt injection patterns (e.g., "ignore previous instructions", "you are now", "system prompt:"), common jailbreak techniques, and injection patterns for tool arguments? 3. Verify patterns run on both inputs and outputs. 4. Verify patterns cover tool/MCP argument surfaces (not just the main prompt endpoint). 5. Test with known patterns and confirm detection. 6. Confirm regex patterns are compiled (not interpreted per-request) for performance. 7. Verify detection extends to encoded payloads — Base64-encoded instructions, Unicode homoglyphs, and mixed-language prompts are common evasion vectors (as of March 2026). | Pattern matching is the simplest detection layer and should be the baseline. Regex-based detection is fast but brittle -- attackers can easily rephrase to avoid exact matches. Palo Alto Unit 42 (2025) observed that 37.8% of real-world indirect prompt injection payloads use visible plaintext delivery, which pattern matching can catch, but 85.2% also employ social engineering jailbreak tactics that evade fixed rules. Rahman & Alouani (2025) demonstrated that GCG-optimized adversarial suffixes achieve 93–99% bypass rates against activation-based probe detectors, and separate research showed 100% evasion success against Azure Prompt Shield and Meta Prompt Guard — underlining that pattern matching alone is necessary but never sufficient. Common evasion techniques observed in production as of early 2026 include Base64-encoded instructions decoded at runtime, invisible DOM element injection targeting web-browsing agents, language-switching/mixed-language prompts, and semantically equivalent rephrasing that preserves malicious intent while appearing benign. The ruleset must be maintained and updated as new attack patterns emerge. Open-source rulesets: LLM Guard rules, OWASP prompt injection patterns, CrowdStrike Falcon AIDR's 75+ built-in classification rules, custom rules based on red-team findings. Compiled regex (Python `re.compile()`, Rust `regex` crate) is essential for performance at scale. Output-side pattern matching catches cases where the model produces content that should have been blocked (e.g., leaked system prompts). The CrowdStrike 2026 Global Threat Report documented adversaries injecting malicious prompts into GenAI tools at more than 90 organizations to generate commands for credential theft and cryptocurrency exfiltration — confirming that pattern matching on tool surfaces is not optional. Arcjet (March 2026) offers inline prompt inspection at the application boundary before requests reach the model, integrating with Vercel AI SDK and LangChain to give developers a policy decision point with full request context (identity, session state, business logic). Multimodal prompt injection is an increasingly practical threat vector: CSA research (March 2026) documented four image-based injection techniques — typographic injection (up to 64% attack success against GPT-4V, Claude 3, and Gemini under stealth constraints), steganographic encoding (24.3% success overall, 31.8% with neural methods), adversarial perturbations (CrossInject framework achieving 30%+ improvement over prior methods), and physical-world signage targeting camera-equipped agents. ArtPrompt research demonstrated that ASCII art and obfuscation encoding achieves up to 76.2% bypass rates across GPT-4, Gemini, Claude, and Llama2. Pattern matching must therefore extend to OCR-based scanning of image inputs and decoded representations of multimodal content where applicable. Note: Rebuff (Protect AI) was archived in May 2025; its pattern library is no longer maintained. |
| **2.8.2** | **Verify that** adaptive detection models adjust sensitivity based on recent attack activity and are updated with new patterns in real time, and trigger risk-adaptive responses (for example disable tools, shrink context, or require HITL approval). | 2 | D/V | Novel attack patterns not covered by static rules; evolving jailbreak techniques; zero-day prompt injection methods; attack campaigns that adapt to bypass fixed detections; indirect prompt injection via poisoned documents/web content; agent goal hijacking (OWASP Agentic Top 10 ASI01). | 1. Review the detection system's architecture for adaptive capabilities: dynamic rule updates, ML model retraining pipeline, or online learning. 2. Verify sensitivity adjustment mechanisms: can detection thresholds be tightened in response to increased attack activity? 3. Test risk-adaptive responses: submit inputs that trigger escalating risk levels and verify graduated responses (e.g., low risk = normal processing, medium risk = disable tools, high risk = HITL required). 4. Confirm new detection patterns can be deployed without full system restart (hot-reload). 5. Verify that the detection layer covers agentic surfaces: inter-agent messages, tool schemas, and RAG retrieval content, not just end-user prompts. | Adaptive detection is a significant step up from static rules. Nasr et al. ("The Attacker Moves Second," October 2025) tested 12 published defenses using adaptive attacks (gradient descent, reinforcement learning, random search, human red teaming) and bypassed every defense with >90% success rates under adaptive conditions -- underscoring that static defenses alone are insufficient. Practical implementations: a rule engine with hot-reloadable rulesets (updated by security team based on observed attacks), ML classifiers retrained on recent data, or online learning systems. Risk-adaptive responses are key -- rather than binary accept/reject, the system degrades capabilities proportionally to risk. "Shrink context" means reducing the context window to give more weight to system instructions. "Disable tools" removes the most dangerous capability (code execution, data access) while keeping basic chat. The OWASP Top 10 for Agentic Applications (2026) emphasizes runtime governance where permissions and tool access are validated continuously, with kill switches and automatic containment for anomalous agent behavior. Commercial services: Lakera Guard (acquired by Check Point, September 2025) claims 98%+ detection rates with sub-50ms latency, collecting 100K+ new adversarial samples daily for continuous model updates; Palo Alto Prisma AIRS provides runtime AI security including RAG context inspection; NVIDIA NeMo Guardrails offers programmable open-source guardrails. As of March 2026, CrowdStrike Falcon AIDR (v0.20.0) integrates with NeMo Guardrails to provide sub-100ms runtime detection with graduated enforcement modes (monitor, redact, encrypt, block) — a practical example of risk-adaptive response. F5 AI Guardrails (January 2026) establishes a continuous feedback loop where its AI Red Team component identifies bypass vectors and automatically updates runtime guardrail policies. Akamai Firewall for AI provides edge-based detection with adaptive security rules, reporting that during technical preview a digital mortgage provider found 5–6% of interactions involved prompt injection or data leakage attempts. Cisco AI Defense (expanded February 2026) now inspects MCP traffic in real time with purpose-built agentic guardrails — it can detect malicious signatures embedded in MCP payloads such as prompt injections designed to exfiltrate device configuration data, and includes a Tool Exploitation guardrail preventing adversaries from hijacking connected tools. The "lethal trifecta" framework (Airia, 2026) identifies systems as critically vulnerable when they combine private data access, untrusted content exposure, and exfiltration capability — all three conditions demand adaptive detection rather than static rules. Microsoft Copilot Studio (February 2026) introduced real-time runtime protection in public preview, enabling organizations to connect external monitoring systems (Microsoft Defender or third-party platforms) for continuous evaluation and control of agent behavior during execution; their "Copilot Studio Agent Security Top 10" identified common misconfigurations — broad sharing, unauthenticated access, unmanaged MCP tools, dormant agents, and hard-coded credentials — that create identity and data-access paths traditional controls miss. Microsoft Agent 365 (GA May 2026) provides an enterprise agent control plane with Defender, Entra, and Purview integrations to observe, secure, and govern agents at scale. For multimodal inputs, the Cross-Agent Multimodal Provenance-Aware Defense Framework (Syed et al., ICCA 2025) combines input sanitization with output validation across agentic pipelines, achieving 94% detection accuracy for prompt injection and a 70% reduction in cross-agent contamination; specific defenses include VLMGuard for unlabeled image injection detection and SmoothVLM which reduces attack success to 0–5% for patched injection variants. |
| **2.8.3** | **Verify that** detection accuracy is improved via contextual analysis of user history, source, and session behavior, including trace metadata (source, tool or MCP server, agent ID, session). | 3 | D/V | Evasion of per-request detection; slow-roll attacks (gradually escalating across sessions); repeat offender attacks from known-bad sources; context-dependent attacks that appear benign in isolation but are malicious in context; crescendo attacks; memory poisoning attacks where adversaries implant malicious instructions in agent long-term storage; second-order prompt injection across agent privilege boundaries. | 1. Verify the detection system has access to user history (previous requests, previous detection flags, risk score trends). 2. Confirm source reputation is factored into detection (e.g., new accounts vs. established users, IP reputation, API key risk level). 3. Test contextual detection: submit a sequence of individually benign requests that form a crescendo attack and verify detection. 4. Verify session behavior analysis: detecting anomalous patterns within a session (e.g., rapid topic switching, progressive probing). 5. Verify cross-agent context: detection can correlate activity across multiple agents in a workflow to catch second-order injection (a low-privilege agent tricking a high-privilege agent). | Level 3 because contextual detection requires significant data infrastructure (user history storage, session tracking, behavioral analytics). Privacy implications: storing user interaction history for security detection must comply with data protection regulations and retention policies. Crescendo attack detection (Russinovich et al., 2024) requires analyzing conversation trajectories, which is architecturally different from per-request classification. ServiceNow disclosed a "second-order" prompt injection incident (late 2025) where a low-privilege agent was fed a malformed request that tricked a higher-privilege agent into exporting an entire case file to an external URL -- bypassing normal access controls. This class of attack requires cross-agent behavioral correlation to detect. Memory poisoning (identified as ASI06 in OWASP Agentic Top 10) means an agent may recall and act on adversary-planted instructions days or weeks later, requiring historical context analysis. The OWASP Top 10 for Agentic Applications (2026) recommends behavioral analytics that understand agent action patterns and flag deviations in real time, noting that traditional SIEM will not catch agentic threats. MITRE ATLAS added 14 agentic AI attack techniques in October 2025 (in collaboration with Zenity Labs), including AI Agent Context Poisoning — manipulating the context used by an agent's LLM to persistently influence its responses. The 2026 ATLAS update shifts focus from model-centric attacks to execution-layer exposure, requiring threat modeling that accounts for autonomous workflow chaining, delegated authority persistence, and API-level orchestration risk. Microsoft elevated AI observability to a mandatory security requirement for enterprise agentic systems (March 2026), recommending behavioral baselines of agent activity — tool call frequencies, retrieval volumes, token consumption, evaluation score distributions — with alerting on meaningful departures rather than static error thresholds. The EchoLeak attack against Microsoft 365 Copilot demonstrated zero-click exploitation: attackers embedded hidden instructions in emails that Copilot retrieved when users asked unrelated questions, encoding and exfiltrating sensitive data via image URL requests. A similar GeminiJack attack targeted Google Gemini Enterprise via RAG-indexed poisoned content. Both illustrate that contextual detection must analyze retrieval provenance and output channels, not just the user prompt. An important practical consideration: agents generate 10–20x more activity than human employees in equivalent timeframes (CSO Online, March 2026), and not all agent platforms produce audit trails by default — some coding agents can even overwrite session logs during replay, destroying incident evidence. This means contextual detection infrastructure must handle dramatically higher event volumes while ensuring log integrity. Practical starting points: maintain a per-user risk score that increases with detected anomalies and decays over time; use IP/API key reputation services; deploy agent-aware behavioral analytics that track action sequences rather than individual requests; capture baseline agent behavior metrics via Azure Monitor, Langfuse, or equivalent platforms; and implement append-only logging with tamper-evident integrity protections for agent activity trails. |
| **2.8.4** | **Verify that** detection performance metrics (detection rate, false positive rate, processing latency) are continuously monitored and optimized, including time-to-block and stage (pre-prompt/post-response). | 3 | D/V | Detection drift (classifier performance degrading over time); unacceptable false positive rates blocking legitimate users; excessive detection latency impacting user experience; inability to quantify security posture. | 1. Confirm detection metrics are collected: true positive rate, false positive rate, false negative rate (from manual review), processing latency (p50, p95, p99), time-to-block. 2. Verify metrics are broken down by detection stage (pre-prompt vs. post-response). 3. Confirm a dashboard or reporting mechanism exists for these metrics. 4. Review optimization cadence: how often are thresholds adjusted, rules updated, or models retrained based on metrics? 5. Verify SLA targets exist for detection latency (e.g., <100ms for pattern matching, <500ms for ML-based detection). 6. Verify feedback loops: runtime detection insights are fed back into build-time controls to refine agent configurations and permissions. | Level 3 because continuous metric monitoring and optimization requires dedicated security engineering resources and infrastructure. Key metrics: detection rate (true positives / total attacks), false positive rate (false alarms / total legitimate requests), latency overhead (added processing time per request), time-to-block (from attack detection to block enforcement). False negative measurement is inherently difficult -- it requires ground truth from manual review or red-team exercises. Lakera Guard reports sub-50ms latency and <0.5% false positive rates as production benchmarks; CrowdStrike Falcon AIDR targets sub-100ms response times as its agentic workflow SLA (March 2026) -- these can serve as reference targets but should be validated independently. Benchmarking against adversarial test suites (e.g., HarmBench, JailbreakBench, AdvBench) can provide periodic ground truth, but Nasr et al. (2025) showed that benchmark performance may not reflect real-world adaptive attacker success rates. The OWASP AI Testing Guide v1 (November 2025) and NIST AI Cybersecurity Profile (NISTIR 8596, December 2025) both recommend continuous security monitoring combined with AI-based log analysis. NIST's COSAiS project is developing SP 800-53 control overlays for five AI use cases including single-agent and multi-agent systems; the predictive AI annotated outline was published for comment in January 2026 with feedback due February 2026, and agentic AI overlays are expected mid-to-late 2026. These will likely formalize continuous monitoring requirements for AI runtime security. The OWASP Top 10 for Agentic Applications (2026) emphasizes a feedback loop where runtime insights continuously inform build-time controls, creating progressively more resilient agents. F5's AI Guardrails + AI Red Team pairing (January 2026) demonstrates this feedback loop in practice: the red team component continuously discovers bypass vectors from an industry vulnerability database of 10,000+ new attack techniques per month, which automatically feed into updated guardrail policies. Regulatory pressure is formalizing these requirements: the EU AI Act's August 2, 2026 deadline for high-risk AI systems mandates continuous monitoring programs, real-world performance tracking, and serious incident reporting to authorities within strict timeframes — with penalties up to EUR 35 million or 7% of global annual turnover for noncompliance (though the European Commission's "Digital Omnibus" proposal may extend some Annex III obligations to December 2027, organizations should plan for the August 2026 date). Build-time vulnerability fixes cost roughly $40 versus $4,000 at runtime (a 100x multiplier), reinforcing the economic case for the feedback loop where runtime detection insights continuously refine build-time controls. |

## Open Research Questions

- What is the optimal architecture for adaptive AI threat detection -- should it be a separate ML model, a rule engine, or a hybrid? Nasr et al. (2025) suggest no single-layer defense suffices, pointing toward defense-in-depth with diverse detection mechanisms.
- How can detection systems be updated in real time without introducing instability or being manipulated by adversarial feedback? Lakera's approach (100K+ samples/day ingestion) demonstrates scale but raises questions about adversarial poisoning of the training pipeline itself.
- What detection latency is acceptable for different AI application types (real-time chat vs. batch processing vs. agentic workflows)? Current benchmarks: sub-50ms for pattern matching, sub-500ms for ML-based detection -- but agentic systems with multi-step tool chains may tolerate higher per-step latency.
- How should detection metrics be benchmarked -- are existing adversarial AI benchmarks (HarmBench, JailbreakBench) sufficient for production evaluation? Evidence suggests benchmark performance diverges from real-world adaptive attacker success rates.
- Can federated threat intelligence (sharing attack patterns across organizations) improve detection without exposing proprietary data? The Check Point/Lakera acquisition model centralizes threat intelligence -- is this more effective or more risky than federated approaches?
- How should runtime detection adapt for agentic architectures where attacks span multiple agents, tools, and sessions (second-order injection, memory poisoning)?
- As the AI security tooling market consolidates (three major acquisitions in 2025) while new entrants emerge (CrowdStrike, F5, Akamai in Q1 2026), how should organizations evaluate vendor lock-in risk for runtime detection infrastructure? The CrowdStrike/NVIDIA blueprint approach of embedding security into open-source agent runtimes may offer more portability than proprietary API-only solutions.
- How should detection systems handle multimodal prompt injection — malicious instructions hidden in images, audio, or video that accompany benign text? As of March 2026, research demonstrates concrete attack viability: typographic image injection reaches 64% ASR, steganographic encoding 24–32%, and ArtPrompt obfuscation 76.2% across major models. Early defenses (VLMGuard, SmoothVLM) show promise but remain research-stage, and no commercial runtime detection platform yet offers production-grade multimodal injection scanning.
- Can adversarial training of detection classifiers keep pace with gradient-based evasion techniques? Rahman & Alouani (2025) showed suffix-based adversarial training achieves 80–100% accuracy, but this is an ongoing arms race where defenders must continuously retrain against new optimization strategies.
- Is boundary-first detection (inspecting at the application/API boundary before the model sees input) fundamentally more effective than model-side guardrails? MIT Technology Review (January 2026) argues "rules fail at the prompt, succeed at the boundary" — Arcjet and Cisco AI Defense both take this approach, but it remains unclear whether boundary inspection can catch semantically subtle attacks that require model-level understanding.
- With the CrowdStrike 2026 report showing 82% of AI-targeted detections are malware-free and identity-based, how should detection metrics (2.8.4) be adapted for attacks that use legitimate credentials and trusted identity flows rather than recognizable exploit patterns?
- How should detection infrastructure scale for agent activity volumes? Agents generate 10–20x more events than human users, and some agent runtimes lack audit trail generation or allow log self-modification. What architectural patterns (append-only logs, external log aggregation, cryptographic log chaining) are necessary to ensure detection systems maintain forensic integrity at agent-scale throughput?
- As the EU AI Act's August 2026 high-risk compliance deadline approaches, how should organizations reconcile regulatory requirements for continuous monitoring and incident reporting with the practical reality that most AI runtime security tools are less than 18 months old? The potential December 2027 extension (Digital Omnibus proposal) adds uncertainty to compliance timelines.
