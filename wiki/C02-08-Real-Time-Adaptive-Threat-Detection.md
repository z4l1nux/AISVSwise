# C2.8 Real-Time Adaptive Threat Detection

> **Parent:** [C02: User Input Validation](C02-User-Input-Validation)

## Purpose

Static detection rules and fixed classifiers will always lag behind evolving attack techniques. This section requires threat detection systems that adapt to new patterns in real time, adjusting sensitivity based on observed attack activity and contextual signals. This ranges from basic pattern matching (Level 1) to fully adaptive detection with continuous performance monitoring (Level 3). The goal is a detection layer that improves over time rather than degrading as attackers learn to bypass fixed rules.

## Requirements

| # | Requirement | Level | Role | Threat Mitigated | Verification Approach | Gaps / Notes |
|---|-------------|:-----:|:----:|-----------------|----------------------|--------------|
| **2.8.1** | **Verify that** pattern matching (e.g., compiled regular expressions) runs on all inputs and outputs (including tool/MCP surfaces). | 1 | D/V | Known prompt injection patterns; known jailbreak phrases; SQL/code injection via tool arguments; known adversarial prefixes/suffixes; policy-violating keyword patterns. | 1. Confirm a pattern matching engine is integrated into the input/output pipeline. 2. Review the ruleset: does it cover known prompt injection patterns (e.g., "ignore previous instructions", "you are now", "system prompt:"), common jailbreak techniques, and injection patterns for tool arguments? 3. Verify patterns run on both inputs and outputs. 4. Verify patterns cover tool/MCP argument surfaces (not just the main prompt endpoint). 5. Test with known patterns and confirm detection. 6. Confirm regex patterns are compiled (not interpreted per-request) for performance. | Pattern matching is the simplest detection layer and should be the baseline. Regex-based detection is fast but brittle -- attackers can easily rephrase to avoid exact matches. Palo Alto Unit 42 (2025) observed that 37.8% of real-world indirect prompt injection payloads use visible plaintext delivery, which pattern matching can catch, but 85.2% also employ social engineering jailbreak tactics that evade fixed rules. The ruleset must be maintained and updated as new attack patterns emerge. Open-source rulesets: LLM Guard rules, OWASP prompt injection patterns, custom rules based on red-team findings. Compiled regex (Python `re.compile()`, Rust `regex` crate) is essential for performance at scale. Output-side pattern matching catches cases where the model produces content that should have been blocked (e.g., leaked system prompts). Note: Rebuff (Protect AI) was archived in May 2025; its pattern library is no longer maintained. |
| **2.8.2** | **Verify that** adaptive detection models adjust sensitivity based on recent attack activity and are updated with new patterns in real time, and trigger risk-adaptive responses (for example disable tools, shrink context, or require HITL approval). | 2 | D/V | Novel attack patterns not covered by static rules; evolving jailbreak techniques; zero-day prompt injection methods; attack campaigns that adapt to bypass fixed detections; indirect prompt injection via poisoned documents/web content; agent goal hijacking (OWASP Agentic Top 10 ASI01). | 1. Review the detection system's architecture for adaptive capabilities: dynamic rule updates, ML model retraining pipeline, or online learning. 2. Verify sensitivity adjustment mechanisms: can detection thresholds be tightened in response to increased attack activity? 3. Test risk-adaptive responses: submit inputs that trigger escalating risk levels and verify graduated responses (e.g., low risk = normal processing, medium risk = disable tools, high risk = HITL required). 4. Confirm new detection patterns can be deployed without full system restart (hot-reload). 5. Verify that the detection layer covers agentic surfaces: inter-agent messages, tool schemas, and RAG retrieval content, not just end-user prompts. | Adaptive detection is a significant step up from static rules. Nasr et al. ("The Attacker Moves Second," October 2025) tested 12 published defenses using adaptive attacks (gradient descent, reinforcement learning, random search, human red teaming) and bypassed every defense with >90% success rates under adaptive conditions -- underscoring that static defenses alone are insufficient. Practical implementations: a rule engine with hot-reloadable rulesets (updated by security team based on observed attacks), ML classifiers retrained on recent data, or online learning systems. Risk-adaptive responses are key -- rather than binary accept/reject, the system degrades capabilities proportionally to risk. "Shrink context" means reducing the context window to give more weight to system instructions. "Disable tools" removes the most dangerous capability (code execution, data access) while keeping basic chat. The OWASP Top 10 for Agentic Applications (2026) emphasizes runtime governance where permissions and tool access are validated continuously, with kill switches and automatic containment for anomalous agent behavior. Commercial services: Lakera Guard (acquired by Check Point, September 2025) claims 98%+ detection rates with sub-50ms latency, collecting 100K+ new adversarial samples daily for continuous model updates; Palo Alto Prisma AIRS provides runtime AI security including RAG context inspection; NVIDIA NeMo Guardrails offers programmable open-source guardrails. |
| **2.8.3** | **Verify that** detection accuracy is improved via contextual analysis of user history, source, and session behavior, including trace metadata (source, tool or MCP server, agent ID, session). | 3 | D/V | Evasion of per-request detection; slow-roll attacks (gradually escalating across sessions); repeat offender attacks from known-bad sources; context-dependent attacks that appear benign in isolation but are malicious in context; crescendo attacks; memory poisoning attacks where adversaries implant malicious instructions in agent long-term storage; second-order prompt injection across agent privilege boundaries. | 1. Verify the detection system has access to user history (previous requests, previous detection flags, risk score trends). 2. Confirm source reputation is factored into detection (e.g., new accounts vs. established users, IP reputation, API key risk level). 3. Test contextual detection: submit a sequence of individually benign requests that form a crescendo attack and verify detection. 4. Verify session behavior analysis: detecting anomalous patterns within a session (e.g., rapid topic switching, progressive probing). 5. Verify cross-agent context: detection can correlate activity across multiple agents in a workflow to catch second-order injection (a low-privilege agent tricking a high-privilege agent). | Level 3 because contextual detection requires significant data infrastructure (user history storage, session tracking, behavioral analytics). Privacy implications: storing user interaction history for security detection must comply with data protection regulations and retention policies. Crescendo attack detection (Russinovich et al., 2024) requires analyzing conversation trajectories, which is architecturally different from per-request classification. ServiceNow disclosed a "second-order" prompt injection incident (late 2025) where a low-privilege agent was fed a malformed request that tricked a higher-privilege agent into exporting an entire case file to an external URL -- bypassing normal access controls. This class of attack requires cross-agent behavioral correlation to detect. Memory poisoning (identified as ASI06 in OWASP Agentic Top 10) means an agent may recall and act on adversary-planted instructions days or weeks later, requiring historical context analysis. The OWASP Top 10 for Agentic Applications (2026) recommends behavioral analytics that understand agent action patterns and flag deviations in real time, noting that traditional SIEM will not catch agentic threats. Practical starting points: maintain a per-user risk score that increases with detected anomalies and decays over time; use IP/API key reputation services; deploy agent-aware behavioral analytics that track action sequences rather than individual requests. |
| **2.8.4** | **Verify that** detection performance metrics (detection rate, false positive rate, processing latency) are continuously monitored and optimized, including time-to-block and stage (pre-prompt/post-response). | 3 | D/V | Detection drift (classifier performance degrading over time); unacceptable false positive rates blocking legitimate users; excessive detection latency impacting user experience; inability to quantify security posture. | 1. Confirm detection metrics are collected: true positive rate, false positive rate, false negative rate (from manual review), processing latency (p50, p95, p99), time-to-block. 2. Verify metrics are broken down by detection stage (pre-prompt vs. post-response). 3. Confirm a dashboard or reporting mechanism exists for these metrics. 4. Review optimization cadence: how often are thresholds adjusted, rules updated, or models retrained based on metrics? 5. Verify SLA targets exist for detection latency (e.g., <100ms for pattern matching, <500ms for ML-based detection). 6. Verify feedback loops: runtime detection insights are fed back into build-time controls to refine agent configurations and permissions. | Level 3 because continuous metric monitoring and optimization requires dedicated security engineering resources and infrastructure. Key metrics: detection rate (true positives / total attacks), false positive rate (false alarms / total legitimate requests), latency overhead (added processing time per request), time-to-block (from attack detection to block enforcement). False negative measurement is inherently difficult -- it requires ground truth from manual review or red-team exercises. Lakera Guard reports sub-50ms latency and <0.5% false positive rates as production benchmarks -- these can serve as reference targets but should be validated independently. Benchmarking against adversarial test suites (e.g., HarmBench, JailbreakBench, AdvBench) can provide periodic ground truth, but Nasr et al. (2025) showed that benchmark performance may not reflect real-world adaptive attacker success rates. The OWASP AI Testing Guide v1 (November 2025) and NIST AI Cybersecurity Profile (NISTIR 8596, December 2025) both recommend continuous security monitoring combined with AI-based log analysis. The OWASP Top 10 for Agentic Applications (2026) emphasizes a feedback loop where runtime insights continuously inform build-time controls, creating progressively more resilient agents. |

## Real-World Incidents

| Date | Incident | Relevance |
|------|----------|-----------|
| 2024 | **CVE-2024-5184: Email Assistant Prompt Injection** -- attacker injected malicious prompts into an LLM-powered email assistant to access sensitive information and manipulate email content. | Demonstrates need for runtime detection on all input surfaces (2.8.1), not just user-facing prompts. |
| Dec 2025 | **AI Ad Review System Bypass** (Palo Alto Unit 42) -- attackers used hidden prompts on malicious websites (reviewerpress.com) to instruct an AI content moderation agent to approve fraudulent military glasses ads. Multiple indirect prompt injection methods combined in a single attack. | Shows that per-request static detection fails against multi-method indirect injection; adaptive detection (2.8.2) and contextual analysis (2.8.3) needed. |
| Late 2025 | **ServiceNow Second-Order Agent Injection** -- a low-privilege agent received a malformed request and tricked a higher-privilege agent into exporting an entire case file to an external URL, bypassing normal access controls. | Cross-agent privilege escalation requires behavioral analytics and inter-agent context correlation (2.8.3). Traditional per-request detection would miss this. |
| 2025 | **Cursor IDE MCP Exploitation (CVE-2025-54135, CVE-2025-54136)** -- attackers exploited trust flaws in Cursor's MCP implementation to execute arbitrary commands on developer machines without user knowledge. | Runtime monitoring must cover tool/MCP surfaces (2.8.1) and detect anomalous tool invocation patterns (2.8.2). |
| Oct 2025 | **"The Attacker Moves Second" (Nasr et al.)** -- researchers bypassed all 12 tested published defenses with >90% success using adaptive attacks (gradient descent, RL, random search, human red teaming). | Demonstrates that static defenses degrade rapidly; continuous metric monitoring (2.8.4) and adaptive updating (2.8.2) are essential. |

## Open Research Questions

- What is the optimal architecture for adaptive AI threat detection -- should it be a separate ML model, a rule engine, or a hybrid? Nasr et al. (2025) suggest no single-layer defense suffices, pointing toward defense-in-depth with diverse detection mechanisms.
- How can detection systems be updated in real time without introducing instability or being manipulated by adversarial feedback? Lakera's approach (100K+ samples/day ingestion) demonstrates scale but raises questions about adversarial poisoning of the training pipeline itself.
- What detection latency is acceptable for different AI application types (real-time chat vs. batch processing vs. agentic workflows)? Current benchmarks: sub-50ms for pattern matching, sub-500ms for ML-based detection -- but agentic systems with multi-step tool chains may tolerate higher per-step latency.
- How should detection metrics be benchmarked -- are existing adversarial AI benchmarks (HarmBench, JailbreakBench) sufficient for production evaluation? Evidence suggests benchmark performance diverges from real-world adaptive attacker success rates.
- Can federated threat intelligence (sharing attack patterns across organizations) improve detection without exposing proprietary data? The Check Point/Lakera acquisition model centralizes threat intelligence -- is this more effective or more risky than federated approaches?
- How should runtime detection adapt for agentic architectures where attacks span multiple agents, tools, and sessions (second-order injection, memory poisoning)?
- As the AI security tooling market consolidates (three major acquisitions in 2025), how should organizations evaluate vendor lock-in risk for runtime detection infrastructure?
