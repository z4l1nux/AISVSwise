# C2.7 Multi-Modal Input Validation

> **Parent:** [C02: User Input Validation](C02-User-Input-Validation)

## Purpose

Modern AI systems accept inputs beyond text -- images, audio, video, PDFs, spreadsheets, and other file types. Each modality introduces unique attack surfaces: images can contain steganographic payloads or adversarial perturbations invisible to humans; audio can include ultrasonic commands; documents can embed hidden instructions in metadata, comments, or invisible layers. Multi-modal validation must handle format verification, content extraction, malware scanning, and adversarial detection for each modality, while also detecting coordinated cross-modal attacks where benign inputs in individual modalities combine to form an attack.

| **2.7.1** | **Verify that** all non-text inputs (images, audio, files) are validated for type, size, and format before processing, and that any extracted text (image-to-text or speech-to-text) and any hidden or embedded instructions (metadata, layers, alt text, comments) are treated as untrusted per 2.1.1. | 1 | D | Malformed file attacks; file type spoofing; injection via extracted text (OCR output containing prompt injection); hidden instructions in EXIF metadata, PDF comments, DOCX tracked changes, image alt text, audio metadata; resource exhaustion via oversized files. | 1. Test with: spoofed file types (JPEG extension with PNG content), oversized files, corrupted files, polyglot files (valid in multiple formats). 2. Verify MIME type validation uses magic bytes (not just file extension). 3. Confirm extracted text (OCR, speech-to-text) passes through the prompt injection detection pipeline (2.1.1). 4. Test with files containing injection payloads in: EXIF metadata, PDF comments, DOCX XML comments, image alt text fields, audio ID3 tags. 5. Verify file size limits are enforced before processing begins (not after full file is loaded into memory). | This is a critical control because multi-modal injection is a growing attack vector. Research by Bagdasaryan et al. (2023) demonstrated injection attacks via images in GPT-4V. Hidden instructions in metadata are trivial to add and easy to miss in validation. Libraries: Python `Pillow` for image validation, `python-magic` for MIME type detection, `mutagen` for audio metadata, `PyMuPDF` for PDF parsing. The "treated as untrusted per 2.1.1" cross-reference is important -- extracted text must go through the same prompt injection screening as direct user input. |
| **2.7.2** | **Verify that** files are scanned for malware and steganographic payloads before ingestion, and that any active content (like scripts or macros) is removed or the file is quarantined. | 2 | D/V | Malware delivery via file uploads; steganographic data exfiltration channels; macro-based attacks in Office documents; script injection via SVG files; PDF JavaScript execution; executable content in archives. | 1. Confirm antimalware scanning is integrated into the file upload pipeline (e.g., ClamAV, Windows Defender, VirusTotal API, commercial AV). 2. Test with EICAR test file and known malware samples. 3. Verify steganography detection is implemented for image inputs (e.g., StegExpose, zsteg, binwalk). 4. Test with: Office documents containing macros, SVG files with embedded JavaScript, PDFs with JavaScript actions, archives containing executables. 5. Confirm active content removal (e.g., using Apache Tika or document sanitization libraries) or quarantine workflow. | Malware scanning is standard practice, but steganographic payload detection is less mature. Steganography detection tools (StegExpose, zsteg) have high false positive rates and cannot detect all steganographic methods. Active content removal is well-supported for common formats: `python-docx` can strip macros, PDF sanitization can remove JavaScript, SVG sanitization can remove script tags. The quarantine workflow needs operational definition: who reviews quarantined files, and what is the SLA? Level 2 reflects that steganography detection adds significant complexity beyond basic malware scanning. |
| **2.7.3** | **Verify that** image/audio inputs are checked for adversarial perturbations or known attack patterns, and detections trigger gating (block or degrade capabilities) before model use. | 2 | D/V | Adversarial image attacks (e.g., perturbations causing misclassification); typographic attacks on vision models (text overlaid on images to manipulate model behavior); adversarial audio attacks (e.g., hidden commands in audio, ultrasonic signals); visual prompt injection (text instructions rendered in images). | 1. Test with known adversarial image examples (e.g., FGSM, PGD, C&W generated perturbations). 2. Test with typographic attack images (text overlaid on images that instructs the vision model). 3. For audio, test with adversarial audio samples and ultrasonic content. 4. Verify detection triggers appropriate gating: blocking the input or degrading capabilities (e.g., disabling tool use for that request). 5. Review the detection method: statistical analysis (checking for unusual pixel/sample distributions), neural network-based detection, or comparison against known attack signatures. | Adversarial perturbation detection for images is an active research area. Tools: IBM Adversarial Robustness Toolbox (ART) provides both attack generation and detection capabilities. Microsoft Counterfit can generate adversarial examples for testing. For vision models, typographic attacks (Goh et al., 2021) and visual prompt injection (Bagdasaryan et al., 2023) are distinct from traditional adversarial perturbations and may require different detection approaches. Detection of adversarial audio is less mature. "Degrade capabilities" is important -- rather than fully blocking, the system can disable high-risk actions (tool calls, data access) while still providing a text response. |
| **2.7.4** | **Verify that** multi-modal input validation failures trigger detailed logging including all input modalities, validation results, threat scores, and trace metadata (source, tool or MCP server, agent ID, session as applicable), and generate alerts for investigation. | 3 | D/V | Undetected multi-modal attacks; inability to correlate attacks across modalities; missing forensic evidence for multi-modal incident response; failure to improve detection based on real-world attack data. | 1. Trigger validation failures across different modalities and verify logging. 2. Confirm log entries include: all input modalities present in the request, per-modality validation results, threat scores per modality, trace metadata (source, agent ID, session). 3. Verify alerts are generated for high-severity multi-modal validation failures. 4. Confirm logs enable reconstruction of the full multi-modal input for forensic analysis (with appropriate redaction of sensitive content). | Level 3 because comprehensive multi-modal logging with cross-modality correlation requires significant infrastructure. The challenge is logging enough detail for forensic analysis while handling the storage requirements of multi-modal data (images, audio). Consider logging references (hashes, storage paths) rather than full payloads for large inputs. Alert generation should account for the higher false positive rate of multi-modal detection to avoid alert fatigue. |
| **2.7.5** | **Verify that** cross-modal attack detection identifies coordinated attacks spanning multiple input types (e.g., steganographic payloads in images combined with prompt injection in text) with correlation rules and alert generation, and that confirmed detections are blocked or require HITL (human-in-the-loop) approval. | 3 | D/V | Coordinated multi-modal attacks; attacks that evade per-modality detection by splitting the payload across modalities; combined steganography + prompt injection attacks; distraction attacks (benign text with malicious image, or vice versa). | 1. Design test cases with coordinated cross-modal payloads (e.g., benign text + image containing injected instructions; normal audio + adversarial text). 2. Verify correlation rules exist that analyze inputs across modalities as a group, not just individually. 3. Test that detections trigger blocking or HITL workflow. 4. Verify the HITL interface provides sufficient context (all modalities, detection results, confidence scores) for human review. | This is a forward-looking requirement (Level 3). Cross-modal correlation for attack detection is an emerging research area with limited production tooling. Current practical approach: flag requests where any single modality scores above a suspicion threshold and the combination of modalities is unusual for the user's history. HITL approval workflow requires operational design: queue management, reviewer training, SLA for review time, and fallback behavior if no reviewer is available. The HITL requirement adds operational cost but is appropriate for the highest-risk scenarios. |

## Open Research Questions

- Can cross-modal adversarial attacks be reliably detected with current tooling, or is this fundamentally a research problem?
- How should multi-modal validation handle new modalities (e.g., 3D models, haptic input, sensor data) as AI systems expand?
- What is the performance overhead of comprehensive multi-modal validation, and how does it impact real-time applications?
- How do vision-language model architectures (e.g., GPT-4o, Gemini) change the attack surface compared to separate vision and language models?
- What are effective defenses against typographic attacks on vision models beyond text detection in images?
