# C4.3 Network Security & Access Control

[Back to C04 Index](C04-Infrastructure.md)

## Purpose

Implement zero-trust networking with default-deny policies and encrypted communications. AI infrastructure has a uniquely large attack surface: inference endpoints are internet-facing, training clusters require high-bandwidth interconnects, model registries hold high-value intellectual property, and data pipelines span multiple network zones. This section ensures that network architecture limits blast radius and prevents lateral movement even when individual components are compromised.

## Requirements

| # | Requirement | Level | Role | Threat Mitigated | Verification Approach | Gaps / Notes |
|---|-------------|:-----:|:----:|-----------------|----------------------|--------------|
| **4.3.1** | Verify that network policies enforce default-deny ingress and egress, with only required services explicitly allowed. | 1 | D/V | Lateral movement from a compromised inference pod to internal services (model registry, training data stores, metadata services). Unauthorized data exfiltration from training workloads. The IBM 2026 X-Force Threat Index reported a 44% increase in attacks exploiting public-facing applications with missing access controls. In GPU clusters, a single compromised endpoint can enable lateral movement across the entire model-serving infrastructure through shared drivers and interconnects. | Inspect Kubernetes NetworkPolicy resources for `policyTypes: [Ingress, Egress]` with empty default rules. Verify with network scanning tools (nmap) that unlisted ports/destinations are unreachable. Test that newly deployed pods without explicit policies cannot reach any services. Validate that the CNI plugin actively enforces policies (Calico, Cilium, Antrea). Use Global Network Policies (supported by Calico/Cilium) to secure both namespaced and non-namespaced resources. Verify with eBPF-based flow logs (Cilium Hubble) that only expected traffic patterns exist. | Kubernetes NetworkPolicy alone does not cover all cases -- it requires a CNI that supports it (Calico, Cilium, Antrea). Kubernetes on its own does not enforce network policies; enforcement depends entirely on the CNI policy engine. AWS Security Groups and VPC NACLs are a separate layer. AI workloads often need broad egress for pulling model weights or datasets during initialization -- these should be time-limited or handled via init containers with different network policies. By default, all pods in a Kubernetes cluster communicate freely, making explicit default-deny a critical first step. Consider layered microsegmentation: VLANs (L2), NACLs (L3/4), Kubernetes NetworkPolicies (application layer), and security groups (container level). Uber's microsegmentation deployment limited breach impact to just 3% of infrastructure. |
| **4.3.2** | Verify that AI workloads across environments (development, testing, production) run in isolated network segments with no direct internet access and no shared identity roles, security groups, or cross-environment connectivity. | 1 | D/V | Development environment compromise leading to production access. Staging models trained on production data being accessible from dev networks. Shared IAM roles allowing cross-environment privilege escalation. The 2025 Drift/Salesloft breach demonstrated how compromised OAuth tokens in one environment can cascade across 700+ organizations via supply chain trust. Shadow AI usage in dev environments cost organizations an average of $670,000 more per breach than traditional incidents. | Verify separate VPCs/VNets per environment with no peering. Confirm distinct IAM roles/service accounts per environment. Attempt cross-environment network access and confirm failure. Audit security group membership for cross-environment overlap. Validate that training jobs receive read-only dataset access with write permissions limited to designated output locations. Confirm model-serving endpoints expose only inference APIs without training or data access capabilities. | Common anti-pattern: sharing a model registry across environments with the same credentials. Each environment should have its own registry or use separate credentials with read-only access for lower environments. GPU clusters are often shared across environments due to cost -- this must be mitigated with strong logical isolation. Prescribe distinct security zones for training vs. inference: training jobs get read-only dataset access; serving endpoints expose only inference APIs. SPIFFE-based workload identity frameworks can provide cryptographic identity per workload independent of network location. |
| **4.3.3** | Verify that administrative and remote access protocols and access to cloud metadata services are restricted and require strong authentication. | 1 | D/V | SSRF attacks from ML workloads accessing cloud metadata services (IMDSv1 on AWS returning IAM credentials). Unauthorized SSH/RDP access to GPU nodes. Kubectl access without proper RBAC. In 2025, Microsoft reported that stolen LLM credentials were resold by criminal gangs to build paid bypass services, with costs exceeding $100,000/day per compromised credential. Over 300,000 ChatGPT credentials were exposed via infostealer malware in 2025, confirming that AI platforms face the same credential risk as core enterprise SaaS. | Confirm IMDSv2 is enforced (hop limit = 1, PUT required) or metadata service is blocked. Verify SSH/RDP requires MFA or certificate-based auth (FIDO2 preferred). Check that Kubernetes API access uses RBAC with least privilege. Audit admin access logs for anomalies. Validate time-bound access with automatic permission revocation. Verify software-defined perimeters replace traditional VPNs for administrative access. | IMDSv1 SSRF is a well-documented attack vector. ML workloads are particularly vulnerable because they often process untrusted input that could be crafted to trigger SSRF. Jupyter notebooks with internet access and IMDS access are a common high-risk combination. Block metadata endpoints from all workload pods unless explicitly needed. Cloudflare's zero-trust network access reduced attack surface by 95% versus traditional VPNs. Consider continuous re-verification (Microsoft's AI infrastructure re-verifies every 10 minutes, blocking 94% of lateral movement from compromised credentials). FIDO2 standards and hardware-backed authentication should be preferred for GPU node access. |
| **4.3.4** | Verify that inter-service communication uses mutual TLS with certificate validation and regular automated rotation. | 2 | D/V | Man-in-the-middle attacks on model serving traffic. Interception of model weights during transfer between training and serving infrastructure. Spoofed inference endpoints returning manipulated results. Prompt injection attacks via MITM on inference traffic can enable data exfiltration -- Trend Micro's Pandora PoC demonstrated how crafted documents trigger code execution and exfiltration to C&C servers when network encryption is absent. | Verify service mesh (Istio, Linkerd) or application-level mTLS is configured. Check certificate rotation frequency (should be < 24 hours for short-lived certs). Confirm certificate validation is strict (no `verify=False` or `InsecureSkipVerify`). Test with expired or wrong-CA certificates. Use cert-manager for automated certificate lifecycle management. Verify Gateway API standard implementation with certificate attachment for encrypted end-to-end traffic. | Service mesh mTLS is straightforward for HTTP-based inference APIs but more complex for gRPC model serving (TensorFlow Serving, Triton) and custom protocols. High-bandwidth training interconnects (NCCL, Gloo) may not support mTLS natively -- consider network-level encryption (WireGuard, IPsec) as an alternative for training cluster traffic. NCCL and InfiniBand/RDMA interconnects used in multi-GPU training are particularly challenging: requirement 4.7.7 covers authenticated interconnects at the hardware layer. CNCF projects cert-manager and Envoy are the recommended tooling for certificate lifecycle and service mesh respectively. |
| **4.3.5** | Verify that egress traffic is restricted to approved destinations and all requests are logged. | 3 | D/V | Data exfiltration of model weights or training data via DNS tunneling, HTTPS to attacker-controlled servers, or cloud storage APIs. Command-and-control communication from compromised workloads. Autonomous exfiltration agents using reinforcement learning to identify data and select optimal exfiltration routes have been demonstrated in research (2025). 96% of ransomware attacks in Q3 2025 involved exfiltration, with median time-to-exfiltration of just 2 days and nearly 1-in-5 cases seeing data stolen within the first hour. Google's least-privilege egress controls prevented 87% of data exfiltration attempts. | Configure egress proxy (Squid, Envoy) or cloud-native egress controls (AWS VPC endpoints, Azure Private Link). Verify DNS resolution is controlled and logged. Confirm logging captures destination, payload size, and source workload identity. Test with unauthorized egress attempts. Deploy Calico Egress Gateway for pod-level routing isolation and consistent IP presentation to external services. Use deep packet inspection for encrypted traffic analysis where legally permitted. Integrate with SIEM for anomaly detection on egress patterns. | ML workloads frequently need to download large files (model weights, datasets) from external sources like HuggingFace Hub, PyPI, or cloud storage. Approved destination lists must be maintained and should use domain-based (FQDN) filtering, not just IP-based. DNS-over-HTTPS from within containers can bypass DNS-based filtering -- consider blocking DoH endpoints. Multi-modal AI agents can be tricked via indirect prompt injection (web-based, image-based zero-click, document-based hidden text) into exfiltrating data to attacker-controlled URLs; network-level URL filtering and content inspection of uploaded documents are additional defenses. AI-aware API gateways (Google GKE Inference Gateway, Azure API Management GenAI Gateway, Apache APISIX) now support token-accounting-based rate limiting and cost attribution, which can detect anomalous exfiltration volumes at the inference API layer. Meta's DLP system prevented 89% of attempted data theft using similar egress controls. |

## Open Research Questions

- [ ] How can network policies be dynamically adjusted for ML training jobs that have different network requirements during data loading vs. training phases? (CNCF blueprint suggests phase-aware policies via Calico/Cilium operators, but production implementations remain immature.)
- [ ] What is the performance impact of mTLS on high-throughput inference endpoints serving thousands of requests per second? (Early benchmarks suggest 2-5% overhead with modern service meshes, but impact on GPU-saturated inference with gRPC streaming is less studied.)
- [ ] How should egress policies handle legitimate ML workflow needs (pulling models from HuggingFace Hub, accessing cloud APIs) without overly broad allowlists? (FQDN-based filtering with time-limited init-container exceptions is the current best practice.)
- [ ] Can eBPF-based network observability tools (Hubble, Retina) provide sufficient visibility for detecting anomalous ML workload communication patterns? (Cilium Hubble now provides context-aware flow visibility integrated with Prometheus/Grafana; real-world AI workload anomaly detection benchmarks are still needed.)
- [ ] How effective are AI-aware API gateways (GKE Inference Gateway, Azure GenAI Gateway, APISIX) at detecting token-level exfiltration patterns vs. traditional request-based egress monitoring?
- [ ] Can reinforcement-learning-based autonomous exfiltration agents be reliably detected through network traffic analysis alone, or do they require application-layer inspection?
