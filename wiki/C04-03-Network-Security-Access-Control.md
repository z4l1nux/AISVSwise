# C4.3 Network Security & Access Control

[Back to C04 Index](C04-Infrastructure.md)

## Purpose

Implement zero-trust networking with default-deny policies and encrypted communications. AI infrastructure has a uniquely large attack surface: inference endpoints are internet-facing, training clusters require high-bandwidth interconnects, model registries hold high-value intellectual property, and data pipelines span multiple network zones. This section ensures that network architecture limits blast radius and prevents lateral movement even when individual components are compromised.

| **4.3.1** | Verify that network policies enforce default-deny ingress and egress, with only required services explicitly allowed. | 1 | D/V | Lateral movement from a compromised inference pod to internal services (model registry, training data stores, metadata services). Unauthorized data exfiltration from training workloads. The IBM 2026 X-Force Threat Index reported a 44% increase in attacks exploiting public-facing applications with missing access controls. In GPU clusters, a single compromised endpoint can enable lateral movement across the entire model-serving infrastructure through shared drivers and interconnects. The CrowdStrike 2026 Global Threat Report found average eCrime breakout time dropped to 29 minutes (fastest: 27 seconds), making network microsegmentation essential since manual response cannot keep up. Cilium now enforces identity-based policies at Layer 7 (HTTP methods, paths, headers) using eBPF directly in the kernel, moving beyond traditional IP-centric L3/L4 controls. | Inspect Kubernetes NetworkPolicy resources for `policyTypes: [Ingress, Egress]` with empty default rules. Verify with network scanning tools (nmap) that unlisted ports/destinations are unreachable. Test that newly deployed pods without explicit policies cannot reach any services. Validate that the CNI plugin actively enforces policies (Calico, Cilium, Antrea). Use Global Network Policies (supported by Calico/Cilium) to secure both namespaced and non-namespaced resources. Verify with eBPF-based flow logs (Cilium Hubble) that only expected traffic patterns exist. For AI factory deployments, verify that NVIDIA BlueField DPU-based security (Palo Alto Prisma AIRS on BlueField-3) enforces policies at line speed without consuming host GPU/CPU resources. | Kubernetes NetworkPolicy alone does not cover all cases -- it requires a CNI that supports it (Calico, Cilium, Antrea). Kubernetes on its own does not enforce network policies; enforcement depends entirely on the CNI policy engine. CVE-2024-7598 demonstrated that during namespace termination, network policies can be deleted before pods, creating a brief window where pods run without policy enforcement -- a race condition that must be mitigated with cluster-scoped policies. AWS Security Groups and VPC NACLs are a separate layer. AI workloads often need broad egress for pulling model weights or datasets during initialization -- these should be time-limited or handled via init containers with different network policies. By default, all pods in a Kubernetes cluster communicate freely, making explicit default-deny a critical first step. Consider layered microsegmentation: VLANs (L2), NACLs (L3/4), Kubernetes NetworkPolicies (application layer), and security groups (container level). Gartner predicts that by 2026, 60% of enterprises pursuing zero trust will use more than one form of microsegmentation, up from less than 5% in 2023. Uber's microsegmentation deployment limited breach impact to just 3% of infrastructure. As of March 2026, both Palo Alto Networks Prisma AIRS and Cisco Secure AI Factory are validated for NVIDIA AI Factory deployments, offloading security processing to BlueField DPUs so 100% of host compute remains available for AI workloads. Cisco extends Hybrid Mesh Firewall policy enforcement to BlueField DPUs with integrated AI Defense for multi-agent system security. In March 2026, Illumio launched Illumio Insights with agentless visibility and Network Posture capabilities that analyze live traffic, policy intent, and enforcement alignment to identify lateral movement risks across hybrid AI infrastructure. Cilium 1.19 (February 2026) introduced strict encryption modes for both IPsec and WireGuard where unencrypted inter-node traffic is dropped, shifting from optional to mandatory encryption between nodes -- particularly valuable for regulated AI workloads. Cilium 1.19 also tightened multi-cluster Cluster Mesh defaults so that network policy selectors without an explicit cluster specification now permit only local-cluster traffic, reducing misconfiguration risk in federated AI training environments. Calico 3.31 (late 2025) expanded its eBPF dataplane to handle approximately 15,000 rules per endpoint direction (up from ~2,000), making it viable for large-scale AI deployments with complex microsegmentation requirements. The Calico Winter 2026 release introduced Last Evaluated Policy Metrics that track when network policies last processed traffic, enabling teams to identify and decommission unused policies to maintain a lean least-privilege posture -- critical in AI clusters where policy sprawl from frequent workload churn can leave stale allow-rules open. Calico Winter 2026 also added Deterministic NetworkSet Matching with namespace-aware lexicographic ordering, ensuring predictable and reproducible policy enforcement in large AI environments with hundreds of NetworkSets. Lateral movement now drives over 70% of successful breaches (ColorTokens 2026), making automated policy lifecycle management essential rather than optional. |
| **4.3.2** | Verify that AI workloads across environments (development, testing, production) run in isolated network segments with no direct internet access and no shared identity roles, security groups, or cross-environment connectivity. | 1 | D/V | Development environment compromise leading to production access. Staging models trained on production data being accessible from dev networks. Shared IAM roles allowing cross-environment privilege escalation. The 2025 Drift/Salesloft breach demonstrated how compromised OAuth tokens in one environment can cascade across 700+ organizations via supply chain trust. Shadow AI usage in dev environments cost organizations an average of $670,000 more per breach than traditional incidents. The CrowdStrike 2026 report found 82% of intrusions are now malware-free -- attackers log in with stolen credentials and native admin tools, making network segmentation (not just endpoint detection) the primary containment mechanism. The Unit 42 2026 Global Incident Response Report (based on 750+ engagements) found that identity weaknesses played a material role in nearly 90% of investigations, with the fastest observed attack completing initial access to data exfiltration in just 72 minutes -- 4x faster than the prior year. Over 87% of intrusions involved activity across multiple attack surfaces (endpoints, networks, cloud, SaaS, identity), reinforcing that network segmentation must work in concert with identity controls. SaaS supply chain attacks via OAuth token abuse surged 3.8x since 2022, now accounting for 23% of all incidents. | Verify separate VPCs/VNets per environment with no peering. Confirm distinct IAM roles/service accounts per environment. Attempt cross-environment network access and confirm failure. Audit security group membership for cross-environment overlap. Validate that training jobs receive read-only dataset access with write permissions limited to designated output locations. Confirm model-serving endpoints expose only inference APIs without training or data access capabilities. For GPU-dense environments, verify that BlueField DPU-based hyperscale security firewalls enforce defense-in-depth at AI factory ingress/egress points, with individual workloads able to selectively route traffic through centralized inspection. | Common anti-pattern: sharing a model registry across environments with the same credentials. Each environment should have its own registry or use separate credentials with read-only access for lower environments. GPU clusters are often shared across environments due to cost -- this must be mitigated with strong logical isolation. Prescribe distinct security zones for training vs. inference: training jobs get read-only dataset access; serving endpoints expose only inference APIs. SPIFFE-based workload identity frameworks can provide cryptographic identity per workload independent of network location. CVE-2026-22039 (Kyverno authorization bypass) showed how authenticated users with namespaced Policy permissions could access and mutate resources across the entire cluster, breaking namespace isolation -- a critical boundary in multi-tenant AI environments. NVIDIA's Vera Rubin NVL72 (announced CES 2026) encrypts every bus across 72 GPUs, 36 CPUs, and the entire NVLink fabric -- the first rack-scale platform delivering confidential computing across CPU, GPU, and interconnect domains. As of March 2026, NVIDIA BlueField Astra (BlueField-4 DPUs paired with ConnectX-9 SuperNICs) extends this further by creating isolated control planes where the SuperNIC control plane is fully managed by the DPU, preventing host OS access to network provisioning and enabling hardware-enforced multi-tenant separation with zero configuration drift. Microsoft's Zero Trust for AI (ZT4AI) framework, announced March 2026, extends the traditional Zero Trust reference architecture with an AI-specific pillar covering agent governance, data security across the AI lifecycle, and prompt security -- prescribing dedicated network segments for AI workloads with strict traffic controls and identity-based access for AI principals (agents, services, operators). A formal ZT4AI assessment tool is expected summer 2026. |
| **4.3.3** | Verify that administrative and remote access protocols and access to cloud metadata services are restricted and require strong authentication. | 1 | D/V | SSRF attacks from ML workloads accessing cloud metadata services (IMDSv1 on AWS returning IAM credentials). Unauthorized SSH/RDP access to GPU nodes. Kubectl access without proper RBAC. In 2025, Microsoft reported that stolen LLM credentials were resold by criminal gangs to build paid bypass services, with costs exceeding $100,000/day per compromised credential. Over 300,000 ChatGPT credentials were exposed via infostealer malware in 2025, confirming that AI platforms face the same credential risk as core enterprise SaaS. | Confirm IMDSv2 is enforced (hop limit = 1, PUT required) or metadata service is blocked. Verify SSH/RDP requires MFA or certificate-based auth (FIDO2 preferred). Check that Kubernetes API access uses RBAC with least privilege. Audit admin access logs for anomalies. Validate time-bound access with automatic permission revocation. Verify software-defined perimeters replace traditional VPNs for administrative access. | IMDSv1 SSRF is a well-documented attack vector. ML workloads are particularly vulnerable because they often process untrusted input that could be crafted to trigger SSRF. Jupyter notebooks with internet access and IMDS access are a common high-risk combination. Block metadata endpoints from all workload pods unless explicitly needed. The OpenClaw AI agent crisis of early 2026 (CVE-2026-25253, CVSS 8.8) demonstrated how AI agents connected to corporate systems enable lateral movement: cross-site WebSocket hijacking gave attackers access to Slack messages, emails, cloud documents, and OAuth tokens. Over 40,000 exposed instances were found globally, with 12,000+ confirmed exploitable via RCE. Cloudflare's zero-trust network access reduced attack surface by 95% versus traditional VPNs. Consider continuous re-verification (Microsoft's AI infrastructure re-verifies every 10 minutes, blocking 94% of lateral movement from compromised credentials). FIDO2 standards and hardware-backed authentication should be preferred for GPU node access. In February 2026, Check Point Research demonstrated that AI assistants with web-browsing capabilities (Copilot, Grok) can be weaponized as covert C2 relay channels -- malware instructs the AI to fetch attacker-controlled URLs and return encoded commands, while from a network perspective the traffic appears indistinguishable from normal AI usage. Organizations must treat AI assistant domains as high-value egress points requiring behavioral monitoring and anomaly detection. In January 2026, the "Reprompt" attack against Microsoft Copilot Personal showed that a single click on a crafted Microsoft URL could establish a continuous exfiltration channel via prompt injection in URL parameters, with no plugins or further user interaction required -- enterprise M365 Copilot was not affected, but the attack demonstrates that AI-integrated tools must be treated as potential administrative access surfaces. The FortiGate campaign of January-February 2026 is particularly instructive: a low-skill Russian-speaking threat actor used commercial GenAI services to compromise 600+ FortiGate devices across 55 countries by systematically scanning exposed management ports (443, 8443, 10443, 4443) and attempting default credentials -- the attack succeeded entirely through exposed admin interfaces with single-factor authentication, validating that restricting management port exposure and enforcing MFA are non-negotiable controls even before considering sophisticated exploits. |
| **4.3.4** | Verify that inter-service communication uses mutual TLS with certificate validation and regular automated rotation. | 2 | D/V | Man-in-the-middle attacks on model serving traffic. Interception of model weights during transfer between training and serving infrastructure. Spoofed inference endpoints returning manipulated results. Prompt injection attacks via MITM on inference traffic can enable data exfiltration -- Trend Micro's Pandora PoC demonstrated how crafted documents trigger code execution and exfiltration to C&C servers when network encryption is absent. | Verify service mesh (Istio, Linkerd) or application-level mTLS is configured. Check certificate rotation frequency (should be < 24 hours for short-lived certs). Confirm certificate validation is strict (no `verify=False` or `InsecureSkipVerify`). Test with expired or wrong-CA certificates. Use cert-manager for automated certificate lifecycle management. Verify Gateway API standard implementation with certificate attachment for encrypted end-to-end traffic. For AI inference workloads on Kubernetes 1.36+, evaluate Istio 1.29 ambient mode which eliminates sidecar overhead -- benchmarks show 15% latency reduction and 20% CPU savings compared to sidecar mode on real inference pipelines. Cilium also integrates mTLS directly into the eBPF data plane without sidecars. | Service mesh mTLS is straightforward for HTTP-based inference APIs but more complex for gRPC model serving (TensorFlow Serving, Triton) and custom protocols. High-bandwidth training interconnects (NCCL, Gloo) may not support mTLS natively -- consider network-level encryption (WireGuard, IPsec) as an alternative for training cluster traffic. NCCL and InfiniBand/RDMA interconnects used in multi-GPU training are particularly challenging: requirement 4.7.7 covers authenticated interconnects at the hardware layer. CNCF projects cert-manager and Envoy are the recommended tooling for certificate lifecycle and service mesh respectively. As of early 2026, performance benchmarks (IEEE published) show traditional sidecar mTLS adds up to 166% latency (Istio sidecar) vs. only 8% (Istio ambient) and 33% (Linkerd) -- architecture choice matters significantly for latency-sensitive inference. Wasm extension support in ambient mode remains limited as of March 2026, so evaluate carefully if custom Envoy Filters are needed. Cilium 1.19 (February 2026) introduces ztunnel integration in beta, enabling transparent mTLS encryption and authentication of TCP connections between workloads without sidecar proxies -- a third option alongside Istio ambient and Linkerd for sidecarless mTLS. Cilium 1.19's strict encryption mode (IPsec/WireGuard) drops any unencrypted inter-node traffic, providing a network-layer encryption floor beneath application-level mTLS for defense-in-depth on AI training clusters. |
| **4.3.5** | Verify that egress traffic is restricted to approved destinations and all requests are logged. | 3 | D/V | Data exfiltration of model weights or training data via DNS tunneling, HTTPS to attacker-controlled servers, or cloud storage APIs. Command-and-control communication from compromised workloads. Autonomous exfiltration agents using reinforcement learning to identify data and select optimal exfiltration routes have been demonstrated in research (2025). 96% of ransomware attacks in Q3 2025 involved exfiltration, with median time-to-exfiltration of just 2 days and nearly 1-in-5 cases seeing data stolen within the first hour. Google's least-privilege egress controls prevented 87% of data exfiltration attempts. | Configure egress proxy (Squid, Envoy) or cloud-native egress controls (AWS VPC endpoints, Azure Private Link). Verify DNS resolution is controlled and logged. Confirm logging captures destination, payload size, and source workload identity. Test with unauthorized egress attempts. Deploy Calico Egress Gateway for pod-level routing isolation and consistent IP presentation to external services. Use deep packet inspection for encrypted traffic analysis where legally permitted. Integrate with SIEM for anomaly detection on egress patterns. | ML workloads frequently need to download large files (model weights, datasets) from external sources like HuggingFace Hub, PyPI, or cloud storage. Approved destination lists must be maintained and should use domain-based (FQDN) filtering, not just IP-based. DNS-over-HTTPS from within containers can bypass DNS-based filtering -- consider blocking DoH endpoints. Multi-modal AI agents can be tricked via indirect prompt injection (web-based, image-based zero-click, document-based hidden text) into exfiltrating data to attacker-controlled URLs; network-level URL filtering and content inspection of uploaded documents are additional defenses. AI-aware API gateways (Google GKE Inference Gateway, Azure API Management GenAI Gateway, Apache APISIX) now support token-accounting-based rate limiting and cost attribution, which can detect anomalous exfiltration volumes at the inference API layer. Meta's DLP system prevented 89% of attempted data theft using similar egress controls. As of early 2026, Check Point Research demonstrated that AI assistants with web-fetch capabilities can serve as covert C2 relays -- the traffic appears as normal AI usage, bypassing traditional egress filters. This means egress policies must include behavioral analysis of AI-bound traffic, not just domain allowlisting. IPv6 egress filtering deserves equal attention: without it, spoofed source addresses can leave the network and unauthorized tunneling mechanisms (including 6to4 and Teredo) can bypass IPv4-only egress controls. The Unit 42 2026 report found the fastest exfiltration completed in 72 minutes from initial access -- 4x faster than the prior year -- making pre-deployed automated egress controls essential since manual response cannot keep pace. GKE Inference Gateway now integrates with Google Cloud Model Armor for AI safety checks and content filtering at the gateway level, while Apigee provides token-level rate limiting, quotas, and analytics specifically for inference workloads -- combining network egress controls with application-layer AI-aware inspection. The industry benchmark for AI security incident detection is 18 days mean time to detect (MTTD), but leading organizations achieve under 4 hours through automated monitoring, highlighting the gap between where most teams are and where they need to be. |

## Open Research Questions

- [ ] How can network policies be dynamically adjusted for ML training jobs that have different network requirements during data loading vs. training phases? (CNCF blueprint suggests phase-aware policies via Calico/Cilium operators, but production implementations remain immature.)
- [ ] What is the performance impact of mTLS on high-throughput inference endpoints serving thousands of requests per second? (IEEE-published benchmarks show Istio ambient mode adds only ~8% latency vs. 166% for traditional sidecars. Real-world AI inference on K8s 1.36 + Istio 1.29 ambient shows 15% latency reduction and 20% CPU savings vs. sidecar mode. However, Wasm extension support remains limited and high-throughput gRPC streaming impact on GPU-saturated inference still needs more study.)
- [ ] How should egress policies handle legitimate ML workflow needs (pulling models from HuggingFace Hub, accessing cloud APIs) without overly broad allowlists? (FQDN-based filtering with time-limited init-container exceptions is the current best practice.)
- [ ] Can eBPF-based network observability tools (Hubble, Retina) provide sufficient visibility for detecting anomalous ML workload communication patterns? (Cilium Hubble now provides context-aware flow visibility integrated with Prometheus/Grafana; real-world AI workload anomaly detection benchmarks are still needed.)
- [ ] How effective are AI-aware API gateways (GKE Inference Gateway, Azure GenAI Gateway, APISIX) at detecting token-level exfiltration patterns vs. traditional request-based egress monitoring?
- [ ] Can reinforcement-learning-based autonomous exfiltration agents be reliably detected through network traffic analysis alone, or do they require application-layer inspection?
- [ ] How will NVIDIA Vera Rubin's rack-scale confidential computing (encrypting all GPU/CPU/NVLink buses) change the network security model for multi-tenant GPU clusters? Will hardware-enforced encryption reduce or eliminate the need for software-based network segmentation within a rack?
- [ ] What is the real-world detection efficacy of BlueField DPU-based security (Prisma AIRS on BlueField-3, Cisco Hybrid Mesh Firewall on BlueField-4) against AI-specific lateral movement patterns compared to traditional host-based network controls?
- [ ] How should network security policies account for AI agent frameworks (like OpenClaw) that bridge local execution environments with corporate SaaS via WebSocket and OAuth? The OpenClaw crisis showed that treating agents as low-privilege local processes is fundamentally wrong -- they have the network reach of admin tools.
- [ ] What is the security impact of BlueField Astra's unified DPU-to-SuperNIC control plane on multi-tenant AI factory deployments? Can DOCA-based microservices enforce fine-grained tenant-aware network policies at scale without introducing new control plane attack surfaces?
- [ ] How should egress monitoring adapt to the AI C2 relay threat (Check Point, February 2026), where AI assistants with web-fetch capabilities serve as bidirectional command channels indistinguishable from legitimate AI usage at the network level? Is behavioral analysis of AI-bound traffic sufficient, or do organizations need application-layer inspection of AI assistant interactions?
- [ ] How does the democratization of network infrastructure attacks via GenAI (as demonstrated by the FortiGate campaign where a low-skill actor compromised 600+ devices) change the calculus for network perimeter hardening? Should organizations assume that any exposed management interface will be systematically probed by GenAI-assisted attackers regardless of the attacker's skill level?
- [ ] Can Calico's Last Evaluated Policy Metrics and similar unused-policy detection capabilities be applied to dynamically prune stale network policies in AI clusters where workloads are frequently created and destroyed, and what is the risk of premature policy decommission in burst-compute training scenarios?
