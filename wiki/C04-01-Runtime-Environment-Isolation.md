# C4.1 Runtime Environment Isolation

[Back to C04 Index](C04-Infrastructure.md)

## Purpose

Prevent container escapes and privilege escalation through OS-level isolation primitives. AI workloads -- particularly inference services processing untrusted input and training jobs with access to sensitive data -- present high-value targets. This section ensures that even if an attacker compromises an AI workload process, they cannot escalate privileges, escape the container boundary, or pivot laterally to other infrastructure.

| **4.1.1** | Verify that all AI workloads run with minimal permissions needed on the operating system, by e.g. dropping unnecessary Linux capabilities in case of a container. | 1 | D/V | Privilege escalation from a compromised ML process. Attackers gaining CAP_SYS_ADMIN or CAP_NET_RAW to escape containers or sniff traffic. NVIDIAScape (CVE-2025-23266, CVSS 9.0) demonstrated that NVIDIA Container Toolkit OCI hooks inherit container environment variables, enabling a three-line Dockerfile to achieve full host compromise via LD_PRELOAD injection into the privileged hook process. | Inspect container security contexts / pod specs for `drop: [ALL]` with explicit add-back of only needed capabilities. Run `capsh --print` inside the running container. Verify no containers run as root (UID 0). Verify NVIDIA Container Toolkit is v1.18.0+ (CDI-first mode; v1.18.2 fixed a bug where `NVIDIA_VISIBLE_DEVICES=none` incorrectly exposed all GPUs) and GPU Operator is v25.3.2+ to mitigate CVE-2025-23266. Check that `enable-cuda-compat` hook is disabled if not needed (`features.disable-cuda-compat-lib-hook = true`). **[2025-2026]** Kubernetes 1.33 made user namespaces default-on for pods with `hostUsers: false`, mapping container root to unprivileged host UIDs. **[2026-03]** Kubernetes 1.35 (March 2026) advances user namespaces to beta/on-by-default and introduces fine-grained supplemental groups control (Strict mode) that prevents malicious container images from escalating privileges via group membership. Kubernetes 1.36 (April 2026) graduates user namespaces to stable and allows `hostNetwork: true` pods with `hostUsers: false`, maintaining user namespace isolation even for pods that need host network access. Verify runtimes support user namespaces (containerd or CRI-O required; cri-dockerd does not support them). | Most ML frameworks do not require elevated capabilities. Common exception: some GPU runtimes historically needed CAP_SYS_ADMIN, but modern NVIDIA Container Toolkit works with reduced caps. Ensure GPU device plugin is configured to avoid privileged mode. **[2025-07]** Wiz estimated ~37% of cloud environments were vulnerable to NVIDIAScape, including major cloud providers offering managed GPU AI services. The vulnerability was reported at Pwn2Own Berlin (May 2025) and patched July 2025. **[2025-2026]** Rootless container support in CRI-O 4.0 (2026) now allows Kubernetes clusters to run containers without requiring privileged access, particularly useful for regulated industries running AI workloads. Podman natively supports rootless containers, eliminating an entire class of privilege escalation attacks. **[2026-03]** Kubernetes 1.35 also enforces cgroups v2 by default (v1 disabled), providing enhanced resource isolation and accounting -- all AI cluster nodes should be running cgroups v2-capable kernels. NVIDIA Container Toolkit v1.18.0 switches to CDI-first mode with systemd auto-generation of CDI specs, simplifying GPU device management and reducing misconfiguration risk. |
| **4.1.2** | Verify that workloads are protected by technologies limiting exploitation such as sandboxing, seccomp profiles, AppArmor, SELinux or similar, and that the configuration is appropriate. | 1 | D/V | Exploitation of vulnerabilities in ML frameworks (e.g., deserialization bugs in pickle, protobuf parsing flaws) leading to arbitrary code execution. November 2025 runC vulnerabilities (CVE-2025-31133, CVE-2025-52565, CVE-2025-52881) exploited race conditions and mount manipulation: CVE-2025-31133 bypassed maskedPaths via /dev/null symlink to procfs; CVE-2025-52565 exploited /dev/console bind-mount redirection; CVE-2025-52881 bypassed LSM checks to redirect writes to `/proc/sysrq-trigger` or `/proc/sys/kernel/core_pattern` for full container breakout. All affect every major container runtime and are patched in runc 1.2.8, 1.3.3, 1.4.0-rc.3. **[2026-02]** CVE-2026-24834 (CVSS 9.3) demonstrated that Kata Containers with Cloud Hypervisor were vulnerable to guest VM privilege escalation via pmem/DAX memory corruption -- a container user could overwrite guest VM binaries and achieve root execution inside the microVM, undermining the VM isolation boundary that Kata is specifically deployed to provide. | Check that seccomp profiles are applied (not `unconfined`). Verify AppArmor/SELinux profiles are loaded and enforcing. Audit profile rules to confirm they block dangerous syscalls (e.g., `mount`, `ptrace`, `kexec_load`). For stronger isolation of untrusted model execution, verify gVisor (RuntimeClass `gvisor`), Kata Containers (v3.27.0+ to address CVE-2026-24834), or Firecracker microVMs are deployed. Verify runc is 1.2.8+/1.3.3+/1.4.0-rc.3+ to address CVE-2025-31133/52565/52881. **[2026 seccomp benchmarks]** Tailored seccomp profiles for ML inference containers add only 2-4% performance overhead versus 8-12% for generic profiles; profile the actual syscall surface with `strace` or `seccomp-bpf` tooling to build workload-specific allowlists. | Default Docker seccomp profile blocks ~44 syscalls but may be too permissive for AI workloads. Custom profiles should be tuned -- e.g., ML workloads typically do not need `clone` with `CLONE_NEWUSER`. **[2025-2026 tooling comparison]** gVisor implements ~70-80% of Linux syscalls in a user-space kernel (Go), providing medium isolation with fast startup. gVisor nvproxy supports CUDA workloads (PyTorch, TensorFlow, and LLM inference) with selected GPU models and NVIDIA driver versions; nvproxy proxies ioctl and mmap calls from virtual device files in the sandbox to the host driver after translation. Kata Containers provides hardware-level VM isolation with near-complete Linux compatibility. Firecracker microVMs offer the strongest isolation (~5MB overhead, ~125ms boot, and as fast as ~28ms with snapshot restore) but lack GPU passthrough support. For GPU workloads needing strong isolation, Kata Containers with NVIDIA GPU Operator is currently the most viable option. **[2026-02]** CVE-2026-24834 is a reminder that even VM-level isolation has vulnerabilities -- defense-in-depth (sandboxing + seccomp + runtime monitoring) remains essential regardless of isolation tier. Production platforms like Northflank now process 2M+ isolated workloads/month using infrastructure-adaptive isolation that auto-selects Kata (with Cloud Hypervisor) when nested virtualization is available, falling back to gVisor otherwise. Major cloud providers (AWS, Azure, GCP) have quietly migrated control planes away from runc toward hardware-enforced isolation, underscoring the limitations of namespace-only containment. **[2026-03]** Google launched GKE Agent Sandbox, a CNCF-track Kubernetes extension built on gVisor specifically for AI agent workloads that execute untrusted code. Pod Snapshots technology enables sub-second cold starts (~90% improvement over standard cold boot) by saving and restoring sandbox state. The industry consensus is converging on a 48-hour critical patch window for container runtime CVEs affecting AI infrastructure. |
| **4.1.3** | Verify that workloads run with a read-only root filesystem, and that any writable mounts are explicitly defined and hardened with restrictive options that prevent execution and privilege escalation (e.g., noexec, nosuid, nodev). | 2 | D/V | Post-exploitation persistence via dropped binaries or modified system files. Attackers writing malicious shared libraries or cron entries. NVIDIAScape specifically exploited the hook process's working directory being set to the container's root filesystem to load malicious shared objects. The runC CVE-2025-52565 attack chain similarly depends on bind-mount redirection that writable filesystems make possible. | Verify `readOnlyRootFilesystem: true` in pod security context. Check that writable volume mounts use `noexec,nosuid,nodev`. Attempt to write to the root filesystem and confirm it fails. Inspect mount options via `mount` or `/proc/mounts`. **[2025]** Kubernetes 1.33 stabilized recursive read-only (RRO) mounts via the `recursiveReadOnly` mount option, ensuring sub-mounts within a volume are also read-only -- previous versions allowed writes to sub-mounts even on "read-only" volumes. Verify RRO is enabled for all AI workload volumes. | AI workloads often need writable `/tmp` or model cache directories. These should be tmpfs mounts with size limits and noexec. HuggingFace Transformers and similar libraries cache models to disk -- ensure cache paths are explicitly defined writable volumes, not the root filesystem. Read-only root filesystem would have partially mitigated the NVIDIAScape attack vector by preventing the malicious .so from being present in the container's root filesystem layer at hook execution time. **[2025]** Recursive read-only mounts close a longstanding gap where sub-mount write paths were exploitable on supposedly read-only volumes. This is particularly relevant for AI workloads that mount multiple model and data paths under a single parent volume. **[2026-03]** Kubernetes 1.35 introduces image pull credential verification (KEP-2535, beta/on-by-default) that re-verifies registry credentials even for cached images, preventing unauthorized reuse of pre-pulled AI model images -- this is especially relevant for large model images that tend to persist on nodes across tenant boundaries. |
| **4.1.4** | Verify that runtime monitoring detects privilege-escalation and container-escape behaviors and automatically terminates offending processes. | 3 | D/V | Active exploitation attempts -- kernel exploits (e.g., CVE-2024-1086 netfilter use-after-free, confirmed by CISA in October 2025 as actively exploited in ransomware campaigns by RansomHub and Akira), runC CVEs (CVE-2025-31133, CVE-2025-52565, CVE-2025-52881), Kata Containers guest VM escape (CVE-2026-24834), namespace escapes, or privilege escalation via SUID binaries. **[2026-03]** A sharp uptick in Kubernetes-specific attack tooling has been reported -- from privilege escalation via exposed API servers to cryptominer deployments through compromised pods -- making runtime detection more urgent than ever for AI clusters. | Deploy runtime security tools (Falco, Sysdig Secure, Tetragon) and verify detection rules trigger on: unexpected process execution, sensitive file access (`/etc/shadow`, `/proc/*/mem`), namespace manipulation, raw socket creation, LD_PRELOAD injection into privileged processes, pmem/DAX region writes (relevant to CVE-2026-24834-style attacks on guest VMs), suspicious symlink creation targeting `/proc/sysrq-trigger` or `/proc/sys/kernel/core_pattern` (runC CVE-2025-52881 indicator), and `setns`/`unshare`/`mount` syscalls from container processes (container escape indicators). Test with simulated escape scenarios including NVIDIAScape-style environment variable injection. **[2026]** A complementary deployment approach pairs Falco for broad rule-based detection coverage with Tetragon for Kubernetes-native inline enforcement -- Falco excels at flexible alerting across diverse container runtimes, while Tetragon can block syscalls in real time at the eBPF layer before they complete, without a userspace round-trip. | Tuning is critical to avoid false positives from legitimate ML operations (e.g., large memory allocations, GPU device access). Falco rules need AI-workload-specific tuning. Automated termination carries risk of killing legitimate long-running training jobs -- consider graduated response (alert -> isolate -> terminate). **[2025-2026]** Stanford HAI AI Index Report found publicly reported AI security/privacy incidents rose 56.4% from 2023 to 2024. eBPF-based tools (Tetragon, Tracee) provide kernel-level enforcement with lower overhead than ptrace-based approaches; Tetragon runs at <1% overhead via eBPF kernel hooks. Rootless container configurations reduce attack surface by an estimated 60-80% compared to traditional root containers, and should be the baseline for all AI workloads where GPU drivers permit it. **[2026]** Sysdig published specific Falco rules for detecting the runC CVE-2025-31133/52565/52881 exploitation patterns, focusing on symlink creation over procfs paths -- these rules should be included in any AI cluster's runtime detection baseline. **[2026-03]** A growing production pattern stacks Kubescape + Falco + Trivy + Cilium/Tetragon for comprehensive runtime defense at 1-2.5% combined CPU overhead, correlating alerts into full attack path analysis. Microsoft Defender for Cloud added container runtime antimalware and binary drift detection (February 2026) that blocks unauthorized modifications to container images at runtime -- useful for detecting post-compromise tampering in AI inference containers. |
| **4.1.5** | Verify that high-risk AI workloads run in hardware-isolated environments (e.g., TEEs, trusted hypervisors, or bare-metal nodes) only after successful remote attestation. | 3 | D/V | Side-channel attacks from co-tenant workloads on shared hardware. Compromised hypervisor or host OS accessing model weights or training data in memory. | Verify TEE attestation reports (e.g., Intel TDX attestation, AMD SEV-SNP attestation) before workload deployment. Confirm hardware isolation (dedicated nodes or VM-level isolation). Check attestation verification is integrated into the deployment pipeline. For GPU workloads, verify composite attestation across CPU, GPU, NIC/DPU, and storage components. Validate attestation via NVIDIA's five-certificate chain with OCSP revocation checks and Reference Integrity Manifest (RIM) file comparison. | **[2025-2026 update]** GPU confidential computing has matured significantly. NVIDIA Hopper (H100) supports single-GPU pass-through for confidential compute with AMD SEV-SNP, but NVLink traffic between GPUs remains unencrypted in this generation. **[2026-03]** NVIDIA Blackwell (HGX/DGX B200) now supports multi-GPU passthrough in TEE mode -- the R590 TRD1 driver enables 1, 2, 4, or 8 GPU configurations within a single Confidential VM, a major step up from Hopper's single-GPU limitation. Blackwell introduces NVLink encryption and TDISP (Trusted Device Interface Security Protocol) for PCIe-level encryption and integrity, using SPDM for device authentication and IDE for traffic protection -- eliminating the software bounce-buffer overhead required on Hopper. Vera Rubin NVL72 systems further extend this with attestation services that verify cryptographic proof of compliance across the full compute stack. Kata Containers + CNCF Confidential Containers (CoCo) stack provides the Kubernetes-native deployment path: RuntimeClass selection for pod-level isolation, Dynamic Resource Allocation (DRA) for GPU assignment, and CDI for device mapping. Note that CVE-2026-24834 demonstrated that even Kata's VM boundary can be compromised -- ensure Kata v3.27.0+ is deployed and combine with CoCo's TEE-backed attestation for defense-in-depth. Intel TDX and AMD SEV-SNP are mature for CPU workloads. |

## Tooling & Implementation Landscape (updated March 2026)

### Isolation Technologies for AI Workloads

| Technology | Isolation Level | GPU Support | Startup Time | Memory Overhead | Best For |
|-----------|----------------|-------------|-------------|-----------------|----------|
| **Standard containers** (Docker/containerd) | Namespace/cgroup (shared kernel) | Full (NVIDIA Container Toolkit) | Fast | Low | Trusted workloads in single-tenant environments |
| **gVisor** | User-space kernel (syscall interception) | CUDA via nvproxy (selected GPU models/driver versions; proxies ioctl/mmap) | Fast | Medium | Semi-trusted workloads; ~70-80% syscall coverage. nvproxy supports PyTorch, TensorFlow, and LLM inference workloads. |
| **Kata Containers** (v3.27.0+) | Hardware VM isolation (KVM/Cloud Hypervisor) | Full (NVIDIA GPU Operator integration, including confidential containers) | ~200ms | Higher (~tens of MB/container) | Multi-tenant AI, compliance-heavy workloads, GPU confidential computing. Ensure v3.27.0+ for CVE-2026-24834 fix. |
| **Firecracker microVM** | Hardware VM isolation (minimal VMM) | None (no GPU passthrough, no hardware accelerators) | ~125ms (cold boot); ~28ms with snapshot restore | Extremely low (~5MB/microVM) | Untrusted AI agent code execution, CPU-only inference. Supports ~150 VMs/second/host. Powers AWS Lambda and Fargate (trillions of executions monthly). |
| **GKE Agent Sandbox** | gVisor-based (CNCF-track project) | CUDA via gVisor nvproxy | Sub-second (with Pod Snapshots) | Medium | AI agent workloads executing untrusted or user-generated code. Pod Snapshots save/restore sandbox state for ~90% cold start improvement. |
| **Confidential Containers** (CoCo) | TEE-backed VM (AMD SEV-SNP / Intel TDX) | NVIDIA Hopper (single-GPU), Blackwell (1/2/4/8 GPU via R590 TRD1 driver) | Medium-slow | Higher | High-risk workloads with model/data confidentiality requirements. Blackwell adds NVLink encryption + TDISP. |

### Runtime Security Monitoring

| Tool | Approach | AI Workload Considerations |
|------|----------|---------------------------|
| **Falco** | Kernel module / eBPF syscall monitoring; modern eBPF driver uses CO-RE (Compile Once, Run Everywhere) | Best for broad rule-based detection across diverse runtimes. Requires AI-specific rule tuning to avoid false positives from GPU device access, large memory allocations. Published detection rules for runC CVE-2025-31133/52565/52881 symlink exploitation patterns. |
| **Tetragon (Cilium)** | eBPF-based enforcement at kernel level; <1% overhead | Best for Kubernetes-native inline enforcement. Can block syscalls in real time before they complete, killing processes or denying operations at the eBPF layer without userspace round-trip. Complementary to Falco: use Falco for detection breadth, Tetragon for enforcement. |
| **Tracee (Aqua)** | eBPF-based runtime detection | Signature and behavioral detection; supports custom rules for ML-specific patterns |
| **Kubescape (ARMO)** | eBPF-based posture and runtime scanning; CNCF project | Combines configuration scanning, image vulnerability detection, and runtime anomaly detection. MCP server integration enables natural-language security queries. Pairs well with Falco + Tetragon in the 1-2.5% overhead stack. |
| **Sysdig Secure** | Kernel module / eBPF with Falco rules | Enterprise features including audit trail and compliance reporting |
| **Microsoft Defender for Cloud** | Agent-based container runtime monitoring | Binary drift detection blocks unauthorized image modifications at runtime. Container-level antimalware scanning added February 2026. Enterprise integration with Azure security posture management. |

### Key NVIDIA GPU Security Components

- **NVIDIA Container Toolkit v1.18.0+**: CDI-first mode with systemd auto-generation of CDI specs; v1.18.2 fixed a bug where `NVIDIA_VISIBLE_DEVICES=none` incorrectly exposed all GPUs to the container (v1.17.8+ patched NVIDIAScape)
- **NVIDIA GPU Operator**: Kubernetes-native GPU lifecycle management; supports Kata + Confidential Containers
- **NVIDIA MIG (Multi-Instance GPU)**: Hardware partitioning of A100/H100/B200 GPUs for tenant isolation
- **NVIDIA Confidential Computing**: Hopper (single-GPU TEE), Blackwell (1-8 GPU TEE via R590 driver + TDISP PCIe encryption + NVLink encryption), Vera Rubin NVL72 (full-stack attestation services)
- **NVIDIA Attestation**: Five-certificate chain validation with OCSP revocation checks; measurements compared against Reference Integrity Manifest (RIM) files covering device identity, VBIOS, and firmware
- **Container Device Interface (CDI)**: Maps GPU/NIC topology into pods; critical for RDMA and PCIe path optimization

## Open Research Questions

- [ ] What is the real-world performance overhead of gVisor vs. Kata Containers for GPU-accelerated ML inference workloads? *Partial answer (2026): Kata with NVIDIA GPU Operator is production-viable; gVisor nvproxy supports CUDA workloads (PyTorch, TensorFlow, LLMs) via ioctl/mmap proxying to host driver but is limited to selected GPU models and driver versions. Firecracker lacks GPU passthrough entirely but achieves ~28ms boot with snapshot restore for CPU-only agent sandboxing.*
- [ ] How effective are current seccomp profiles at blocking ML-framework-specific exploitation chains (e.g., pickle deserialization leading to arbitrary syscalls)? *Partial answer (2026): Tailored seccomp profiles block ~85% of dangerous syscalls with only 2-4% overhead for ML inference. However, no published benchmark specifically maps pickle/protobuf exploitation chains to syscall sequences.*
- [ ] Can eBPF-based runtime enforcement (Tetragon, Tracee) provide lower overhead than ptrace-based approaches for high-throughput inference? *Largely answered (2026): Tetragon runs at <1% overhead via eBPF kernel hooks, can block syscalls in real time before completion. Modern Falco uses CO-RE (Compile Once, Run Everywhere) eBPF driver. The recommended approach is pairing Falco for detection breadth with Tetragon for inline enforcement -- both are production-ready for Kubernetes AI clusters.*
- [ ] How does composite attestation (CPU + GPU + NIC/DPU + storage) scale across multi-node training clusters? *New question added 2026-03: NVIDIA is working on this with the CoCo community but multi-node attestation chains are not yet standardized.*
- [ ] What is the security impact of TDISP (Trusted Device Interface Security Protocol) on PCIe-level attack vectors for GPU workloads? *Partial answer (2026-03): TDISP uses SPDM for device authentication and IDE for PCIe traffic encryption/integrity. Eliminates software bounce-buffer overhead from Hopper era. Independent security analysis of TDISP's threat model is still pending.*
- [ ] How does CVE-2026-24834 (Kata pmem/DAX bypass) affect the trust model for VM-isolated AI workloads? *New question added 2026-03: The vulnerability demonstrated that even hardware VM boundaries can be subverted via memory-mapped device interfaces. What other device passthrough paths (GPU, NIC, storage) might have analogous trust assumptions?*
- [ ] Are the runC race condition patterns (CVE-2025-31133/52565/52881) indicative of a broader class of TOCTOU vulnerabilities in container runtimes? *New question added 2026-03: All three CVEs exploited race windows during container setup for mount manipulation. Similar timing windows may exist in other container runtimes (crun, youki) or in the OCI spec itself. No systematic audit of container runtime TOCTOU surfaces has been published.*
- [ ] How do Kubernetes user namespaces interact with GPU device access requirements for AI workloads? *Partial answer (2026-03): User namespaces progressed from K8s 1.33 (default-on) through 1.35 (beta/on-by-default) to 1.36 (stable). K8s 1.36 additionally allows hostNetwork pods with user namespace isolation. GPU device files and NVIDIA driver interactions may require specific UID mappings -- NVIDIA Container Toolkit v1.18.0 CDI-first mode simplifies device mapping but compatibility testing with user namespaces across GPU passthrough configurations is still needed.*
- [ ] How effective is the emerging Kubescape + Falco + Trivy + Tetragon stack for AI-specific runtime defense, and what are the diminishing returns of adding more layers? *New question added 2026-03: The pattern claims 1-2.5% combined CPU overhead with full attack path correlation, but independent benchmarks on GPU-heavy AI clusters are lacking.*
