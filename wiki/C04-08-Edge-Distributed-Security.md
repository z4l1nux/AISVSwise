# C4.8 Edge & Distributed AI Security

[Back to C04 Index](C04-Infrastructure.md)

## Purpose

Secure distributed AI deployments including edge computing, federated learning, and multi-site architectures. Edge and mobile AI deployments operate in fundamentally hostile environments -- devices are physically accessible to attackers, network connectivity is intermittent, compute resources are constrained, and models deployed to devices can be extracted, reverse-engineered, or tampered with. This section addresses the unique security challenges of running AI inference on devices outside the controlled data center perimeter.

## Requirements

| # | Requirement | Level | Role | Threat Mitigated | Verification Approach | Gaps / Notes |
|---|-------------|:-----:|:----:|-----------------|----------------------|--------------|
| **4.8.1** | Verify that edge AI devices authenticate to central infrastructure using mutual authentication with certificate validation (e.g., mutual TLS). | 1 | D/V | Rogue devices impersonating legitimate edge AI nodes to receive model updates, exfiltrate data, or inject poisoned model updates. Man-in-the-middle attacks on device-to-cloud communication. | Verify mTLS configuration on both device and server side. Check certificate validation includes chain verification, revocation checking (CRL/OCSP), and hostname verification. Test with invalid or expired certificates and confirm rejection. Verify certificate provisioning uses secure enrollment (EST, SCEP, or device attestation). | Certificate management at scale (thousands of edge devices) requires automated provisioning and renewal. Short-lived certificates (< 24 hours) via device attestation are preferred over long-lived device certificates that are difficult to revoke. Consider device identity platforms (Azure IoT Hub, AWS IoT Core) that handle certificate lifecycle. Pre-provisioned device certificates must be stored in secure hardware (TPM, secure enclave). |
| **4.8.2** | Verify that models deployed to edge or mobile devices are cryptographically signed during packaging, and that the on-device runtime validates these signatures or checksums before loading or inference; unverified or altered models must be rejected. | 1 | D/V | Model tampering during distribution (man-in-the-middle replacing model files). Modified models that produce incorrect or biased results. Backdoored models deployed to edge devices via compromised update channels. | Verify model packaging includes digital signature (code signing certificate or custom signing scheme). Check that the on-device runtime validates signatures before model loading. Test with a modified model file and confirm it is rejected. Verify signature validation cannot be bypassed by downgrading to an unsigned model format. | Model signing should cover the complete model artifact: weights, configuration, tokenizer, and any custom operators. For mobile apps, platform code signing (Android APK signing, iOS code signing) provides a baseline but does not cover dynamically downloaded model files. Consider using content integrity verification (hash trees/Merkle trees) for large model files to enable partial verification. |
| **4.8.3** | Verify that edge devices implement secure boot with verified signatures and rollback protection to prevent firmware downgrade attacks. | 2 | D/V | Firmware downgrade to a version with known vulnerabilities. Persistent rootkit installed by replacing the boot chain. Physical attacker modifying boot firmware to bypass security controls. | Verify secure boot chain from hardware root of trust through bootloader, OS, and application. Confirm anti-rollback counters prevent firmware downgrade. Test downgrade attempt and verify rejection. Check that all boot chain components are signed by authorized keys. | Secure boot is well-established for x86 (UEFI Secure Boot) and ARM (ARM Trusted Firmware). Edge AI devices (NVIDIA Jetson, Google Coral, Intel NUC) have varying secure boot support. NVIDIA Jetson supports fuse-based secure boot with anti-rollback. Raspberry Pi-based edge AI deployments have limited secure boot options. Consider the full boot chain including GPU/accelerator firmware. |
| **4.8.4** | Verify that mobile or edge inference applications implement platform-level anti-tampering protections (e.g., code signing, verified boot, runtime integrity checks) that detect and block modified binaries, repackaged applications, or attached instrumentation frameworks. | 2 | D/V | Application repackaging with modified inference logic (returning manipulated results). Attacker attaching debuggers (Frida, lldb) to inspect model weights in memory. Instrumentation frameworks hooking into inference calls to intercept inputs/outputs. | Verify code signing is enforced (Android: V2+ signing, iOS: mandatory). Check for runtime integrity mechanisms: root/jailbreak detection, debugger detection, hooking framework detection (Frida, Xposed, Substrate). Test with instrumentation tools and verify detection. | Anti-tampering is a defense-in-depth measure, not a guarantee -- determined attackers can bypass runtime checks. Layered approach recommended: code signing (baseline), root/jailbreak detection (detect hostile environment), integrity checks (detect modification), obfuscation (increase effort). Commercial solutions: Google Play Integrity API, Apple DeviceCheck/App Attest, third-party RASP tools (Guardsquare, Zimperium). |
| **4.8.5** | Verify that distributed AI coordination uses Byzantine fault-tolerant consensus mechanisms with participant validation and malicious node detection. | 3 | D/V | Poisoning attacks in federated learning where malicious participants submit crafted model updates to corrupt the global model. Sybil attacks where an attacker creates multiple fake participants to amplify poisoning. Free-rider attacks where participants submit minimal updates while benefiting from the global model. | Verify BFT consensus protocol handles up to f Byzantine faults among 3f+1 participants. Check participant validation (identity verification, contribution quality scoring). Test with simulated malicious participants submitting poisoned updates and verify detection/exclusion. | BFT consensus for federated learning is an active research area. Practical approaches include: robust aggregation (trimmed mean, Krum, FoolsGold), anomaly detection on model updates, and reputation systems. Full BFT consensus adds significant communication overhead (O(n^2) messages). For most federated learning deployments, robust aggregation with anomaly detection is more practical than formal BFT consensus. |
| **4.8.6** | Verify that edge-to-cloud communication supports bandwidth throttling, data compression, and secure offline operation with encrypted local storage. | 3 | D/V | Bandwidth exhaustion attacks consuming expensive cellular or satellite connectivity. Data loss during extended offline periods. Unencrypted local storage exposing inference data or cached model updates when device is physically compromised. | Verify bandwidth management policies are configured (rate limiting, compression). Test offline operation: disconnect device and confirm inference continues with cached model. Verify local storage encryption (device-level FDE or application-level encryption). Check that queued data is encrypted and integrity-protected during offline periods. | Edge AI devices in industrial/military/remote settings may operate offline for extended periods. Local storage must protect: cached model weights, inference results queue, local training data (for federated learning), and device configuration. Consider storage encryption key management -- keys derived from network authentication cannot be used during offline operation. Hardware-backed key storage (TPM, secure enclave) enables offline encryption without network dependency. |
| **4.8.7** | Verify that on-device inference runtimes enforce process, memory, and file access isolation to prevent model dumping, debugging, or extraction of intermediate embeddings and activations. | 3 | D/V | Model extraction by dumping process memory containing model weights. Intermediate activation extraction revealing sensitive information about training data. Debugging-based extraction of model architecture and parameters. | Verify inference runtime runs in an isolated process with restricted debugging (ptrace disabled). Check that memory regions containing model weights are marked as non-dumpable. Verify file access is restricted to prevent reading model files from the filesystem. Test with memory dump tools and confirm model weights are not recoverable. | Full protection against model extraction on a device the attacker physically controls is extremely difficult. This is a defense-in-depth measure. Techniques include: disabling core dumps, using `prctl(PR_SET_DUMPABLE, 0)`, memory encryption (ARM MTE, pointer authentication), and TEE-based inference (OP-TEE, ARM TrustZone). On Android, consider running inference in a separate isolated process with restricted SELinux policy. |
| **4.8.8** | Verify that model weights and sensitive parameters stored locally are encrypted using hardware-backed key stores or secure enclaves (e.g., Android Keystore, iOS Secure Enclave, TPM/TEE), with keys inaccessible to user space. | 3 | D/V | Physical extraction of model weights from device storage (flash memory, SD card). Software-based key extraction from application memory. Cloning attacks where model is copied from one device to another. | Verify encryption keys are stored in hardware-backed keystore (Android Keystore with StrongBox, iOS Secure Enclave, TPM 2.0). Confirm keys are marked as non-exportable. Check that decryption requires device authentication (biometric or PIN). Test that encrypted model files are not decryptable on a different device. | Hardware-backed encryption binds model weights to a specific device, preventing model copying. Key consideration: model loading performance -- decryption of large model files (hundreds of MB) from hardware-backed keys may have throughput limitations. Consider encrypting only the most sensitive layers/components rather than the entire model file. AES-256-GCM via hardware accelerator typically achieves 1-4 GB/s on modern mobile SoCs, which is adequate for most edge model sizes. |
| **4.8.9** | Verify that models packaged within mobile, IoT, or embedded applications are encrypted or obfuscated at rest, and decrypted only inside a trusted runtime or secure enclave, preventing direct extraction from the app package or filesystem. | 3 | D/V | Model extraction from APK/IPA files by unpacking the application bundle. Filesystem-level extraction of model files from rooted/jailbroken devices. Competitive intelligence through reverse engineering of proprietary models. | Verify model files within the application package are encrypted (not just compressed). Check that decryption occurs in memory within a secure context, not to a temporary file. Test by extracting the APK/IPA and attempting to load the model file -- confirm it is not usable. Verify model format is not trivially identifiable (e.g., not a plain TFLite or ONNX file). | Common mobile ML frameworks (TFLite, Core ML, ONNX Runtime) load models from plaintext files by default. Custom model loading with decryption is required. Solutions: custom model encryption with application-specific keys, TFLite model encryption (experimental), and ONNX Runtime custom model loading. Obfuscation alone (renaming files, custom formats) provides minimal security -- encryption is necessary. Consider that determined attackers can extract decrypted models from memory even with encrypted-at-rest protections. |

## Open Research Questions

- [ ] How effective are current anti-tampering techniques (obfuscation, integrity checks) against determined attackers with physical access to edge AI devices?
- [ ] What are the practical tradeoffs between model protection strength and inference latency on resource-constrained edge devices?
- [ ] How can federated learning aggregation be made robust against Byzantine participants without prohibitive communication overhead?
- [ ] What is the minimum hardware security capability (TPM, TEE, secure enclave) needed to meaningfully protect model weights on edge devices?
- [ ] How should model update delivery be secured for edge devices with intermittent connectivity and limited bandwidth?
