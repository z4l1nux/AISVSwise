# C4.7 AI Hardware Security

[Back to C04 Index](C04-Infrastructure.md)

## Purpose

Secure AI-specific hardware components including GPUs, TPUs, and specialized AI accelerators. AI hardware presents unique security challenges absent from traditional computing: GPU memory is shared across workloads by default, accelerator interconnects carry sensitive model data at high bandwidth, firmware vulnerabilities can compromise entire GPU clusters, and side-channel attacks on accelerators can leak model weights or training data. This section addresses hardware-level security controls specific to AI accelerator infrastructure.

| **4.7.1** | Verify that before workload execution, AI accelerator integrity is validated using hardware-based attestation mechanisms (e.g., TPM, DRTM, or equivalent). | 2 | D/V | Compromised GPU firmware or driver that intercepts model weights, training data, or inference inputs. Rootkit-level persistence in accelerator firmware surviving OS reinstalls. | Check for TPM-based measured boot that includes GPU firmware measurements in the boot chain. Verify DRTM (Dynamic Root of Trust for Measurement) is supported and enabled. Validate attestation reports include GPU firmware version and configuration hashes. | **Hopper/Blackwell attestation (2024-2025):** NVIDIA H100 GPUs support hardware attestation via the NVIDIA Attestation SDK, anchored in an on-die hardware root of trust (RoT) using device-unique ECC-384 key pairs. Attestation reports are verified against the NVIDIA Certificate Authority with OCSP revocation checks. AMD and Intel accelerator attestation support remains limited compared to NVIDIA. |
| **4.7.2** | Verify that accelerator (GPU) memory is isolated between workloads through partitioning mechanisms with memory sanitization between jobs. | 2 | D/V | Cross-workload data leakage through GPU memory. Previous tenant's model weights, training data, or inference inputs remaining in GPU memory and being read by the next workload. | Verify GPU memory partitioning is active (NVIDIA MIG, AMD memory partitioning). Confirm memory sanitization occurs between job switches. Test by writing known patterns to GPU memory, terminating the workload, and checking if the pattern persists for the next workload. | **LeftoverLocals (CVE-2023-4969, disclosed January 2024):** Trail of Bits researchers demonstrated that GPUs from Apple, AMD, Qualcomm, and Imagination fail to properly clear local memory between application executions. On an AMD Radeon RX 7900 XT, the vulnerability leaks ~5.5 MB per GPU invocation -- for a 7B parameter LLM on llama.cpp, this totals ~181 MB per query, enough to reconstruct full LLM responses. The attack requires only co-resident access to GPU compute APIs (OpenCL, Vulkan, Metal) with no elevated privileges, implementable in under 10 lines of code. Notably, NVIDIA GPUs were found not vulnerable to LeftoverLocals. Patch status as of early 2024: Apple patched A17/M3 but M2 remains affected; Qualcomm issued partial firmware fix (v2.07) for some devices; AMD was still investigating mitigations; Imagination released fix in DDK 23.3. Without explicit sanitization, `cudaFree()` does not zero memory by default on NVIDIA GPUs. NVIDIA driver version 525+ supports `RmProfilingAdminOnly` to prevent memory snooping. CUDA 12.0+ supports `cudaMemPoolSetAttribute` for secure memory pools. Multi-tenant GPU environments must enforce memory clearing between tenants at the driver/hypervisor level. |
| **4.7.3** | Verify that AI accelerator firmware is version-pinned, signed, and attested at boot; unsigned or debug firmware is blocked. | 2 | D/V | Supply chain attack via compromised firmware update. Downgrade attack to firmware version with known vulnerabilities. Debug firmware left in production enabling memory inspection or bypassing security features. | Verify firmware signing is enforced (check GPU driver/firmware update policies). Confirm firmware version matches the approved version in configuration management. Check that debug or unsigned firmware is rejected at boot. Audit firmware update procedures for approval workflow. | NVIDIA GPU firmware (VBIOS) is signed by NVIDIA and verified at boot via the on-die hardware root of trust. On H100/Blackwell, the secure boot sequence establishes a chain of trust via SPDM sessions and cryptographically signed attestation reports. Third-party accelerators and FPGAs may have weaker firmware signing. Driver versions should also be pinned -- NVIDIA driver updates have historically introduced both security fixes and new vulnerabilities. **Supply chain risks (2024-2026):** CNAS research highlights that hardware trojans (malicious modifications to chip designs) can steal data, weaken security, and sabotage systems. University of Missouri researchers (2025) introduced AI-driven methods using LLMs to scan chip designs for hidden threats. The CHIPS Act is driving domestic semiconductor manufacturing ramps in 2026, but supply chain integrity remains a concern for imported accelerators. CNAS recommends NIST develop AI hardware security standards building on FIPS 140-3, coordinating with industry consortia like the Open Compute Project. |
| **4.7.4** | Verify that VRAM and on-package memory are zeroed between jobs/tenants and that device reset policies prevent cross-tenant data remanence. | 2 | D/V | Data remanence in GPU VRAM allowing next tenant to read previous tenant's model weights, activations, or training data. Sensitive data (PII in training sets, proprietary model parameters) persisting in accelerator memory. | Verify GPU reset occurs between workloads (check for `nvidia-smi -r` or equivalent). Confirm VRAM is zeroed (write test patterns, reset, read back). Check device reset policy in the container runtime or hypervisor. Validate that HBM (High Bandwidth Memory) on-package is included in sanitization. | This overlaps with 4.7.2 but focuses specifically on memory sanitization rather than isolation during concurrent execution. Full GPU reset (`nvidia-smi --gpu-reset`) is more thorough than memory deallocation but causes a brief service interruption. HBM3 on H100/A100 has large capacity (80GB+) -- zeroing takes measurable time. Consider async zeroing or NVIDIA's compute sanitizer tool for validation. **Confidential computing memory protection:** On H100 in CC-On mode, on-package HBM3 memory is considered secure against standard physical attack tools and remains unencrypted for performance, while all data crossing the PCIe bus is encrypted through bounce buffers. On Blackwell, all data in GPU memory is encrypted in CC mode, including model weights, training data, and inference results. This provides defense-in-depth against data remanence -- even if memory is not zeroed, contents are encrypted with session-specific keys. |
| **4.7.5** | Verify that partitioning/isolation features (e.g., MIG/VM partitioning) are enforced per tenant and prevent peer-to-peer memory access across partitions. | 2 | D/V | Cross-tenant memory access via CUDA IPC, peer-to-peer DMA, or NVLink. One tenant reading another tenant's GPU memory through shared address spaces. | Verify MIG (Multi-Instance GPU) is enabled and configured per tenant. Confirm CUDA IPC is disabled across partition boundaries. Test peer-to-peer memory access between partitions and confirm it fails. Check that NVLink is restricted to authorized partition pairs only. | NVIDIA MIG (available on A100, H100, Blackwell) creates up to 7 isolated GPU instances with separate memory, cache, and compute resources. Without MIG, CUDA applications on the same GPU can potentially access each other's memory via `cudaIpcGetMemHandle`. MIG partitions have their own error isolation -- a fault in one partition does not affect others. Limitation: MIG reduces flexibility in GPU memory allocation. **Confidential computing as complementary isolation:** On Blackwell with CC mode and TDISP/IDE support, direct communication between GPU and Confidential Virtual Machine uses inline encryption, eliminating the bounce buffer latency seen on H100. This provides hardware-enforced isolation beyond MIG partitioning. For multi-GPU workloads, NVLink pathways are also encrypted on Blackwell, preventing cross-partition snooping at the interconnect level. |
| **4.7.6** | Verify that hardware security modules (HSMs) or equivalent tamper-resistant hardware protect AI model weights and cryptographic keys, with certification to an appropriate assurance level (e.g., FIPS 140-3 Level 3 or Common Criteria EAL4+). | 3 | D/V | Physical extraction of model weights or encryption keys from storage media. Key compromise from software-based key storage. Regulatory non-compliance for AI systems handling sensitive data (financial, healthcare, government). | Verify HSM deployment (on-premise HSMs or cloud HSM services). Check certification level (FIPS 140-3 Level 3 requires physical tamper resistance). Confirm model encryption keys are generated and stored within the HSM boundary. Validate key usage logging and access controls. | FIPS 140-3 Level 3 HSMs provide physical tamper evidence and response mechanisms. Cloud HSM services (AWS CloudHSM, Azure Dedicated HSM, GCP Cloud HSM) offer managed FIPS-certified HSMs. Key consideration for AI: model weight encryption keys may need to be used at high frequency during model loading -- HSM performance (operations per second) must be validated against workload requirements. Consider envelope encryption where the HSM holds master keys and data keys are derived per model. **Hardware-Enabled Mechanisms (HEMs) for AI chips (2024-2025):** CNAS proposes embedding security features directly into AI chips, including offline licensing (chips requiring active licenses that deplete during operation), delay-based geolocation verification using encrypted pings, bandwidth bottlenecking to restrict unauthorized large-scale training, and compute metering for privacy-preserving compliance verification. These mechanisms are designed to avoid backdoor pitfalls (lessons from the Clipper chip) while enabling hardware-rooted trust. NIST standardization of AI hardware security features building on FIPS 140-3 is recommended. |
| **4.7.7** | Verify that accelerator interconnects (NVLink/PCIe/InfiniBand/RDMA/NCCL) are restricted to approved topologies and authenticated endpoints; plaintext cross-tenant links are disallowed. | 3 | D/V | Eavesdropping on inter-GPU communication carrying model gradients, activations, or weights. Man-in-the-middle attacks on distributed training traffic. Unauthorized nodes joining training clusters via unsecured interconnects. | Verify InfiniBand subnet management restricts partition membership. Confirm PCIe ACS (Access Control Services) is enabled to prevent peer-to-peer DMA attacks. Check NVLink topology matches approved configuration. Verify NCCL communication uses encrypted channels where supported. | **Generational progress on interconnect encryption (2024-2026):** On H100 in CC mode, all command buffers and CUDA kernels are encrypted and signed before crossing the PCIe bus, using software-based bounce buffers at ~4 GB/s throughput (limited by CPU encryption performance). NVLink on H100 does not natively encrypt in non-CC mode. **Blackwell (B200/GB200, 2025):** Introduced TDISP and IDE protocols enabling direct inline encryption between GPU and Confidential VM, eliminating bounce buffer latency. NVLink pathways are encrypted in CC mode, securing GPU-to-GPU data movement at scale. NVIDIA claims near-identical throughput performance compared to unencrypted modes. **Vera Rubin NVL72 (announced CES 2026, shipping H2 2026):** Third-generation NVIDIA Confidential Computing expands to full rack-scale -- encrypting every bus across 72 Rubin GPUs, 36 Vera CPUs, and the entire NVLink fabric as a unified trusted execution environment. This represents the first rack-scale confidential computing platform with end-to-end encrypted interconnects. InfiniBand supports partition keys (PKeys) for access control but not encryption natively. RDMA traffic is typically unencrypted -- consider IPsec or MACsec for network-level encryption of RoCE traffic. |
| **4.7.8** | Verify that accelerator telemetry (power draw, temperature, error correction, performance counters) is exported to centralized security monitoring and alerts on anomalies indicative of side-channels or covert channels. | 3 | D | Side-channel attacks using power analysis or electromagnetic emanation to extract model weights or training data. Covert channel communication via GPU utilization patterns. Cryptomining detection. Hardware fault exploitation. | Verify NVIDIA DCGM (Data Center GPU Manager) or equivalent is deployed and exporting metrics. Check that alerting rules exist for: unusual power consumption patterns, abnormal temperature profiles, excessive ECC errors, unexpected performance counter values. Validate alert routing and response procedures. | **Demonstrated GPU side-channel attacks (2023-2025):** (1) **GPU.zip (IEEE S&P 2024):** Exploits software-transparent graphical data compression present in nearly all modern GPUs (AMD, Apple, Arm, Intel, NVIDIA, Qualcomm) to steal pixel data via rendering time differences. In Chrome, can extract usernames from cross-origin iframes in 30 minutes (Ryzen) to 215 minutes (Intel) at 97-98% accuracy. As of 2024, no GPU vendor or Google has patched this. (2) **BarraCUDA (USENIX Security 2025):** Uses correlation electromagnetic analysis (CEMA) to extract neural network weights and biases from NVIDIA Jetson Nano and Orin Nano chips during inference. Demonstrated extraction of CNN weights from TensorRT workloads despite the highly parallel, noisy GPU environment. Targets edge/embedded AI devices where adversaries may have physical proximity. (3) **LeftoverLocals (CVE-2023-4969):** Memory-based side channel leaking LLM responses (see 4.7.2 notes). These attacks underscore the need for detective controls. Key telemetry metrics: SM utilization patterns, memory bandwidth utilization, PCIe transfer patterns, ECC error rates (may indicate fault injection attempts), and anomalous power draw patterns. Note: H100 CC-On mode disables performance counters, creating a trade-off between security and monitoring visibility. Organizations must evaluate whether CC mode's protections outweigh the loss of telemetry-based detection. |

## Implementation Guidance

### Confidential Computing Evolution (2024-2026)

NVIDIA Confidential Computing has progressed through three generations, each expanding the hardware trust boundary:

| Generation | Platform | Key Capabilities | Limitations |
|-----------|----------|-----------------|-------------|
| **Gen 1** | H100 Hopper (2023-2024) | On-die hardware RoT, TEE with encrypted PCIe via bounce buffers, GPU attestation SDK, CC-On/CC-Off/CC-DevTools modes | ~4 GB/s PCIe encryption throughput (CPU-limited), NVLink unencrypted in non-CC mode, performance counters disabled in CC-On |
| **Gen 2** | B200/GB200 Blackwell (2025) | TDISP/IDE inline encryption (no bounce buffers), encrypted NVLink, GPU memory encryption in CC mode, dual attestation (Intel TDX + NVIDIA), near-native performance | Requires compatible host CPUs (AMD SEV-SNP or Intel TDX) |
| **Gen 3** | Vera Rubin NVL72 (H2 2026) | Rack-scale confidential computing across 72 GPUs + 36 CPUs + full NVLink fabric, unified TEE across entire rack, attestation services with cryptographic compliance proof | Not yet shipping; availability expected H2 2026 |

**Practical deployment note:** CUDA applications should run without code changes in CC mode -- encryption is handled transparently. The primary overhead on H100 is CPU-GPU data transfer bandwidth; applications with high compute-to-data ratios see minimal impact. Blackwell eliminates this bottleneck with inline encryption.

### GPU Side-Channel Attack Landscape (2023-2025)

The threat landscape for GPU-specific side channels has expanded significantly:

**Software-exploitable (remote/co-tenant):**
- **LeftoverLocals (CVE-2023-4969):** Co-resident attacker reads uncleared GPU local memory to reconstruct LLM responses. Affected: Apple, AMD, Qualcomm, Imagination GPUs. NVIDIA GPUs not affected. Requires only user-level GPU compute access.
- **GPU.zip (IEEE S&P 2024):** Browser-based pixel-stealing attack exploiting GPU data compression. Affects all major GPU vendors. Currently unpatched across the industry. Exploitable via malicious web pages in Chrome.

**Physical proximity (edge/embedded):**
- **BarraCUDA (USENIX Security 2025):** Electromagnetic side-channel extraction of neural network weights from NVIDIA Jetson devices running TensorRT inference. Demonstrates that edge AI deployments face IP theft risks from physical access.
- **Power/EM analysis research (ongoing):** Multiple academic groups have demonstrated DNN architecture extraction from power and electromagnetic traces, though practical exploitation requires physical proximity or co-tenancy.

**Mitigation hierarchy:** (1) Use NVIDIA Confidential Computing where available to encrypt memory and interconnects at the hardware level. (2) Enable MIG for multi-tenant isolation. (3) Enforce memory sanitization between workloads. (4) Deploy telemetry monitoring for anomaly detection. (5) For edge devices, apply physical security controls and consider model obfuscation.

### AI Chip Supply Chain Security

As AI accelerators become critical infrastructure, supply chain integrity is an emerging concern:

- **Hardware trojans:** Malicious modifications embedded during chip design or fabrication can exfiltrate data or weaken security. AI-driven detection methods using LLMs to scan chip designs are being researched (University of Missouri, 2025).
- **Hardware-Enabled Mechanisms (HEMs):** CNAS proposes embedding governance features directly into AI chips -- location verification, compute metering, offline licensing, and bandwidth controls -- to enable post-export enforcement without remote kill switches.
- **Domestic manufacturing:** CHIPS Act mega-fabs beginning high-volume production in 2026 aim to reduce dependency on overseas fabrication, but supply chain verification from design to deployment remains critical.
- **Standards development:** NIST is recommended to develop AI hardware security standards extending FIPS 140-3, in coordination with the Open Compute Project and semiconductor industry consortia.

## Open Research Questions

- [ ] How effective are GPU power/EM side-channel attacks against production AI workloads in data center environments (as opposed to edge devices), and what countermeasures beyond physical isolation are practical at scale?
- [ ] What is the real-world performance overhead of NVIDIA Blackwell Confidential Computing with TDISP/IDE inline encryption across different model architectures, batch sizes, and multi-GPU training workloads?
- [ ] Will the Vera Rubin NVL72 rack-scale confidential computing model become the baseline expectation for multi-tenant AI infrastructure, and how will competing platforms (AMD, Intel) respond?
- [ ] How should GPU.zip-class browser-based side channels be mitigated -- at the GPU hardware level (disabling compression), browser level (restricting GPU access), or application level (preventing cross-origin iframe rendering)?
- [ ] Can Hardware-Enabled Mechanisms (HEMs) for AI chip governance be designed to resist sophisticated adversaries with physical chip access while avoiding the civil liberties concerns of government backdoors?
- [ ] How will on-chip export control mechanisms (compute metering, location verification, offline licensing) interact with AI hardware security requirements, and could they introduce new attack surfaces?
- [ ] What monitoring strategies can compensate for the loss of performance counter visibility when GPUs operate in CC-On mode, where telemetry is intentionally restricted?
- [ ] As Google TPU (Trillium/Ironwood) and custom NPUs proliferate, what attestation and confidential computing capabilities will emerge for non-NVIDIA accelerators?
