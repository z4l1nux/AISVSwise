# C4.7 AI Hardware Security

[Back to C04 Index](C04-Infrastructure.md)

## Purpose

Secure AI-specific hardware components including GPUs, TPUs, and specialized AI accelerators. AI hardware presents unique security challenges absent from traditional computing: GPU memory is shared across workloads by default, accelerator interconnects carry sensitive model data at high bandwidth, firmware vulnerabilities can compromise entire GPU clusters, and side-channel attacks on accelerators can leak model weights or training data. This section addresses hardware-level security controls specific to AI accelerator infrastructure.

On an AMD Radeon RX 7900 XT, the vulnerability leaks ~5.5 MB per GPU invocation -- for a 7B parameter LLM on llama.cpp, this totals ~181 MB per query, enough to reconstruct full LLM responses. Notably, NVIDIA GPUs were found not vulnerable to LeftoverLocals. By exploiting GPU parallelism with multi-warp CUDA hammering patterns, they induced 8 single-bit flips across DRAM banks with a minimum activation count of ~12K. A single bit flip targeting FP16 weight exponents dropped ImageNet DNN model accuracy from 80% to 0.1% across all five tested models. The attack requires only user-level CUDA access, making it viable in shared cloud GPU environments. NVIDIA recommends enabling ECC (`nvidia-smi -e 1`) as mitigation, though this incurs up to 10% performance reduction and 6.25% memory capacity loss. Newer architectures (H100 HBM3, RTX 5090 GDDR7) feature on-die ECC that mitigates single-bit flips, but multi-bit Rowhammer patterns may bypass ECC in the future. |

On H100/Blackwell, the secure boot sequence establishes a chain of trust via SPDM sessions and cryptographically signed attestation reports. Third-party accelerators and FPGAs may have weaker firmware signing. Driver versions should also be pinned -- NVIDIA driver updates have historically introduced both security fixes and new vulnerabilities. **Supply chain risks (2024-2026):** CNAS research highlights that hardware trojans (malicious modifications to chip designs) can steal data, weaken security, and sabotage systems. University of Missouri researchers (2025) introduced AI-driven methods using LLMs to scan chip designs for hidden threats. The CHIPS Act is driving domestic semiconductor manufacturing ramps in 2026, but supply chain integrity remains a concern for imported accelerators. CNAS recommends NIST develop AI hardware security standards building on FIPS 140-3, coordinating with industry consortia like the Open Compute Project. |

Sensitive data (PII in training sets, proprietary model parameters) persisting in accelerator memory. | Verify GPU reset occurs between workloads (check for `nvidia-smi -r` or equivalent). Validate that HBM (High Bandwidth Memory) on-package is included in sanitization. | This overlaps with 4.7.2 but focuses specifically on memory sanitization rather than isolation during concurrent execution. HBM3 on H100/A100 has large capacity (80GB+) -- zeroing takes measurable time. Consider async zeroing or NVIDIA's compute sanitizer tool for validation. **Confidential computing memory protection:** On H100 in CC-On mode, on-package HBM3 memory is considered secure against standard physical attack tools and remains unencrypted for performance, while all data crossing the PCIe bus is encrypted through bounce buffers. On Blackwell, all data in GPU memory is encrypted in CC mode, including model weights, training data, and inference results. This provides defense-in-depth against data remanence -- even if memory is not zeroed, contents are encrypted with session-specific keys. |

This affects all Container Toolkit versions through 1.17.7 and GPU Operator through 25.3.0. This provides hardware-enforced isolation beyond MIG partitioning. For multi-GPU workloads, NVLink pathways are also encrypted on Blackwell, preventing cross-partition snooping at the interconnect level. |

These mechanisms are designed to avoid backdoor pitfalls (lessons from the Clipper chip) while enabling hardware-rooted trust. Organizations using HSMs for AI model key protection should verify their modules are FIPS 140-3 validated or plan migration. **Marvell LiquidSecurity 2 (2025):** Azure now offers single-tenant HSM services built on Marvell's FIPS 140-3 Level 3 certified LiquidSecurity platform, supporting up to 1 million key objects and 120,000 simultaneous ECC operations per PCIe card -- performance levels more suitable for high-frequency AI model key operations than traditional HSMs. |

Man-in-the-middle attacks on distributed training traffic. Unauthorized nodes joining training clusters via unsecured interconnects. | Verify InfiniBand subnet management restricts partition membership. Confirm PCIe ACS (Access Control Services) is enabled to prevent peer-to-peer DMA attacks. Check NVLink topology matches approved configuration. Verify NCCL communication uses encrypted channels where supported. | **Generational progress on interconnect encryption (2024-2026):** On H100 in CC mode, all command buffers and CUDA kernels are encrypted and signed before crossing the PCIe bus, using software-based bounce buffers at ~4 GB/s throughput (limited by CPU encryption performance). NVLink on H100 does not natively encrypt in non-CC mode. **Blackwell (B200/GB200, 2025):** Introduced TDISP and IDE protocols enabling direct inline encryption between GPU and Confidential VM, eliminating bounce buffer latency. NVLink pathways are encrypted in CC mode, securing GPU-to-GPU data movement at scale. NVIDIA claims near-identical throughput performance compared to unencrypted modes. **Vera Rubin NVL72 (announced CES 2026, shipping H2 2026):** Third-generation NVIDIA Confidential Computing expands to full rack-scale -- encrypting every bus across 72 Rubin GPUs, 36 Vera CPUs, and the entire NVLink fabric as a unified trusted execution environment. This represents the first rack-scale confidential computing platform with end-to-end encrypted interconnects. InfiniBand supports partition keys (PKeys) for access control but not encryption natively. RDMA traffic is typically unencrypted -- consider IPsec or MACsec for network-level encryption of RoCE traffic. |
| **4.7.8** | Verify that accelerator telemetry (power draw, temperature, error correction, performance counters) is exported to centralized security monitoring and alerts on anomalies indicative of side-channels or covert channels. | 3 | D | Side-channel attacks using power analysis or electromagnetic emanation to extract model weights or training data. Covert channel communication via GPU utilization patterns. Cryptomining detection. Hardware fault exploitation. | Verify NVIDIA DCGM (Data Center GPU Manager) or equivalent is deployed and exporting metrics. Validate alert routing and response procedures. | **Demonstrated GPU side-channel attacks (2023-2025):** (1) **GPU.zip (IEEE S&P 2024):** Exploits software-transparent graphical data compression present in nearly all modern GPUs (AMD, Apple, Arm, Intel, NVIDIA, Qualcomm) to steal pixel data via rendering time differences. In Chrome, can extract usernames from cross-origin iframes in 30 minutes (Ryzen) to 215 minutes (Intel) at 97-98% accuracy. As of 2024, no GPU vendor or Google has patched this. (2) **BarraCUDA (USENIX Security 2025):** Uses correlation electromagnetic analysis (CEMA) to extract neural network weights and biases from NVIDIA Jetson Nano and Orin Nano chips during inference. Demonstrated extraction of CNN weights from TensorRT workloads despite the highly parallel, noisy GPU environment. Targets edge/embedded AI devices where adversaries may have physical proximity. Key telemetry metrics: SM utilization patterns, memory bandwidth utilization, PCIe transfer patterns, ECC error rates (may indicate fault injection attempts), and anomalous power draw patterns. Note: H100 CC-On mode disables performance counters, creating a trade-off between security and monitoring visibility. Organizations must evaluate whether CC mode's protections outweigh the loss of telemetry-based detection. **(4) GPUHammer (USENIX Security 2025):** Rowhammer-induced bit flips on GDDR6 GPU memory can degrade DNN model accuracy catastrophically. Elevated ECC correctable error rates on GDDR6 GPUs may indicate active Rowhammer exploitation attempts -- monitoring `ecc_errors.volatile.corrected` via DCGM should be a standard alerting rule. |

## Implementation Guidance

### Confidential Computing Evolution (2024-2026)

NVIDIA Confidential Computing has progressed through three generations, each expanding the hardware trust boundary:

| Generation | Platform | Key Capabilities | Limitations |
|-----------|----------|-----------------|-------------|
| **Gen 1** | H100 Hopper (2023-2024) | On-die hardware RoT, TEE with encrypted PCIe via bounce buffers, GPU attestation SDK, CC-On/CC-Off/CC-DevTools modes | ~4 GB/s PCIe encryption throughput (CPU-limited), NVLink unencrypted in non-CC mode, performance counters disabled in CC-On |
| **Gen 2** | B200/GB200 Blackwell (2025) | TDISP/IDE inline encryption (no bounce buffers), encrypted NVLink, GPU memory encryption in CC mode, dual attestation (Intel TDX + NVIDIA), near-native performance | Requires compatible host CPUs (AMD SEV-SNP or Intel TDX) |
| **Gen 3** | Vera Rubin NVL72 (H2 2026) | Rack-scale confidential computing across 72 GPUs + 36 CPUs + full NVLink fabric, unified TEE across entire rack, NRAS integration verifying CPUs/GPUs/NICs/firmware/drivers/workloads against known-good measurements | In production at NVIDIA; cloud deployment (AWS, GCP, Azure, OCI, CoreWeave, Lambda) expected H2 2026 |

**Practical deployment note:** CUDA applications should run without code changes in CC mode -- encryption is handled transparently. The primary overhead on H100 is CPU-GPU data transfer bandwidth; applications with high compute-to-data ratios see minimal impact. Blackwell eliminates this bottleneck with inline encryption.

**Critical caveat -- TEE.Fail (October 2025):** All three generations of GPU confidential computing depend on CPU-side TEE attestation (Intel TDX or AMD SEV-SNP) as their trust anchor. The TEE.Fail attack demonstrated that a sub-$1,000 DDR5 memory-bus interposer can extract attestation keys from these CPU TEEs, enabling forged attestation quotes that pass verification. This means GPU CC protections can be bypassed by an adversary with physical access to the server's memory bus. Organizations relying on GPU confidential computing for high-value workloads should assess whether their threat model includes physical-access adversaries and consider complementary controls (physical security, tamper-evident server chassis, restricted datacenter access).

### GPU Side-Channel and Hardware Attack Landscape (2023-2026)

The threat landscape for GPU-specific attacks has expanded significantly, spanning software-exploitable side channels, physical fault injection, and TEE bypass attacks:

**Software-exploitable (remote/co-tenant):**
- **LeftoverLocals (CVE-2023-4969):** Co-resident attacker reads uncleared GPU local memory to reconstruct LLM responses. Affected: Apple, AMD, Qualcomm, Imagination GPUs. NVIDIA GPUs not affected. Requires only user-level GPU compute access.
- **GPU.zip (IEEE S&P 2024):** Browser-based pixel-stealing attack exploiting GPU data compression. Affects all major GPU vendors. Currently unpatched across the industry. Exploitable via malicious web pages in Chrome.
- **GPUHammer (USENIX Security 2025):** First demonstrated Rowhammer attack on GPU GDDR6 memory. Induces bit flips via multi-warp parallel CUDA hammering on NVIDIA RTX A6000, degrading DNN model accuracy from 80% to 0.1% with a single bit flip. Requires only user-level CUDA access, making it viable in shared cloud environments. ECC mitigates single-bit flips but incurs ~10% performance overhead. Newer HBM3 and GDDR7 architectures have on-die ECC that should resist current attack patterns.

**Physical proximity (edge/embedded):**
- **BarraCUDA (USENIX Security 2025):** Electromagnetic side-channel extraction of neural network weights from NVIDIA Jetson devices running TensorRT inference. Demonstrates that edge AI deployments face IP theft risks from physical access.
- **Power/EM analysis research (ongoing):** Multiple academic groups have demonstrated DNN architecture extraction from power and electromagnetic traces, though practical exploitation requires physical proximity or co-tenancy.

**TEE and attestation bypass (physical access):**
- **TEE.Fail (October 2025):** Georgia Tech, Purdue, and Synkhronix researchers demonstrated a DDR5 memory-bus interposition attack using under $1,000 in off-the-shelf components to extract attestation keys from Intel TDX and AMD SEV-SNP. Because NVIDIA GPU Confidential Computing depends on CPU CVM attestation as its trust anchor, forged attestation quotes can undermine the entire GPU TEE chain. Intel and AMD consider physical memory-bus attacks out of scope for their threat models. This is a fundamental limitation: hardware confidential computing currently assumes the memory bus is trusted.

**Container escape:**
- **NVIDIAScape (CVE-2025-23266, CVSS 9.0, July 2025):** Critical container escape in NVIDIA Container Toolkit allowing full host root access via a three-line Dockerfile exploiting `LD_PRELOAD` in the `nvidia-ctk` OCI hook. Fixed in Container Toolkit 1.17.8 / GPU Operator 25.3.1.

**Mitigation hierarchy:** (1) Use NVIDIA Confidential Computing where available to encrypt memory and interconnects at the hardware level. (2) Enable MIG for multi-tenant isolation. (3) Enforce memory sanitization between workloads. (4) Enable ECC on GDDR6 GPUs in shared environments. (5) Deploy telemetry monitoring for anomaly detection. (6) Keep NVIDIA Container Toolkit and GPU Operator patched to latest versions. (7) For edge devices, apply physical security controls and consider model obfuscation.

### AI Chip Supply Chain Security

As AI accelerators become critical infrastructure, supply chain integrity is an emerging concern:

- **Hardware trojans:** Malicious modifications embedded during chip design or fabrication can exfiltrate data or weaken security. AI-driven detection methods using LLMs to scan chip designs are being researched (University of Missouri, 2025).
- **Hardware-Enabled Mechanisms (HEMs):** CNAS proposes embedding governance features directly into AI chips -- location verification, compute metering, offline licensing, and bandwidth controls -- to enable post-export enforcement without remote kill switches.
- **Domestic manufacturing:** CHIPS Act mega-fabs beginning high-volume production in 2026 aim to reduce dependency on overseas fabrication, but supply chain verification from design to deployment remains critical.
- **Standards development:** NIST is recommended to develop AI hardware security standards extending FIPS 140-3, in coordination with the Open Compute Project and semiconductor industry consortia.

## Open Research Questions

- [ ] How effective are GPU power/EM side-channel attacks against production AI workloads in data center environments (as opposed to edge devices), and what countermeasures beyond physical isolation are practical at scale?
- [ ] What is the real-world performance overhead of NVIDIA Blackwell Confidential Computing with TDISP/IDE inline encryption across different model architectures, batch sizes, and multi-GPU training workloads?
- [ ] Will the Vera Rubin NVL72 rack-scale confidential computing model become the baseline expectation for multi-tenant AI infrastructure, and how will competing platforms (AMD, Intel) respond?
- [ ] How should GPU.zip-class browser-based side channels be mitigated -- at the GPU hardware level (disabling compression), browser level (restricting GPU access), or application level (preventing cross-origin iframe rendering)?
- [ ] Can Hardware-Enabled Mechanisms (HEMs) for AI chip governance be designed to resist sophisticated adversaries with physical chip access while avoiding the civil liberties concerns of government backdoors?
- [ ] How will on-chip export control mechanisms (compute metering, location verification, offline licensing) interact with AI hardware security requirements, and could they introduce new attack surfaces?
- [ ] What monitoring strategies can compensate for the loss of performance counter visibility when GPUs operate in CC-On mode, where telemetry is intentionally restricted?
- [ ] As Google TPU (Trillium/Ironwood) and custom NPUs proliferate, what attestation and confidential computing capabilities will emerge for non-NVIDIA accelerators?
- [ ] Given TEE.Fail's demonstration that DDR5 memory-bus interposition can forge attestation quotes, what architectural changes (e.g., integrity trees, non-deterministic encryption modes) would be needed to harden CPU TEEs against physical-access adversaries, and how would this affect GPU CC trust chains?
- [ ] Will GPUHammer-class Rowhammer attacks evolve to produce multi-bit flips that bypass on-die ECC in newer GPU architectures (HBM3, GDDR7), and should cloud providers mandate ECC enablement for all shared GPU workloads despite the performance penalty?
- [ ] How should organizations balance the systemic risk of NVIDIA Container Toolkit vulnerabilities (CVE-2025-23266) against the operational dependency on containerized GPU workloads, and are micro-VM or unikernel isolation approaches mature enough for production AI workloads?
