# C4.7 AI Hardware Security

[Back to C04 Index](C04-Infrastructure.md)

## Purpose

Secure AI-specific hardware components including GPUs, TPUs, and specialized AI accelerators. AI hardware presents unique security challenges absent from traditional computing: GPU memory is shared across workloads by default, accelerator interconnects carry sensitive model data at high bandwidth, firmware vulnerabilities can compromise entire GPU clusters, and side-channel attacks on accelerators can leak model weights or training data. This section addresses hardware-level security controls specific to AI accelerator infrastructure.

---

## Requirements

| # | Requirement | Level | Role | Threat Mitigated | Verification Approach | Gaps / Notes |
|---|-------------|:-----:|:----:|-----------------|----------------------|--------------|
| **4.7.1** | Verify that before workload execution, AI accelerator integrity is validated using hardware-based attestation mechanisms (e.g., TPM, DRTM, or equivalent). | 2 | D/V | Compromised GPU firmware or driver that intercepts model weights, training data, or inference inputs. Rootkit-level persistence in accelerator firmware surviving OS reinstalls. | Check for TPM-based measured boot that includes GPU firmware measurements in the boot chain. Verify DRTM (Dynamic Root of Trust for Measurement) is supported and enabled. Validate attestation reports include GPU firmware version and configuration hashes. | GPU attestation is an emerging capability. NVIDIA Hopper (H100) supports GPU attestation via the NVIDIA Attestation SDK. Older GPUs (A100, V100) do not support hardware attestation natively. TPM-based platform attestation can cover the host but not GPU-internal firmware state. AMD and Intel accelerators have varying attestation support. |
| **4.7.2** | Verify that accelerator (GPU) memory is isolated between workloads through partitioning mechanisms with memory sanitization between jobs. | 2 | D/V | Cross-workload data leakage through GPU memory. Previous tenant's model weights, training data, or inference inputs remaining in GPU memory and being read by the next workload. | Verify GPU memory partitioning is active (NVIDIA MIG, AMD memory partitioning). Confirm memory sanitization occurs between job switches. Test by writing known patterns to GPU memory, terminating the workload, and checking if the pattern persists for the next workload. | Without explicit sanitization, GPU memory retains data from previous workloads. `cudaFree()` does not zero memory by default. NVIDIA driver version 525+ supports `RmProfilingAdminOnly` to prevent memory snooping. CUDA 12.0+ supports `cudaMemPoolSetAttribute` for secure memory pools. Multi-tenant GPU environments (cloud providers) must enforce memory clearing between tenants at the driver/hypervisor level. |
| **4.7.3** | Verify that AI accelerator firmware is version-pinned, signed, and attested at boot; unsigned or debug firmware is blocked. | 2 | D/V | Supply chain attack via compromised firmware update. Downgrade attack to firmware version with known vulnerabilities. Debug firmware left in production enabling memory inspection or bypassing security features. | Verify firmware signing is enforced (check GPU driver/firmware update policies). Confirm firmware version matches the approved version in configuration management. Check that debug or unsigned firmware is rejected at boot. Audit firmware update procedures for approval workflow. | NVIDIA GPU firmware (VBIOS) is signed by NVIDIA and verified at boot. However, third-party accelerators and FPGAs may have weaker firmware signing. Driver versions should also be pinned -- NVIDIA driver updates have historically introduced both security fixes and new vulnerabilities. Coordinate firmware/driver updates with vendor security advisories. |
| **4.7.4** | Verify that VRAM and on-package memory are zeroed between jobs/tenants and that device reset policies prevent cross-tenant data remanence. | 2 | D/V | Data remanence in GPU VRAM allowing next tenant to read previous tenant's model weights, activations, or training data. Sensitive data (PII in training sets, proprietary model parameters) persisting in accelerator memory. | Verify GPU reset occurs between workloads (check for `nvidia-smi -r` or equivalent). Confirm VRAM is zeroed (write test patterns, reset, read back). Check device reset policy in the container runtime or hypervisor. Validate that HBM (High Bandwidth Memory) on-package is included in sanitization. | This overlaps with 4.7.2 but focuses specifically on memory sanitization rather than isolation during concurrent execution. Full GPU reset (`nvidia-smi --gpu-reset`) is more thorough than memory deallocation but causes a brief service interruption. HBM3 on H100/A100 has large capacity (80GB+) -- zeroing takes measurable time. Consider async zeroing or NVIDIA's compute sanitizer tool for validation. |
| **4.7.5** | Verify that partitioning/isolation features (e.g., MIG/VM partitioning) are enforced per tenant and prevent peer-to-peer memory access across partitions. | 2 | D/V | Cross-tenant memory access via CUDA IPC, peer-to-peer DMA, or NVLink. One tenant reading another tenant's GPU memory through shared address spaces. | Verify MIG (Multi-Instance GPU) is enabled and configured per tenant. Confirm CUDA IPC is disabled across partition boundaries. Test peer-to-peer memory access between partitions and confirm it fails. Check that NVLink is restricted to authorized partition pairs only. | NVIDIA MIG (available on A100, H100) creates up to 7 isolated GPU instances with separate memory, cache, and compute resources. Without MIG, CUDA applications on the same GPU can potentially access each other's memory via `cudaIpcGetMemHandle`. MIG partitions have their own error isolation -- a fault in one partition does not affect others. Limitation: MIG reduces flexibility in GPU memory allocation. |
| **4.7.6** | Verify that hardware security modules (HSMs) or equivalent tamper-resistant hardware protect AI model weights and cryptographic keys, with certification to an appropriate assurance level (e.g., FIPS 140-3 Level 3 or Common Criteria EAL4+). | 3 | D/V | Physical extraction of model weights or encryption keys from storage media. Key compromise from software-based key storage. Regulatory non-compliance for AI systems handling sensitive data (financial, healthcare, government). | Verify HSM deployment (on-premise HSMs or cloud HSM services). Check certification level (FIPS 140-3 Level 3 requires physical tamper resistance). Confirm model encryption keys are generated and stored within the HSM boundary. Validate key usage logging and access controls. | FIPS 140-3 Level 3 HSMs provide physical tamper evidence and response mechanisms. Cloud HSM services (AWS CloudHSM, Azure Dedicated HSM, GCP Cloud HSM) offer managed FIPS-certified HSMs. Key consideration for AI: model weight encryption keys may need to be used at high frequency during model loading -- HSM performance (operations per second) must be validated against workload requirements. Consider envelope encryption where the HSM holds master keys and data keys are derived per model. |
| **4.7.7** | Verify that accelerator interconnects (NVLink/PCIe/InfiniBand/RDMA/NCCL) are restricted to approved topologies and authenticated endpoints; plaintext cross-tenant links are disallowed. | 3 | D/V | Eavesdropping on inter-GPU communication carrying model gradients, activations, or weights. Man-in-the-middle attacks on distributed training traffic. Unauthorized nodes joining training clusters via unsecured interconnects. | Verify InfiniBand subnet management restricts partition membership. Confirm PCIe ACS (Access Control Services) is enabled to prevent peer-to-peer DMA attacks. Check NVLink topology matches approved configuration. Verify NCCL communication uses encrypted channels where supported. | NVLink and NVSwitch do not natively support encryption -- encryption must be applied at the software layer (NCCL with TLS) or by using confidential computing (NVIDIA CC encrypts PCIe/NVLink traffic on H100). InfiniBand supports partition keys (PKeys) for access control but not encryption natively. RDMA traffic is typically unencrypted -- consider IPsec or MACsec for network-level encryption of RoCE traffic. This is a significant gap for multi-tenant GPU clusters. |
| **4.7.8** | Verify that accelerator telemetry (power draw, temperature, error correction, performance counters) is exported to centralized security monitoring and alerts on anomalies indicative of side-channels or covert channels. | 3 | D | Side-channel attacks using power analysis or electromagnetic emanation to extract model weights or training data. Covert channel communication via GPU utilization patterns. Cryptomining detection. Hardware fault exploitation. | Verify NVIDIA DCGM (Data Center GPU Manager) or equivalent is deployed and exporting metrics. Check that alerting rules exist for: unusual power consumption patterns, abnormal temperature profiles, excessive ECC errors, unexpected performance counter values. Validate alert routing and response procedures. | GPU side-channel research has demonstrated extraction of DNN architectures from power/EM traces and cache timing. Practical exploitation requires physical proximity or co-tenancy. Telemetry monitoring serves as a detective control -- it cannot prevent side-channels but can detect anomalous patterns. Key metrics: SM utilization patterns, memory bandwidth utilization, PCIe transfer patterns, and ECC error rates (may indicate fault injection attempts). |
