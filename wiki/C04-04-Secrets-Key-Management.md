# C4.4 Secrets & Cryptographic Key Management

[Back to C04 Index](C04-Infrastructure.md)

## Purpose

Protect secrets and cryptographic keys with secure storage, automated rotation, and strong access controls. AI systems manage a wide variety of sensitive credentials: API keys for model providers, database credentials for training data stores, signing keys for model artifacts, encryption keys for model weights, and service account tokens for cloud resources. Compromise of any of these can lead to data theft, model tampering, or unauthorized access to expensive GPU infrastructure.

The scale of this problem is growing rapidly. GitGuardian's 2026 State of Secrets Sprawl report found 28.65 million new secrets on public GitHub in 2025 alone (a 34% year-over-year increase), with AI service secrets reaching 1,275,105 leaked credentials -- an 81% surge. LLM infrastructure components (orchestration, RAG, and vector storage) leaked credentials 5x faster than core model provider services. Additionally, 64% of secrets validated as active in 2022 remained valid through January 2026, demonstrating persistent failure in rotation and remediation.

## Requirements

| # | Requirement | Level | Role | Threat Mitigated | Verification Approach | Gaps / Notes |
|---|-------------|:-----:|:----:|-----------------|----------------------|--------------|
| **4.4.1** | Verify that secrets are stored in a dedicated secrets management system with encryption at rest and isolated from application workloads. | 1 | D/V | Secrets exposed through container image layers, configuration files, or application logs. Compromise of one workload leading to access to all secrets. GitGuardian (2026) identified 113,000 leaked DeepSeek API keys and 24,008 unique secrets in MCP configuration files alone, often caused by documentation that normalizes hardcoded credential patterns. | Confirm use of a secrets manager (HashiCorp Vault, AWS Secrets Manager, Azure Key Vault, GCP Secret Manager, Infisical). Verify encryption at rest is enabled (check KMS key configuration). Confirm the secrets store runs in a separate network segment from AI workloads. For MCP-based agents, verify that MCP server configs do not contain inline API keys. Scan for secrets in Jupyter notebooks, `.env` files, and `mcp.json` configs using GitGuardian, detect-secrets, or TruffleHog. | ML notebooks (Jupyter, Colab) are a common source of leaked secrets -- developers embed API keys directly in notebook cells. AI coding assistants amplify the problem: Claude Code-assisted commits showed a 3.2% secret-leak rate vs. a 1.5% baseline (GitGuardian 2026). MCP configuration files are an emerging leak vector -- 8.8% of discovered MCP secrets validated as active credentials. Consider pre-commit hooks (detect-secrets, TruffleHog, GitGuardian ggshield) to catch secrets before they reach version control. |
| **4.4.2** | Verify that access to production secrets requires strong authentication. | 1 | D/V | Unauthorized access to production model serving keys, training data encryption keys, or cloud provider credentials. Insider threat from developers with overly broad access. In May 2025, an xAI employee leaked a private API key on GitHub that provided access to 60+ fine-tuned and private LLMs containing internal data from SpaceX, Tesla, and Twitter/X (Krebs on Security). | Verify authentication to the secrets manager uses MFA or certificate-based auth for human access. Confirm workload identity (IRSA, Workload Identity, pod identity) is used for service access rather than long-lived credentials. Audit access logs for anomalous patterns. For agentic AI systems, verify that agent credentials are dynamically issued, task-scoped, context-bound, and ephemeral (seconds to minutes). | AI workloads often run as shared service accounts with broad secrets access. Use fine-grained policies: inference services should not access training data encryption keys. Vault policies or AWS IAM conditions should scope access by workload identity, not just by role. Agentic AI systems break traditional vault assumptions: agents change operational context in real-time, making pre-assigned static credentials dangerous. Identity-first access systems (e.g., Aembit, workload identity federation) that make real-time access decisions based on context are emerging as alternatives to vault-centric approaches. |
| **4.4.3** | Verify that secrets are deployed to applications at runtime through a dedicated secrets management system. Secrets must never be embedded in source code, configuration files, build artifacts, container images, or environment variables. | 1 | D/V | Secrets leaking through container image layers (recoverable via `docker history`), git history, CI/CD logs, or process environment dumps (`/proc/*/environ`). Analysis of 38,630 AI-enabled Android apps found 72% contained at least one hardcoded secret, averaging 5.1 secrets per app. Researchers found nearly 12,000 valid live secrets (API keys for AWS, Slack, GitHub, etc.) in the Common Crawl December 2024 dataset, which is used to train LLMs -- meaning models may reproduce leaked credentials in generated code. | Scan container images for embedded secrets (Trivy, Syft). Search git history for secrets patterns. Verify secrets are injected via volume mounts (Vault Agent, CSI driver) or API calls, not environment variables. Check CI/CD logs for secret masking. For mobile/edge AI apps, scan APKs/IPAs for hardcoded credentials using tools like MobSF or Ostorlab. | Environment variables are explicitly excluded because they appear in process listings, crash dumps, and debug logs. The Kubernetes Secrets CSI driver or Vault Agent sidecar are preferred injection mechanisms. Note: many ML frameworks read API keys from environment variables by default (e.g., `OPENAI_API_KEY`, `HF_TOKEN`) -- wrappers may be needed to read from mounted secret files instead. Google API keys originally deployed for public services (e.g., Maps) now also authenticate to Gemini, creating unintended exposure -- nearly 3,000 such keys were found on public websites (Truffle Security 2025). The External Secrets Operator for Kubernetes is becoming standard for bridging secrets managers to production clusters. |
| **4.4.4** | Verify that cryptographic keys are generated and stored in hardware-backed modules (e.g., HSMs, cloud KMS). | 2 | D/V | Key extraction from software keystores via memory dumps, cold boot attacks, or host compromise. Weak key generation from poor entropy sources. Future quantum computing threats to current cryptographic algorithms protecting model weights and AI infrastructure. | Verify KMS or HSM is used for key generation (not `openssl genrsa` on a dev machine). Check that keys are non-exportable from HSMs. Confirm cloud KMS keys use HSM-backed key types (AWS KMS with CloudHSM, Azure Managed HSM, GCP Cloud HSM). Begin inventorying cryptographic algorithms in use to prepare for post-quantum migration per NIST IR 8547 timelines. | Applicable to model signing keys, data encryption keys, and TLS certificate private keys. For AI-specific use cases: model weight encryption keys and inference API authentication keys should use HSM-backed storage. Cloud KMS services provide a practical middle ground between software keystores and dedicated HSMs. Post-quantum readiness: NIST finalized ML-KEM, ML-DSA, and SLH-DSA in August 2024, with HQC selected for standardization in March 2025 (expected finalized 2026-2027). NIST IR 8547 sets a deprecation timeline for quantum-vulnerable algorithms by 2035, but organizations should begin crypto-agility planning now. AI model signing and weight encryption keys are long-lived assets that may be vulnerable to "harvest now, decrypt later" attacks. |
| **4.4.5** | Verify that secrets rotation is automated. | 2 | D/V | Long-lived credentials remaining valid after team member departure, credential leak, or periodic compromise. Difficulty determining blast radius of a compromised credential when rotation history is unclear. GitGuardian found 64% of leaked secrets from 2022 were still valid in January 2026, underscoring that rotation and remediation are widely neglected. | Verify automated rotation is configured (Vault dynamic secrets, AWS Secrets Manager rotation lambdas, Infisical automatic rotation). Check rotation frequency is appropriate (database credentials: 30-90 days; API keys: 90 days; certificates: < 24 hours for short-lived). Confirm rotation does not cause downtime. For agentic AI, verify that agents use ephemeral, dynamically generated credentials rather than long-lived static secrets. | Dynamic secrets (Vault, AWS STS) are preferred over static secrets with rotation. For ML-specific credentials: model provider API keys (OpenAI, Anthropic, DeepSeek) may not support automated rotation via API -- manual rotation workflows need documentation. GPU cloud provider credentials (Lambda Labs, CoreWeave) may have limited rotation API support. For agentic AI workloads, the hybrid approach is gaining traction in 2026: a secrets backend (Vault or AWS Secrets Manager) combined with External Secrets Operator for Kubernetes and tools like Doppler for CI/CD injection. Traditional vault-centric rotation breaks down when AI agents assemble and execute pipelines dynamically -- short-lived credentials tied to agent identity and task context are essential. |

## Incident Case Studies

| Date | Incident | Impact | Lesson |
|------|----------|--------|--------|
| May 2025 | xAI employee leaked a private API key on GitHub, undetected for two months | Access to 60+ private fine-tuned LLMs with internal data from SpaceX, Tesla, and Twitter/X | Pre-commit secret scanning and real-time GitHub monitoring are essential. Even advanced AI companies are vulnerable to basic credential hygiene failures. |
| Feb 2025 | Truffle Security found ~12,000 valid live secrets in Common Crawl (Dec 2024 dataset) | AWS, Slack, GitHub, Mailchimp keys embedded in public web pages and ingested into LLM training data | LLMs trained on public data can reproduce valid secrets in generated code. Training data must be scanned and sanitized for credentials. |
| 2025 | Google API keys for Maps found to also authenticate to Gemini | ~3,000 publicly exposed keys gained unintended access to Gemini LLM, uploaded files, and cached data | API key scope can silently expand as providers add new services. Regularly audit what each credential can access. |
| 2025 | LangSmith proxy injection via LangChain Hub prompts | OpenAI API keys, prompt data, and attachments silently exfiltrated through attacker-controlled proxy | AI tool chains introduce novel exfiltration vectors. Validate proxy and network configurations in LLM orchestration platforms. |
| Dec 2025 | CVE-2025-68664 (LangGrinch) -- langchain-core serialization vulnerability | Secret exfiltration via deserialization of LLM response fields (`additional_kwargs`, `response_metadata`), arbitrary code execution | LLM framework deserialization is a potent attack surface. Pin and patch AI framework dependencies aggressively. |
| 2025 | Analysis of 38,630 AI Android apps on Google Play | 72% contained hardcoded secrets, averaging 5.1 secrets per app | Mobile AI apps are a massive credential leak surface. Automated scanning of app binaries should be standard. |
| 2025-2026 | GitGuardian 2026 report: AI service credentials | 1.275M AI service secrets leaked (81% YoY increase), 113K DeepSeek keys, 24K secrets in MCP configs | AI infrastructure is the fastest-growing category of leaked secrets. AI coding tools double the secret-leak rate. |

## Open Research Questions

- [ ] How should secrets management handle the unique lifecycle of ML experiment tracking credentials (many short-lived experiments across multiple researchers)?
- [ ] What is the best practice for managing API keys to external model providers (OpenAI, Anthropic, DeepSeek, etc.) where the provider's rotation API support varies?
- [ ] How can secrets injection work seamlessly with distributed training frameworks (Horovod, DeepSpeed) that spawn processes across multiple nodes?
- [ ] How should organizations implement crypto-agility for AI model signing and weight encryption keys in preparation for post-quantum migration (NIST IR 8547 timeline)?
- [ ] What identity and credential models are appropriate for agentic AI systems that dynamically compose tool chains and change operational context at runtime?
- [ ] How can LLM training pipelines be systematically scanned to prevent ingestion of valid secrets from public datasets (e.g., Common Crawl)?
- [ ] What controls can prevent prompt injection attacks from exfiltrating secrets through LLM response serialization (as demonstrated by CVE-2025-68664)?
