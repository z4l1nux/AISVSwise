# C4.4 Secrets & Cryptographic Key Management

[Back to C04 Index](C04-Infrastructure.md)

## Purpose

Protect secrets and cryptographic keys with secure storage, automated rotation, and strong access controls. AI systems manage a wide variety of sensitive credentials: API keys for model providers, database credentials for training data stores, signing keys for model artifacts, encryption keys for model weights, and service account tokens for cloud resources. Compromise of any of these can lead to data theft, model tampering, or unauthorized access to expensive GPU infrastructure.

The scale of this problem is growing rapidly. GitGuardian's 2026 State of Secrets Sprawl report found 28.65 million new secrets on public GitHub in 2025 alone (a 34% year-over-year increase), with AI service secrets reaching 1,275,105 leaked credentials -- an 81% surge. LLM infrastructure components (orchestration, RAG, and vector storage) leaked credentials 5x faster than core model provider services. Additionally, 64% of secrets validated as active in 2022 remained valid through January 2026, demonstrating persistent failure in rotation and remediation.

The problem extends beyond individual developers. As of November 2025, Wiz Research found that 65% of companies on the Forbes AI 50 list had verified secret leaks on GitHub -- representing a combined valuation exceeding $400 billion. One anonymous company's exposure included a Hugging Face token granting access to approximately 1,000 private models alongside multiple Weights & Biases keys that compromised proprietary training data. The MCP ecosystem compounds the risk further: Astrix's analysis of over 5,000 MCP server implementations found that 88% require credentials to operate, yet 53% rely on insecure static API keys or personal access tokens, and only 8.5% have adopted OAuth. An estimated 20,000 MCP server implementations exist on public GitHub, with 79% passing API keys through environment variables.

Non-human identities are now the primary attack surface for credential compromise. SpyCloud's 2026 Identity Exposure Report recaptured 18.1 million exposed API keys and tokens during 2025, including 6.2 million credentials tied specifically to AI tools and services. Unlike human credentials, these machine identities typically lack MFA, rotate infrequently, and carry broad permissions. Tenable's 2026 Cloud and AI Security Risk Report found 65% of organizations harbor "ghost" credentials -- unused or unrotated keys -- with 17% tied to critical administrative privileges and 18% of organizations having granted AI services administrative permissions that are rarely audited. The Darktrace 2026 Annual Threat Report confirmed the shift, finding that nearly 70% of incidents across the Americas began with stolen or misused credentials rather than traditional exploits. The OWASP Non-Human Identities (NHI) Top 10, published in 2025, formalizes these risks: NHI2 (Secret Leakage), NHI7 (Long-Lived Secrets), and NHI4 (Insecure Authentication) map directly to AI credential management failures. CSA's State of NHI and AI Security survey (January 2026) found less than 25% of organizations have documented policies for creating or removing AI identities, and only 12% expressed high confidence in preventing attacks via non-human identities.

## Requirements

| # | Requirement | Level | Role | Threat Mitigated | Verification Approach | Gaps / Notes |
|---|-------------|:-----:|:----:|-----------------|----------------------|--------------|
| **4.4.1** | Verify that secrets are stored in a dedicated secrets management system with encryption at rest and isolated from application workloads. | 1 | D/V | Secrets exposed through container image layers, configuration files, or application logs. Compromise of one workload leading to access to all secrets. GitGuardian (2026) identified 113,000 leaked DeepSeek API keys and 24,008 unique secrets in MCP configuration files alone, often caused by documentation that normalizes hardcoded credential patterns. | Confirm use of a secrets manager (HashiCorp Vault, AWS Secrets Manager, Azure Key Vault, GCP Secret Manager, Infisical). Verify encryption at rest is enabled (check KMS key configuration). Confirm the secrets store runs in a separate network segment from AI workloads. For MCP-based agents, verify that MCP server configs do not contain inline API keys. Scan for secrets in Jupyter notebooks, `.env` files, and `mcp.json` configs using GitGuardian, detect-secrets, TruffleHog, or Betterleaks (open-source successor to Gitleaks released March 2026, featuring token-efficiency-based detection with 98.6% recall vs. 70.4% for entropy methods). Assess compliance with OWASP NHI Top 10 risks NHI2 (Secret Leakage) and NHI6 (Insecure Cloud Deployment Configurations). | ML notebooks (Jupyter, Colab) are a common source of leaked secrets -- developers embed API keys directly in notebook cells. AI coding assistants amplify the problem: AI-assisted commits showed a 3.2% secret-leak rate vs. a 1.5% baseline (GitGuardian 2026). MCP configuration files are an emerging leak vector -- 8.8% of discovered MCP secrets validated as active credentials. Supply chain attacks now target AI toolchains directly: the SANDWORM_MODE attack (February 2026) used 19 typosquatted npm packages to install rogue MCP servers into AI coding tools, exfiltrating SSH keys, AWS credentials, and npm tokens via prompt injection. Consider pre-commit hooks (detect-secrets, TruffleHog, GitGuardian ggshield) to catch secrets before they reach version control. |
| **4.4.2** | Verify that access to production secrets requires strong authentication. | 1 | D/V | Unauthorized access to production model serving keys, training data encryption keys, or cloud provider credentials. Insider threat from developers with overly broad access. In May 2025, an xAI employee leaked a private API key on GitHub that provided access to 60+ fine-tuned and private LLMs containing internal data from SpaceX, Tesla, and Twitter/X (Krebs on Security). | Verify authentication to the secrets manager uses MFA or certificate-based auth for human access. Confirm workload identity (IRSA, Workload Identity, pod identity) is used for service access rather than long-lived credentials. Audit access logs for anomalous patterns. For agentic AI systems, verify that agent credentials are dynamically issued, task-scoped, context-bound, and ephemeral (seconds to minutes). | AI workloads often run as shared service accounts with broad secrets access. Use fine-grained policies: inference services should not access training data encryption keys. Vault policies or AWS IAM conditions should scope access by workload identity, not just by role. Agentic AI systems break traditional vault assumptions: agents change operational context in real-time, making pre-assigned static credentials dangerous. As of early 2026, HashiCorp Vault Enterprise 1.21 natively supports SPIFFE authentication for non-human identities including AI agents, issuing X.509-SVIDs to authenticated workloads. Identity-first access systems (e.g., Aembit, workload identity federation) that make real-time access decisions based on context are emerging alongside vault-centric approaches. The AIMS (Agent Identity Management System) draft framework (March 2026) composes WIMSE, SPIFFE, and OAuth 2.0 into a unified agent identity model, and NIST launched two converging initiatives in February 2026: an NCCoE concept paper on AI agent identity/authorization, and the AI Agent Standards Initiative covering security controls and identity. CSA's January 2026 NHI survey found only 12% of organizations are highly confident in preventing NHI-based attacks, and over 16% do not track new AI-related identity creation at all. Tenable's 2026 report found 49% of identities with critical-severity excessive permissions are dormant, and non-human identities carry higher risk (52%) than human users (37%) in cloud environments. The OWASP NHI Top 10 (2025) provides a structured framework for assessing these risks -- NHI1 (Improper Offboarding), NHI5 (Overprivileged NHI), and NHI7 (Long-Lived Secrets) are directly relevant to AI workload authentication. |
| **4.4.3** | Verify that secrets are deployed to applications at runtime through a dedicated secrets management system. Secrets must never be embedded in source code, configuration files, build artifacts, container images, or environment variables. | 1 | D/V | Secrets leaking through container image layers (recoverable via `docker history`), git history, CI/CD logs, or process environment dumps (`/proc/*/environ`). Analysis of 38,630 AI-enabled Android apps found 72% contained at least one hardcoded secret, averaging 5.1 secrets per app. Researchers found nearly 12,000 valid live secrets (API keys for AWS, Slack, GitHub, etc.) in the Common Crawl December 2024 dataset, which is used to train LLMs -- meaning models may reproduce leaked credentials in generated code. | Scan container images for embedded secrets (Trivy, Syft). Search git history for secrets patterns. Verify secrets are injected via volume mounts (Vault Agent, CSI driver) or API calls, not environment variables. Check CI/CD logs for secret masking. For mobile/edge AI apps, scan APKs/IPAs for hardcoded credentials using tools like MobSF or Ostorlab. | Environment variables are explicitly excluded because they appear in process listings, crash dumps, and debug logs. The Kubernetes Secrets CSI driver or Vault Agent sidecar are preferred injection mechanisms. Note: many ML frameworks read API keys from environment variables by default (e.g., `OPENAI_API_KEY`, `HF_TOKEN`) -- wrappers may be needed to read from mounted secret files instead. Google API keys originally deployed for public services (e.g., Maps) now also authenticate to Gemini, creating unintended exposure -- nearly 3,000 such keys were found on public websites (Truffle Security 2025). The External Secrets Operator for Kubernetes is becoming standard for bridging secrets managers to production clusters. AI platforms with user-facing agent features face additional risk: the Moltbook incident (January 2026) exposed 1.5 million API tokens -- including plaintext OpenAI keys shared between agents -- due to client-side JavaScript containing hardcoded Supabase connection details with no row-level security. Wiz found that nearly half of affected Forbes AI 50 companies either failed to receive or did not respond to disclosure attempts, indicating limited preparedness for credential exposure reports. GitGuardian's 2026 report notes that internal repositories are roughly 6x more likely than public ones to contain hardcoded secrets, and approximately 28% of secret incidents originate entirely outside repositories -- in Slack messages, Jira tickets, and Confluence pages. Betterleaks (March 2026) introduces byte-pair-encoding token efficiency analysis as a detection technique, achieving 98.6% recall on the CredData benchmark, and includes CEL-based validation rules for programmatic secret confirmation. Terraform and IaC state files are another growing vector: CSA warns that AI API keys and database credentials in state files are frequently written in plaintext when not explicitly marked as sensitive, and shared cloud storage buckets storing these files are high-value targets. |
| **4.4.4** | Verify that cryptographic keys are generated and stored in hardware-backed modules (e.g., HSMs, cloud KMS). | 2 | D/V | Key extraction from software keystores via memory dumps, cold boot attacks, or host compromise. Weak key generation from poor entropy sources. Future quantum computing threats to current cryptographic algorithms protecting model weights and AI infrastructure. | Verify KMS or HSM is used for key generation (not `openssl genrsa` on a dev machine). Check that keys are non-exportable from HSMs. Confirm cloud KMS keys use HSM-backed key types (AWS KMS with CloudHSM, Azure Managed HSM, GCP Cloud HSM). Begin inventorying cryptographic algorithms in use to prepare for post-quantum migration per NIST IR 8547 timelines. | Applicable to model signing keys, data encryption keys, and TLS certificate private keys. For AI-specific use cases: model weight encryption keys and inference API authentication keys should use HSM-backed storage. Cloud KMS services provide a practical middle ground between software keystores and dedicated HSMs. Post-quantum readiness: NIST finalized ML-KEM, ML-DSA, and SLH-DSA in August 2024, with HQC selected as a backup for ML-KEM in March 2025 (final standard expected 2027). HSM vendors are now shipping PQC support: Entrust nShield XC, nShield 5, and nShield as a Service support ML-DSA (FIPS 204) in firmware as of 2025, and SEALSQ targets 2025-2026 for full NIST PQC algorithm integration into secure elements. NIST IR 8547 sets a deprecation timeline for quantum-vulnerable algorithms by 2035, but organizations should begin crypto-agility planning now. AI model signing and weight encryption keys are long-lived assets that may be vulnerable to "harvest now, decrypt later" attacks. |
| **4.4.5** | Verify that secrets rotation is automated. | 2 | D/V | Long-lived credentials remaining valid after team member departure, credential leak, or periodic compromise. Difficulty determining blast radius of a compromised credential when rotation history is unclear. GitGuardian found 64% of leaked secrets from 2022 were still valid in January 2026, underscoring that rotation and remediation are widely neglected. | Verify automated rotation is configured (Vault dynamic secrets, AWS Secrets Manager rotation lambdas, Infisical automatic rotation). Check rotation frequency is appropriate (database credentials: 30-90 days; API keys: 90 days; certificates: < 24 hours for short-lived). Confirm rotation does not cause downtime. For agentic AI, verify that agents use ephemeral, dynamically generated credentials rather than long-lived static secrets. | Dynamic secrets (Vault, AWS STS) are preferred over static secrets with rotation. For ML-specific credentials: model provider API keys (OpenAI, Anthropic, DeepSeek) may not support automated rotation via API -- manual rotation workflows need documentation. GPU cloud provider credentials (Lambda Labs, CoreWeave) may have limited rotation API support. For agentic AI workloads, the hybrid approach is gaining traction in 2026: a secrets backend (Vault or AWS Secrets Manager) combined with External Secrets Operator for Kubernetes and tools like Doppler for CI/CD injection. Traditional vault-centric rotation breaks down when AI agents assemble and execute pipelines dynamically -- short-lived credentials tied to agent identity and task context are essential. HashiCorp's validated pattern for AI agent authentication uses OAuth 2.0 On-Behalf-Of (OBO) token exchange with Vault: agents exchange user JWTs for scoped, short-lived credentials, preserving user attribution through the full execution chain. Google's Agent-to-Agent (A2A) protocol with Vault on Kubernetes provides another reference pattern for secure inter-agent credential management. Tenable's 2026 report quantifies the failure: 65% of organizations harbor "ghost" credentials (unused or unrotated keys), with 17% tied to critical administrative privileges. The Darktrace 2026 Annual Threat Report found credential abuse has overtaken traditional exploits as the dominant initial access vector, with nearly 70% of Americas-region incidents beginning with stolen or misused accounts. OWASP NHI Top 10 risk NHI7 (Long-Lived Secrets) directly addresses this: NHIs with no expiration or distant future expiration create prolonged attack windows that automated rotation must close. |

## Implementation Maturity

| Requirement | Maturity | Notes |
|-------------|:---:|-------|
| 4.4.1 Dedicated secrets management | Mature | HashiCorp Vault, AWS Secrets Manager, Azure Key Vault, GCP Secret Manager, and Infisical are all production-grade. External Secrets Operator bridges K8s to any backend. Secret scanning tooling is evolving rapidly: Betterleaks (March 2026) succeeds Gitleaks with token-efficiency-based detection and CEL validation rules. Gap: ML notebooks and MCP configs remain major leak vectors — Astrix found 53% of ~20,000 MCP servers on GitHub use static API keys, with only 8.5% adopting OAuth. SpyCloud recaptured 18.1M exposed API keys/tokens in 2025, including 6.2M tied to AI tools. Supply chain attacks like SANDWORM_MODE (Feb 2026) now inject rogue MCP servers into AI coding tools to exfiltrate credentials. |
| 4.4.2 Strong authentication for secrets access | Mature | Workload identity (AWS IRSA, GCP Workload Identity, Azure Pod Identity) and MFA for human access are well-established. Vault Enterprise 1.21 adds native SPIFFE authentication for non-human identities. The OWASP NHI Top 10 (2025) now provides a structured risk framework for non-human identity security. Gap: agentic AI systems still lack mature identity standards — the AIMS draft framework (March 2026) and NIST's NCCoE AI agent identity initiative (February 2026) are early-stage efforts. CSA's January 2026 NHI survey found only 12% of organizations are highly confident in preventing NHI-based attacks, and less than 25% have documented AI identity lifecycle policies. |
| 4.4.3 Runtime injection, no embedded secrets | Maturing | Vault Agent sidecar, K8s Secrets CSI driver, and External Secrets Operator handle injection well. Pre-commit scanning (detect-secrets, TruffleHog, ggshield, Betterleaks) catches leaks. Gap: many ML frameworks default to env-var-based keys (`OPENAI_API_KEY`, `HF_TOKEN`) — wrappers needed. AI coding assistants double secret-leak rates (3.2% vs 1.5% baseline per GitGuardian 2026). Internal repos are 6x more likely than public ones to contain secrets, and 28% of incidents originate outside repos (Slack, Jira, Confluence). IaC state files are an undermonitored vector for plaintext AI credentials. |
| 4.4.4 HSM/KMS-backed key storage | Mature | AWS KMS (CloudHSM), Azure Managed HSM, GCP Cloud HSM are production-ready for model signing, data encryption, and TLS keys. HSM vendors are shipping PQC support: Entrust nShield supports ML-DSA in firmware, SEALSQ targets 2025-2026 for full NIST PQC integration. Gap: most organizations have not started crypto-agility planning despite NIST IR 8547 deprecation timelines. AI model signing keys are long-lived and vulnerable to "harvest now, decrypt later." |
| 4.4.5 Automated secrets rotation | Maturing | Vault dynamic secrets, AWS rotation lambdas, and Infisical auto-rotation work well. HashiCorp's validated AI agent pattern uses OAuth 2.0 OBO token exchange for just-in-time credential issuance. Gap: model provider API keys (OpenAI, Anthropic, DeepSeek) often lack rotation API support — manual workflows needed. GitGuardian found 64% of 2022 secrets still valid in Jan 2026, showing rotation is widely neglected. Tenable (2026) found 65% of organizations harbor ghost credentials with 17% tied to admin privileges — rotation failure at scale. Agentic AI needs ephemeral, dynamically generated credentials tied to task context — Google's A2A protocol with Vault provides an emerging reference implementation. OWASP NHI7 (Long-Lived Secrets) formalizes this risk category. |

- [ ] How should organizations implement crypto-agility for AI model signing and weight encryption keys in preparation for post-quantum migration (NIST IR 8547 timeline)?

- [ ] What controls can prevent prompt injection attacks from exfiltrating secrets through LLM response serialization (as demonstrated by CVE-2025-68664)?

- [ ] How should AI agent platforms that aggregate user credentials (as in the Moltbook case) implement credential isolation to prevent platform-level breaches from exposing all user secrets?
- [ ] How can organizations address the 65% "ghost credential" problem (Tenable 2026) in AI-heavy environments where service accounts and API keys proliferate faster than governance processes can track them?
- [ ] What tooling and processes are needed to detect secret leakage outside traditional code repositories -- in Slack, Jira, Confluence, and IaC state files -- where GitGuardian 2026 reports 28% of incidents originate?
- [ ] How should organizations operationalize the OWASP NHI Top 10 (2025) alongside existing secrets management practices to address the full lifecycle of non-human identities in AI workloads?
