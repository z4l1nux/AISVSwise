# C10.4: Schema, Message, and Input Validation

> **Parent:** [C10 MCP Security](C10-MCP-Security)
> **Requirements:** 8 | **IDs:** 10.4.1–10.4.8

## Purpose

This section ensures that all data flowing through MCP — tool definitions, tool arguments, tool responses, and JSON-RPC framing — is validated for integrity, authenticity, and safety. MCP is fundamentally a protocol for injecting external data into an LLM's context window, which makes it a prime vector for indirect prompt injection. Tool responses that are injected into the context without validation can manipulate the model's behavior. Tool definitions themselves can contain hidden instructions in their descriptions (tool poisoning). Schema tampering can change what arguments a tool accepts, enabling parameter injection. This section addresses all of these vectors.

Research published in early 2026 ("Breaking the Protocol," arXiv:2601.17549) provided rigorous quantification of these risks. The study found that MCP's architecture amplifies prompt injection attack success rates by 23-41% depending on attack type, with cross-server propagation attacks succeeding 61.3% of the time (vs. 19.7% baseline) and tool response manipulation reaching 52.1% (vs. 28.4% baseline). The JSON-RPC specification underlying MCP provides no application-layer protections — no message authentication, no replay protection, no capability binding, and no application-level origin identification. The study proposed AttestMCP, a backward-compatible protocol extension using HMAC-SHA256 message signatures and cryptographic capability certificates, which reduced overall attack success from 52.8% to 12.4% (a 76.5% reduction). The residual 12.4% reflects indirect injection through legitimately-retrieved content — a fundamental LLM limitation beyond protocol remediation. These findings validate every requirement in this section and underscore the urgency of schema and message validation in MCP deployments.

This can lead to request smuggling — where one party interprets a message boundary differently from another — enabling injection of unauthorized tool invocations. | Send malformed JSON-RPC messages (truncated JSON, oversized payloads, nested messages, interleaved requests) and verify the server rejects them cleanly without processing partial content. Verify maximum payload size enforcement. Test with concurrent requests to ensure proper framing isolation. | The MCP spec uses JSON-RPC 2.0 over streamable HTTP, which provides reasonable framing via HTTP request/response boundaries. The risk is higher with SSE transport, where message boundaries are delimited by newlines and are more susceptible to injection. | **10.4.4** | **Verify that** MCP servers perform strict input validation for all function calls, including type checking, boundary validation, enumeration enforcement, and rejection of unrecognized or oversized parameters. | 2 | D/V | **Parameter injection / type confusion.** The model (or an attacker via prompt injection) supplies unexpected argument types, out-of-range values, or additional unrecognized parameters to a tool invocation. Test with: (1) wrong types (string where number expected), (2) values outside declared ranges, (3) extra parameters not in the schema, (4) oversized string values, (5) nested objects where scalars are expected. Verify all are rejected before reaching business logic. | Standard input validation practice, but critical in MCP because the model generates the arguments — and the model can be manipulated via prompt injection to generate malicious arguments. Without change detection, the client continues to use the modified tools under the original trust assumption. This is the "rug-pull" variant of tool poisoning. | Verify the client stores hashes or versioned snapshots of tool definitions from `tools/list`. Modify a tool definition on the server side and trigger `notifications/tools/list_changed`. Verify the client detects the change and requires re-approval (user confirmation or automated policy check) before allowing invocation of the modified tool. | The MCP spec defines `notifications/tools/list_changed` for tool updates, but does not mandate change detection or re-approval. This requirement adds a security layer on top of the protocol. Implementation approaches: content-hash comparison, semantic diff of descriptions, or full re-approval workflow. |
| **10.4.6** | **Verify that** MCP server error and exception responses do not expose stack traces, internal file paths, tokens, or tool implementation details to the client or model context. | 1 | D/V | **Information disclosure via error messages.** Verbose error responses from MCP servers can leak internal implementation details: stack traces revealing framework versions and code paths, file paths revealing server architecture, database error messages revealing schema structure, or partial tokens in error context. This information aids attackers in crafting targeted exploits. When errors are injected into the model context, the model may inadvertently repeat sensitive details to the user. | Trigger various error conditions (invalid arguments, server-side exceptions, timeout, resource not found) and inspect the error responses for sensitive information. Verify error responses use generic error codes and messages. Review server error-handling code for proper exception sanitization before returning JSON-RPC error responses. | Standard error-handling hygiene, but MCP adds a unique dimension: error messages are often injected into the model context, where the model may summarize or quote them to the user. Sensitive information in errors thus has two exposure paths: direct (client inspection) and indirect (model repeating it in conversation). |
| **10.4.7** | **Verify that** MCP implementations reject JSON-RPC messages containing duplicate keys at any nesting level, preventing parser disagreement where different components resolve the same message to different values. | 2 | D/V | **Parser differential attacks via duplicate keys.** The JSON specification (RFC 8259) does not require unique keys — it says behavior with duplicate keys is undefined. Different JSON parsers handle duplicates differently: some take the first value, some take the last, some merge. An attacker can craft a JSON-RPC message with duplicate keys where the authorization middleware sees one value (e.g., `"tool": "safe_tool"`) and the execution engine sees another (e.g., `"tool": "dangerous_tool"`). | Send JSON-RPC messages with duplicate keys at various nesting levels and verify the server rejects them with a parse error. Test with duplicate `method`, `params`, and `id` fields. Verify that the JSON parser is configured to reject duplicates (e.g., `JSON.parse` with a reviver that detects duplicates, or a strict parser like `json5` in strict mode). | Most standard JSON parsers accept duplicate keys silently (taking the last value in JavaScript, the last in Python, the first in some Go parsers). Strict rejection requires explicit configuration. This is a defense-in-depth control against a sophisticated but real attack vector documented in parser differential research. |
| **10.4.8** | **Verify that** intermediaries evaluating message content either forward the canonicalized representation they evaluated or reject messages where multiple byte representations could produce different parsed structures. | 3 | D/V | **Canonicalization attacks through intermediaries.** An intermediary (proxy, WAF, API gateway) evaluates a JSON-RPC message for security policy (e.g., blocking certain tool names) but then forwards the original bytes. If the original message uses Unicode escapes, UTF-8 edge cases, or other encoding tricks, the downstream MCP server may parse it differently from the intermediary, bypassing the security check. Example: intermediary sees `"method": "\u0074ools/call"` and does not recognize it as `"tools/call"`. | Deploy a security intermediary (WAF or API gateway) in front of the MCP server. Send messages using various encoding representations of the same content (Unicode escapes, overlong UTF-8 sequences, BOM characters) and verify the intermediary either normalizes the message before forwarding or rejects ambiguous encodings. Verify the MCP server and intermediary agree on parsing of edge-case encodings. | Level 3 because this is a sophisticated attack vector that requires specific intermediary configurations to be exploitable. The mitigation (canonicalize-then-forward) is well-understood from web security (HTTP request smuggling parallels) but tooling support for MCP-specific canonicalization is not yet available. |

## Research Findings: Attack Amplification Through MCP (2025-2026)

The "Breaking the Protocol" study (arXiv:2601.17549, January 2026) quantified how MCP's architecture amplifies known attack vectors:

| Attack Type | Baseline Success | With MCP | Amplification |
|---|---|---|---|
| Cross-Server Propagation | 19.7% | 61.3% | +41.6% |
| Tool Response Manipulation | 28.4% | 52.1% | +23.7% |
| Sampling-Specific Attacks | — | 58-72% | — |
| Data Exfiltration via Sampling | — | 42-61% | — |
| **Overall** | **26.4%** | **52.8%** | **+26.4%** |

Three protocol-level vulnerabilities were identified:
1. **Capability Attestation Absence:** Servers self-assert capabilities without cryptographic verification, allowing malicious servers to claim any capability and escalate permissions post-initialization.
2. **Unauthenticated Sampling:** The `sampling/createMessage` mechanism allows servers to inject prompts processed identically to user input, with no tested MCP host providing visual distinction for server-sourced messages.
3. **Implicit Trust Propagation:** In multi-server deployments, tool outputs from one server influence operations on others without provenance tracking, enabling cross-server data exfiltration.

The proposed AttestMCP extension addresses these with HMAC-SHA256 message signatures, cryptographic capability certificates, origin tagging, and replay protection (timestamp/nonce validation) — adding median overhead of only 2.4ms per message (cached).

Additional research by Palo Alto Unit 42 documented prompt injection attack vectors specifically through MCP sampling, and Equixly published offensive security testing methodologies for MCP servers (February 2026) demonstrating practical exploitation of schema validation gaps.

## Open Research Questions

- [ ] Is there a viable approach to sanitizing natural-language tool output against prompt injection without losing semantic content?
- [ ] Should the MCP spec define a standard schema-signing mechanism, and what key management model would be appropriate?
- [ ] Can static analysis of tool definitions detect potential tool-poisoning payloads in descriptions (e.g., hidden instructions)?
- [ ] What is the right response when a rug-pull is detected — block the tool, alert the user, or quarantine the server?
- [ ] Should MCP mandate a canonical JSON representation (e.g., JCS / RFC 8785) to eliminate parser differential attacks?
- [ ] Should AttestMCP (or a similar message authentication extension) be incorporated into the core MCP specification, given the demonstrated 76.5% reduction in attack success?
- [ ] How should MCP implementations handle the residual 12.4% attack success rate that stems from legitimate content containing indirect injection — is this a protocol problem or an LLM problem?
