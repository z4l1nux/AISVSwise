# C9.6: Authorization, Delegation, and Continuous Enforcement

[Back to C09 Index](C09-Orchestration-and-Agents.md)

## Purpose

Agents act on behalf of users and interact with systems that have their own access control policies. This section ensures that agent actions are authorized against fine-grained policies at execution time, that delegation context (who the agent is acting for, with what scope) is propagated and enforced throughout the call chain, and that authorization decisions are never made by the AI model itself. The core principle: access control must be enforced by deterministic policy engines, not probabilistic language models.

## Implementation Guidance

### The Authorization Crisis for Agentic AI (2025--2026)

Traditional IAM frameworks were designed for human users and static service accounts. Research and industry analysis in 2025--2026 identifies seven critical failures when applying these frameworks to AI agents (ISACA, 2025):

1. **Coarse-grained permissions:** Agents need fine-grained, task-specific permissions that dynamically change based on context, not static role-based access with long-lived tokens.
2. **Single-entity model limitations:** Current systems cannot track complex nested delegations where agents spawn sub-agents or represent multiple principals simultaneously.
3. **Context blindness:** Access decisions lack awareness of runtime context, agent intent, or risk level -- permissions remain static throughout sessions.
4. **Scalability pressures:** Hundreds or thousands of transient agents generate overwhelming token management demands (issuance, validation, revocation).
5. **Inadequate inter-agent authentication:** OAuth and SAML rely on hierarchical trust models unsuitable for decentralized agent networks that require peer-to-peer trust.
6. **Secret sprawl:** Each agent's multiple API integrations create a multiplicative factor in secrets requiring rotation and storage.
7. **Revocation complexity:** Revoking access in one place does not automatically cut off access elsewhere across distributed ephemeral sessions.

A 2026 survey (Grantex) found that 93% of popular AI agent projects use unscoped API keys, and fewer than half of companies deploying agents had comprehensive governance in place.

### Emerging Standards and Frameworks

- **NIST AI Agent Standards Initiative (February 2026):** NIST's Center for AI Standards and Innovation (CAISI) launched the AI Agent Standards Initiative alongside a concept paper on "Accelerating the Adoption of Software and Artificial Intelligence Agent Identity and Authorization." The initiative proposes NCCoE lab demonstrations exploring agent identification (distinguishing agents from humans), authorization (OAuth 2.0/2.1 extensions and policy-based access control), and access delegation (linking user identities to agents for accountability). The concept paper recommends ABAC over RBAC for agent permissions and explicitly calls for transitive trust limits preventing unbounded permission cascading. Standards under consideration include MCP, OAuth 2.0/2.1, OpenID Connect, SPIFFE/SPIRE, SCIM, and Next Generation Access Control.
- **IETF AIMS Framework (draft-klrc-aiagent-auth-00, March 2026):** The Agent Identity Management System composes WIMSE, SPIFFE, and OAuth 2.0 into a layered architecture. It defines five layers: identifiers (SPIFFE IDs), credentials (short-lived X.509/JWT bound to private keys), attestation (TEE evidence, binary validation, Kubernetes PSAT), authentication (mTLS + WIMSE Workload Proof Tokens), and authorization (OAuth 2.0 with transaction tokens). Three authorization scenarios are defined: user-delegated (authorization code grant), agent self-authorized (client credentials), and agent-as-resource-server. The framework explicitly requires audit logs recording authenticated agent ID, delegated subject, action, decision, and attestation state.
- **Agent Relationship-Based Identity and Authorization (ARIA):** Treats delegation relationships as explicit, cryptographically verifiable entities in a graph. Every delegation is a distinct relationship that enables surgical permission revocation and full auditability. ARIA integrates OAuth 2.0 Rich Authorization Requests, OAuth 2.0 token exchange (RFC 8693), and OpenID AuthZEN for context-aware policy evaluation.
- **OpenID AuthZEN 1.0 (Approved January 2026):** Now a Final Specification, AuthZEN standardizes a transport-agnostic API where any Policy Enforcement Point queries any Policy Decision Point regardless of vendor. The information model is a four-element tuple (Subject, Action, Resource, Context) that maps cleanly to agent tool invocations. This enables fine-grained, context-aware policies without abandoning installed-base OAuth infrastructure -- applying constraints such as geo-fences, budget thresholds, and temporal windows.
- **Zero Trust Identity with DIDs:** Combines Decentralized Identifiers, Verifiable Credentials, Agent Naming Services, and zero-knowledge proofs for privacy-preserving compliance verification in agent identity.

### Practical Authorization Patterns

- **Scopes for coarse-grained limits, claims for fine-grained context:** Use OAuth scopes to define broad permission boundaries and token claims to carry runtime context (user identity, tenant, data classification, risk score) that policy engines evaluate per-call. URI-style scope naming (e.g., `urn:infra:monitoring:read`) improves cross-service compatibility.
- **On-behalf-of delegation with dual-identity tokens:** OAuth 2.0 token exchange (RFC 8693) cryptographically binds an actor (the agent) to a delegator (user or service), preserving the chain of responsibility without exposing the user's credentials. JWT tokens should encode the agent as `sub` and the delegated user as `act.sub`, with nested `act` claims supporting multi-level delegation chains (user -> agent1 -> agent2 -> service). Recommended token lifetime is 5--15 minutes.
- **Policy engines at the enforcement point:** OPA, Cedar, and Zanzibar-style systems enforce fine-grained policies. The policy engine must sit between the model's tool selection and actual tool execution -- never after. As of March 2026, AWS Bedrock AgentCore demonstrates this pattern concretely: every agent-to-tool request is evaluated against Cedar policies before execution, with policies definable programmatically or generated from natural-language descriptions.
- **Just-in-time credential issuance:** Request credentials at runtime for each tool invocation and discard them immediately after task completion. This eliminates standing access vulnerabilities inherent in long-lived tokens. The Oso platform implements this pattern with delegated access where the agent inherits the user's identity via token exchange and the downstream service enforces the user's current permissions in real-time -- if the user loses access, so does the agent immediately.
- **Short-lived, single-use credentials:** Replace long-lived API keys with ephemeral tokens scoped to individual tasks. This limits blast radius and eliminates the stale-authorization window. The Grantex 2026 survey found 93% of popular agent projects still use unscoped API keys -- this is the single most common authorization failure in deployed agent systems.
- **Human-in-the-loop approval gates:** For sensitive or destructive operations (healthcare data access, record deletion, compliance actions), require manual approval. Grant access to action *types* rather than individual operations to keep the approval burden manageable.

### Recent Incidents and Attack Patterns

Several real-world incidents in 2025--2026 demonstrate why authorization enforcement matters for agents:

- **MITRE ATLAS AML.T0053 -- AI Agent Tool Invocation:** Formally catalogued as a technique where adversaries exploit an agent's authorized tool access to perform unauthorized actions. Microsoft Copilot M365 and GitHub Copilot Chat have both been exploited through this vector, with agents directed to exfiltrate data or perform actions outside intended scope.
- **OpenClaw Malicious Skills (February 2026):** The largest confirmed supply chain attack targeting agent infrastructure. MITRE ATLAS investigators confirmed 1,184 malicious skills across the ClawHub ecosystem (roughly one in five packages), using direct and indirect prompt injection combined with unauthorized tool invocation.
- **Langflow Authentication Bypass (2025):** CVE-2025-34291 demonstrated how permissive CORS settings combined with credential forwarding created a complete authentication bypass in an agent workflow platform. Default configurations allowed cross-origin requests with credentials from any origin.
- **Git MCP Server Exploit Chain (January 2026):** Three CVEs (CVE-2025-68143, CVE-2025-68144, CVE-2025-68145) targeting Anthropic's Git MCP server demonstrated path traversal, argument injection, and repository scoping bypass -- all of which are authorization boundary failures.
- **Microsoft 365 Copilot BizChat Privilege Escalation (March 2026):** CVE-2026-26137 (CVSS 8.9) -- an SSRF vulnerability in Copilot BizChat that allowed an authorized attacker to escalate privileges over the network. The vulnerability is notable because the AI assistant itself becomes the privilege escalation vector: if an attacker can move the assistant into a higher-authority state, it becomes a tool for scaling access across the organization.
- **MS-Agent Shell Injection (2026):** CVE-2026-2256 -- the MS-Agent framework for autonomous AI workflow automation failed to sanitize user-controlled or AI-generated inputs passed through its built-in Shell tool, enabling prompt injection to trigger arbitrary command execution with the agent's privileges. This is a textbook example of why tool parameter validation must happen at the policy engine level, not in the agent itself.
- **2026 Incident Volume:** Industry tracking reports 520 confirmed tool misuse and privilege escalation incidents involving AI agents in 2026 alone, making it the most common reported agent security incident category.

### Semantic Privilege Escalation

A distinct class of authorization failure is emerging that traditional access control cannot address: **semantic privilege escalation**. In this attack, the agent never exceeds its technical permissions -- every API call, every tool invocation is individually authorized. But the *sequence* of actions falls outside the semantic scope of the assigned task.

Consider: a user asks an agent to summarize a document. The document contains hidden prompt injection (white text, invisible Unicode). The agent, following the injected instructions, scans the user's drive for API keys and emails them to an external address. Each individual action (read document, search files, send email) may be within the agent's technical authorization. The escalation is in *intent*, not in *permissions*.

Defending against semantic privilege escalation requires capabilities beyond traditional IAM:

1. **Intent capture and tracking** -- Record what the user actually asked for and maintain it as a reference throughout execution.
2. **Intent-action alignment evaluation** -- At each step, compare the proposed action against the captured intent. Does "summarize this document" semantically justify "search for files matching *.key"?
3. **Behavioral anomaly detection** -- Flag action sequences that deviate from typical patterns for the task type, even if each action is individually authorized.
4. **Security-annotated execution traces** -- Log not just what happened, but the semantic justification chain, enabling post-hoc analysis of whether the agent stayed within scope.

As of March 2026, no production authorization framework fully addresses semantic privilege escalation. This is an active research area where the gap between what is technically authorized and what is semantically appropriate remains largely unpoliced.

### Cloud-Native Agent Authorization Detection

Cloud providers are beginning to ship purpose-built detection for agent authorization abuse. Google Cloud Security Command Center (SCC) now includes a dedicated category of AI agent threat findings that map to MITRE ATT&CK tactics:

- **AI Agent Suspicious Token Generation Using Implicit Delegation** -- Flags when agents misuse implicit delegation to generate elevated-privilege tokens (maps to T1528: Steal Application Access Token).
- **AI Agent Suspicious Cross-Project Access Token Generation** -- Detects agents attempting to generate tokens for resources outside their authorized project scope.
- **AI Agent Suspicious Token Generation Using signJwt** -- Catches agents using the `signJwt` API to forge unauthorized JWT credentials.
- **AI Agent Unauthorized Service Account API Call** -- Identifies API calls exceeding the agent's assigned permissions.
- **AI Agent Excessive Permission Denied Actions** -- Tracks repeated failed authorization attempts, indicating reconnaissance or brute-force probing of privilege boundaries.

These detections are significant because they represent the first shift from generic workload monitoring to agent-specific authorization threat models. Organizations deploying agents on GCP should enable these findings and route them to their SIEM/SOAR pipeline.

### MCP-Specific Authorization Patterns

The Model Context Protocol (MCP) introduces specific authorization challenges because MCP servers currently expose all tools to connected agents by default -- there is no built-in selective disclosure mechanism. This creates both context window bloat (the agent sees tools it should not use) and security risk (the agent can invoke tools outside its authorization scope).

Recommended patterns for MCP authorization:

- **Selective tool disclosure:** MCP servers should implement tool filtering, revealing only tools the connecting agent/user is authorized to use. This reduces both the attack surface and the risk of the model selecting inappropriate tools.
- **Per-request authorization at the MCP boundary:** Rather than relying on the MCP client (the agent) to self-police, embed a Policy Decision Point within or in front of each MCP server. Cerbos provides an open-source policy-as-code engine that can run as a sidecar or library for this purpose.
- **Token downscoping at each hop:** When an MCP client calls an MCP server which itself calls downstream services, apply OAuth 2.0 token exchange (RFC 8693) at each boundary to narrow credentials. Do not forward the client's original token.
- **SPIFFE identities for MCP workloads:** Assign SPIFFE IDs to MCP servers and clients, enabling mutual TLS authentication and cryptographically verified identity at every connection. This is more robust than static API keys and aligns with the IETF AIMS framework's identity layer.

### CSA Agentic Trust Framework

The Cloud Security Alliance published the Agentic Trust Framework (ATF) in February 2026, applying Zero Trust governance specifically to AI agents. The ATF organizes controls around five pillars: identity (authentication, authorization, session management), behavior (observability, anomaly detection, intent analysis), data governance (input validation, PII protection, output controls), segmentation (access boundaries, resource limits, policy enforcement), and incident response (circuit breakers, kill switches, containment). A distinctive feature of the ATF is its **progressive autonomy model** -- agents are deployed with minimal autonomy and earn increased autonomy through demonstrated safe behavior, with continuous monitoring at every level. This contrasts with traditional "grant and forget" access models.

### Anti-Patterns to Avoid

- Letting the AI model make access control decisions based on conversation context (prompt injection can manipulate these).
- Using the end user's actual credentials for agent actions instead of scoped delegation tokens.
- Caching authorization decisions for long-running agent tasks without event-driven invalidation.
- Granting agents broad "admin" or "full-access" scopes because their task is open-ended.
- Using static API keys for agent identity -- agents are workloads that should use short-lived, cryptographically-bound credentials (the IETF AIMS framework explicitly labels static API keys as an anti-pattern).
- Allowing unlimited delegation depth in multi-agent chains without policy-enforced limits on transitive trust.

## Open Research Questions

- How do you define "minimum necessary scope" for delegation tokens when the agent's task is open-ended (e.g., "research and summarize")? The NIST concept paper recommends ABAC but does not yet provide concrete policy templates.
- Can authorization policies for agent actions be auto-generated from tool manifests and user role definitions? AWS Bedrock AgentCore's natural-language-to-Cedar generation is an early attempt, but the correctness of generated policies has not been independently validated at scale.
- What are the performance implications of per-call authorization in high-throughput agent systems, and what caching strategies are safe? AuthZEN's standardized PEP-to-PDP API may reduce integration overhead, but latency benchmarks for agent-scale workloads are not yet published.
- How should authorization work for agents that operate across organizational boundaries (federated agent systems)? The IETF AIMS framework defines identity chaining and cross-domain flows, but real-world deployments are still in pilot stage.
- How will the NIST AI Agent Standards Initiative (comment period closing April 2026) shape mandatory requirements for agent authorization in federal and regulated environments?
- Can the ARIA graph-based delegation model and the AIMS layered architecture converge into a unified standard, or will organizations need to choose between them?
- What attestation mechanisms (TEE evidence, binary validation, supply-chain provenance) are practical for agent identity in cloud-native environments, and how do they interact with existing SPIFFE/SPIRE deployments?
- How can semantic privilege escalation be detected in real-time when each individual agent action is technically authorized? Intent-tracking and behavioral anomaly detection are proposed approaches, but no production system has demonstrated reliable detection without unacceptable false-positive rates.
- As cloud providers ship agent-specific authorization detections (Google Cloud SCC's AI threat findings being the first), will these become a de facto standard that other providers must match? What coverage gaps exist in current detections?
- How should MCP authorization evolve? The current protocol lacks built-in selective tool disclosure and per-request authorization primitives. Should these be added to the protocol specification, or should they remain the responsibility of individual MCP server implementations?
